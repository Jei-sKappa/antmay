<objective>
Ship the first V1 spine skill: `capture-inbox` — a standalone, harness-agnostic skill that writes a thread-local Inbox item to `inbox/open/` using the V1 record-form filename grammar, with capture trigger behavior gated on whether the run is user-active or autonomous. Wire it into all four registration touchpoints required by `CLAUDE.md` for every new skill: the skill folder, the `JeisKappa-workflow` marketplace plugin entry, the `.vscode/settings.json` conventional-commit scopes array (alphabetical), and the `README.md` "Available skills" index.

Purpose: Without `capture-inbox`, downstream Phase 2 skills (`discussion`, `seeded-discussion`) and every future review/implementation skill have no canonical way to park a side-finding under a thread — they would either silently lose the finding or have to redefine the record-form grammar in their own body. Wiring all four registration files in the same plan is mandated by `CLAUDE.md` "When adding a new skill" — partial registration leaves the skill invisible to `npx skills list` (no marketplace entry), silently misclassified at commit time (no scope entry), or undiscoverable to users (no README entry).

Output: One new skill folder (`skills/capture-inbox/SKILL.md`), three edits to existing registration files. No reference docs (`capture-inbox` is small enough that all its rules fit in the skill body).

Depends on: Nothing in Phase 2 — Plan 02-01 is the foundation Plans 02-02 and 02-03 build on (both later plans cite `capture-inbox` from their own bodies as the canonical way to park out-of-scope branches).
</objective>
