<!-- Source authority: docs/threads/260520095223Z-agentic-workflow/discussions/260518200115Z-agentic-workflow-design-discussion.md (D50) -->

Both spec-* skills MUST produce artifacts that cover ALL 8 elements. Section headings are at executor discretion (D52 — no mandatory "Decisions" heading), but every element must be present in the body:

1. **Intended outcome** — what this spec, when implemented, produces for the user
2. **Context** — why this is being built, what came before
3. **Scope / non-scope** — boundary statement, including what's explicitly out
4. **Expected behavior** — the observable behaviors a future executor needs
5. **Constraints** — tech / repo / harness / safety constraints
6. **Explicit decisions** — settled trade-offs, INLINED into the body where operative (scope, constraints, expected behavior, acceptance), with citations to source decision logs by path + D<N> (per D52, SPEC-05)
7. **Unresolved questions** — open issues that don't block emission
8. **Acceptance guidance** — how a reviewer will know the implementation is right

The 4 anti-sycophancy marker phrases (verbatim) that spec-interactive MUST carry, taken from skills/discussion/SKILL.md:

1. "make them feel good"        (line 15 in the ## Anti-Sycophancy Stance prefatory paragraph)
2. "Disagree when you disagree" (line 19 bullet label)
3. "pushback as correctness"    (line 23 bullet label fragment "Do not treat pushback as correctness")
4. "refuse to log"              (line 73 dissent-handling rule)

The mantra is heightened for spec-interactive: bad design calls in the spec become expensive in the implementation phase — see 03-CONTEXT.md "Skill body voice and structure".

From docs/workflow/v1/filename-grammar.md — versioned form (what spec-* emits):

```text
<YYMMDDHHMMSSZ>-v<N>[-<kebab-descriptor>]-<artifact-type>.md
```

For `spec-*` skills:
- `<artifact-type>` is the literal token `spec`
- First emission is `v1`, NO descriptor (just `<UTC>-v1-spec.md`)
- N starts at 1, not 0 [D47]
- Subsequent revisions emit `v2`, `v3`, etc. — NOT in-place edits [D39, immutability.md]

Examples:
- First emission: `260521120000Z-v1-spec.md` (mainline v1, no descriptor)
- First emission with descriptor: `260521120000Z-v1-onboarding-spec.md`
- Second mainline emission: `260521130000Z-v2-spec.md`

From docs/workflow/v1/thread-layout.md — target folder:

```text
docs/threads/<YYMMDDHHMMSSZ-slug>/specs/
```

The `specs/` folder is created on-demand on the first spec written.

From docs/workflow/v1/immutability.md — emitted spec artifacts are immutable. A typo discovered in an emitted spec means a new version (`v2`), not an in-place edit. To resolve "the spec" when multiple candidates exist, ASK the user — there is no global "latest" algorithm.

From skills/derive-spec/SKILL.md — what spec-* must NOT do:
- spec-auto / spec-interactive FORWARD-DESIGN a spec FROM a proposal / discussion / issue / prompt
- derive-spec REVERSE-ENGINEERS a spec FROM an existing codebase
These are non-overlapping concerns and live as separate skills [SPEC-04]
