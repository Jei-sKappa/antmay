# Separate workflow-bound skills from context-agnostic reusable skills

The skills repository currently hosts two kinds of skills side by side under `skills/`: skills that explicitly serve the thread-based, spec-driven development workflow (discussion capture, spec, plan, implement, review, reconcile, roadmap, and the primitives they compose into) and skills that are useful in any context, independent of that workflow (meta-prompting, consult-the-expert, brief-the-recipient, afk-exploration, and similar). The owner wants this repository to become the reference repository for the workflow itself — the home of the skill suite that supports spec-driven development starting from a thread, and the base for a future CLI that can orchestrate these skills autonomously, along with add-ons or plugins designed specifically for workflow-supporting skills.

This thread exists to decide whether and how to separate the two kinds of skills. Two candidate directions are on the table: (a) move the reusable, context-agnostic skills into a completely separate folder within this repository to distinguish them more clearly, or (b) extract them into an entirely separate repository containing only reusable skills (probably named "skills"), renaming the current repository to a new, still-to-be-decided name dedicated solely to the thread-based workflow.

Intended outcome: a settled separation strategy — including repository naming if a split or rename happens — that leaves this repository focused on the workflow suite and clears the path toward the future orchestration CLI and its workflow-specific add-ons.
