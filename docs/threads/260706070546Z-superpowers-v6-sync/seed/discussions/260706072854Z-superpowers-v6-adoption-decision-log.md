# Decision log — superpowers v6 adoption scope (genesis, targets the seed)

Thread: docs/threads/260706070546Z-superpowers-v6-sync/
Target: seed/seed.md
Subject: deciding which upstream obra/superpowers v6 changes (Subagent-Driven Development, Writing Plans) to adopt into implement-plan-with-subagents, plan-strict, and possibly plan-loose, before writing a spec.

## P1: Merge the two per-task reviewers into one dispatch

Point: the deepest structural fork: do we follow upstream and merge our two per-task reviewers into one?

What you need to know: Upstream's merge isn't an isolated simplification — it's half of a rebalance. They narrowed the per-task gate (one reviewer, two verdicts, one diff read) *and* added a broad whole-branch review at the end on the strongest model. Their stated core principle: "Fresh subagent per task + task review (spec + quality) + broad final review = high quality, fast iteration." Notably, their v5 code-quality reviewer was thin (it mostly pointed at their generic code-reviewer doc), so for them the merge lost little. Ours is different: we have two substantive method files (`references/plan-compliance-reviewer.md`, `references/code-quality-reviewer.md`) with strict lane separation, and our fix loop re-runs only the failed pass — including a code-quality re-run after baseline-gate fixes. Also relevant: our workflow already externalizes the "broad review" — `review-implementation` and `review-code` exist as separate skills invoked after implementation — so the compensating mechanism upstream had to *add*, we already have at workflow level.

Decision: Merge the dispatch, keep the lanes. Each orchestration cycle dispatches ONE reviewer subagent that loads BOTH existing method files (`references/plan-compliance-reviewer.md` and `references/code-quality-reviewer.md`) and must return the two verdicts separately — plan-compliance and code-quality — each judged strictly within its own lane. A task passes only when both verdicts are clean; the fix loop keeps routing by which verdict failed.

Rationale: The dispatch-cost argument for merging is real, and upstream's at-scale experience is exactly the evidence this thread was opened to use — but our two method files are substantive lane definitions worth preserving as-is rather than blending into one rubric. Merging only the dispatch halves reviewer subagent runs and diff reads while keeping each lane's method auditable and the fix loop's per-verdict routing intact. Trade-off accepted: one reviewer context reads both lenses, so attention independence between lanes is partly lost.

## P2: No internal broad final review

Point: the other half of upstream's rebalance: do we add a broad whole-branch final review step inside `implement-plan-with-subagents`?

What you need to know: Upstream's per-task narrowing is explicitly compensated by a one-time broad review of the entire branch at the end, dispatched on the most capable available model, with all its findings batched into ONE fix subagent (their data point: a per-finding fix wave once cost more than all the tasks combined). In our workflow the equivalent already exists *outside* the implement skill: `review-implementation` (verifies against spec acceptance criteria) and `review-code` (whole-change quality review) are separate skills a session invokes after implementation, and our repo rules forbid coupling one skill to another by name anyway. Also worth noting: our per-task fix loop already complies with their batching lesson — a fresh implementer receives the reviewer's whole report at once, never one dispatch per finding.

Decision: Do NOT add an internal final-review step to `implement-plan-with-subagents`. The broad review stays externalized in the workflow's separate post-implementation review skills. Add one natural-language sentence to the skill (in or near the implementation-report section) noting that per-task reviews are deliberately task-scoped gates and the implementation as a whole is expected to receive a broader review afterward — awareness without naming any skill. No change to fix-loop batching, which already batches by reviewer report.

Rationale: Duplicating the broad review inside the skill would double-review every implementation and violate the repo's skill self-containment rule (no coupling by skill name). Upstream added the broad final review because they had nowhere else to put it; our workflow already stages it separately. The added sentence preserves upstream's insight — per-task gates are intentionally narrow — for a reader of the skill alone.

## P3: Deprecate plan-loose

Point: what happens to `plan-loose`?

What you need to know: `plan-loose` v3.0.0 is a near-verbatim twin of `plan-strict` differing only in task body shape (3-sentence ceiling vs six mandatory fields). It's positioned for "human-leaning implementers and exploratory work." The user has never used it. It is, however, published: distributed via skills.sh, listed in README and `marketplace.json`, and referenced structurally by `adjust-plan-granularity` (whose whole purpose is converting between the two granularities) and by both implement skills ("both granularities are valid input"). The repo has an established retirement path — `skills/deprecated/` plus the `JeisKappa-deprecated` marketplace plugin — so removal doesn't break existing installs. If strict goes multi-file while loose stays single-file, every plan consumer needs to handle two structurally different plan artifacts, not just two prescriptiveness levels. (Context from discussion: the low-ceremony niche plan-loose was built for is already better served by the `implement` skill, which takes unstructured input and needs no plan artifact at all.)

Decision: Deprecate `plan-loose` — move it to `skills/deprecated/`, update README and `marketplace.json` accordingly. The `implement` skill covers the low-ceremony niche. All plan consumers (`implement-plan-with-subagents`, `implement-plan`, `review-plan`) then assume ONE plan structure and drop their loose-granularity branches (inference paths, the "orchestrator must track touched files" fallback, per-granularity review rigor).

Rationale: The user's own usage says the loose slot never gets used — when a plan is worth writing, strict wins — and unstructured work goes to `implement` directly. Deprecation dissolves the only listed con of the multi-file strict plan (structural divergence between the two plan skills) and permanently simplifies every downstream plan consumer. Flagged scope growth: the thread's seed was "sync upstream v6 lessons," and deprecating a skill goes beyond it — accepted deliberately because the multi-file plan decision forces the loose question. Follow-on accepted as open: `adjust-plan-granularity` loses its loose endpoint and must be reworked or deprecated (next point). Residual risk accepted: no telemetry exists, so an unknown external user of plan-loose can't be ruled out; the deprecated bucket keeps it installable.

## P4: Deprecate adjust-plan-granularity with no successor

Point: the fate of `adjust-plan-granularity` — repurpose it into a plan-task surgery skill, or drop it entirely?

What you need to know: With loose gone, the old skill's core vocabulary (`looser`/`stricter`) is dead. The user's original motivation for the skill was readability: an older workflow produced 30+-task plans too long to read, and the skill existed to compress them. The user then realized the plan needs no human approval gate at all — the spec is the audited artifact; the plan is disposable compiler-IR with no frontmatter and no status latch, compiled by `plan-strict` and checked by `review-plan`, which already auto-fixes plan-fault deviations and routes spec-fault findings to the human. So the plan's quality loop is fully agentic, and a human-invoked surgery skill only has a seat if a human routinely reads plans — but the monolith-readability problem is exactly what the multi-file plan shape plus authoring-time Task Right-Sizing dissolves. Correction kept from discussion: autonomy means no skill may *depend* on a human reading or editing a plan, not that plans are invisible — `review-plan` still routes spec faults to the human, halts still surface, and the index file remains a skim surface. Verified detail: the literal phrase "plan-adjustment pass" appears in `implement-plan-with-subagents/SKILL.md` (lines 236, 242) and identically in `implement-plan/SKILL.md` (lines 155, 161), already framed as "edited in place by its own authoring/adherence loop" but with "the user re-shapes the plan" wording.

Decision: Deprecate `adjust-plan-granularity` with NO successor skill (no `adjust-plan-tasks`). Upstream's Task Right-Sizing rubric lands in `plan-strict` (authoring rubric, folded into the existing self-review's over/under-splitting checks) and `review-plan` (checking rubric). Both implement skills (`implement-plan-with-subagents` lines 236/242, `implement-plan` lines 155/161) get their deviation-policy wording adjusted: plan revision happens upstream — a spec fault means fix the spec and recompile the plan; a plan fault means revise the living plan by re-running planning or editing it in place under the Plan Artifact Contract — replacing the "the user re-shapes the plan in a separate plan-adjustment pass" framing. No named skill reference is added (self-containment).

Rationale: The skill compensated for a defect in the plan format (unreadable monoliths), and this thread fixes the format instead. Revising compiler-IR is just recompiling — a dedicated published skill around that is ceremony. The plan pipeline (plan-strict → review-plan → implement) is fully agentic with human touch points only at spec approval and surfaced findings, so a human-invoked surgery skill has no seat in it. This reverses the earlier in-discussion proposal to rename/repurpose the skill; the user's autonomy realization was the new fact that changed the recommendation.

## P5: Multi-file strict plan — index plus per-task files

Point: the strict plan's file shape and what each subagent reads — confirming the concrete multi-file design (extracted `.wip/` briefs lost their reason to exist once the plan can be born as briefs).

What you need to know: The current grammar is `plans/NNN[-<desc>]/plan.md`, one file, alive in place, no frontmatter — assumed by `plan-strict`, `review-plan`, both implement skills, and the V2 thread-layout docs. Whatever shape we pick, the index must stay authoritative about task count and order (an index that says 12 tasks while the folder holds 13 is the new drift risk this shape introduces — consumers must treat the index as the source of truth and flag mismatches). Upstream's Global Constraints block matters here because it's the one piece of *plan-level* context an isolated task-file reader would otherwise lose.

Decision: Adopt the multi-file strict plan shape:
- `plans/NNN[-<desc>]/plan.md` remains, as the **index**: plan-level objective and context, the Global Constraints block, and an ordered task list — number, title, one-line objective, relative pointer to the task file. The V2 layout grammar keeps "the plan" as `plan.md`.
- Tasks live at `plans/NNN[-<desc>]/tasks/NN-<kebab-slug>.md`, two-digit ordinal, one file per task, each carrying the full strict field set. Ordering stays implicit in the numbering; the no-parallelization contract is untouched.
- Implementer dispatch = task file path + index path (+ reviewer findings on fix-loop respawns); implementers read nothing else from the plan.
- Reviewer dispatch = the same two files + the diff.
- Alive-in-place and no-frontmatter apply to the whole `plans/NNN/` folder; git remains the only history.
- No per-task-file frontmatter and no status markers in task files — execution state lives in commits + the implementation report, so the plan stays a stateless IR and the index never needs updating mid-run.
- Ripple: `plan-strict` (emits the shape), `review-plan` (reads index first, then tasks one at a time), `implement-plan` and `implement-plan-with-subagents` (parse index, dispatch/read per task file), V2 thread-layout doc.

Rationale: The plan born pre-briefed beats upstream's runtime brief extraction — no extraction step, no duplicated text, no drift between plan and brief; per-task context cost stays flat for every fix-loop respawn; scope bleed from whole-plan reading disappears; and the index doubles as the human skim surface when a halt or review finding surfaces. Accepted trade-off: index/task-file consistency is a new invariant consumers must check, and plan authoring/consumption touches multiple files.

## P6: Mandatory Consumes/Produces field in strict tasks, notation-agnostic

Point: the last Writing Plans v6 addition: the per-task Interfaces block (`Consumes:` / `Produces:` with exact signatures).

What you need to know: Upstream added it for exactly the situation P5 just created: an implementer who sees only its own task can't discover what names, types, and signatures neighboring tasks established or expect — someone has to tell it, at plan time. Our `plan-strict` task format already has an "Input/context" field, but it's prose-level ("upstream deps, cite decisions by path + P<N>"), not signature-level. The two aren't redundant: Input/context says *why and from where*, the hand-off contract says *the exact thing* — a named, checkable artifact rather than "builds on task 3's parser". Side benefit: an explicit contract gives `review-plan` a mechanical cross-task check — every consumed entry must match some earlier task's produced entry — which is precisely the kind of index/task consistency verification the multi-file shape wants anyway.

Decision: Add a seventh mandatory element to the strict task format: the two lines `Consumes:` (exact things this task uses from earlier tasks) and `Produces:` (exact things later tasks will rely on) — no "Interfaces" umbrella heading. `none` is an explicit legal value for either line. The rule is precision, not notation: each entry names the exact thing in whatever notation is native to it — a function in the target language's own signature style, a CLI invocation as the literal command line, an HTTP endpoint as method + path, a file as its path and format; natural-language entries are fine as long as a later implementer could use the named thing without guessing its name or shape. `review-plan` gains the matching cross-task check: every non-`none` `Consumes:` must correspond to an earlier task's `Produces:`; dangling `Produces:` entries are flagged as a smell, not an error (a final task legitimately produces for the world outside the plan).

Rationale: Isolated task-file implementers (P5) need cross-task names pinned at plan time or independent tasks won't converge on the same identifiers. Mandatory-with-explicit-`none` beats optional because optional fields rot, while `none` forces the planner to actually consider the question at zero fabrication cost. The user's constraint — don't bake in TypeScript's Interface concept — is honored by dropping the "Interfaces" name and making exactness, not typed syntax, the bar.

## P7: Subagent scratch in thread .wip/, write-once file per dispatch, capped replies

Point: file-handoff discipline, remaining half: what subagents send back to the orchestrator, and where their scratch files live.

What you need to know: Upstream's v6 rule caps the subagent's chat return (full report to a file; ≤15-line summary back), because everything a subagent prints back stays resident in the controller's context for the rest of the run. Grounded reading of our skill showed the current design is already file-first: reviewer files go to "a `.wip/` scratch file under the active thread"; `.wip/` is recursively gitignored and "never used for emitted artifacts"; the implementer's outcome file is deliberately CONDITIONAL — written "ONLY when there is diff-blind content to persist," because the working tree is the primary output and the orchestrator inspects `git status`/`git diff` itself rather than trusting reply prose; fix-loop dispatches are already "disambiguated by the stamp alone" (new file per dispatch). Upstream needs a mandatory report because their controller reads nothing else; our orchestrator reads the diff, so a mandatory report would re-narrate the diff.

Decision: Three parts. (1) Subagent scratch stays in the thread's gitignored `.wip/` — thread-scoped so a halted run resumed by a fresh session finds it; per-cycle commits stage only listed task files so scratch never enters history; and because `.wip/` is disposable, no durable artifact may point into it — anything a durable record needs is copied into that record. (2) Every dispatch writes to a NEW orchestrator-named file, write-once, never overwriting or appending a prior dispatch's file — preserving the per-iteration provenance the fix-loop convergence audit relies on. (3) Keep the conditional-file design (outcome/review files only when there is content) and formalize the reply cap: a fixed short return shape — status/verdict token(s), file list, one-line verification result, concerns flag, scratch-file path if written — with a hard line ceiling, for implementers and the merged reviewer alike. This walks back the earlier in-discussion idea of a mandatory implementer report.

Rationale: Adopts upstream's context-economics lesson (capped returns) without importing the part our design already solves better (working-tree-as-ground-truth makes a mandatory report pure duplication). Write-once per-dispatch files keep the audit trail; thread-scoped scratch keeps resumability. Folder layout within `.wip/` and file naming were split to their own point (P8); the task-report block to P9.

## P8: Task-scoped scratch folders with orchestrator-assigned dispatch ordinals

Point: scratch folder layout and file naming under the thread's `.wip/`.

What you need to know: Today's layout is flat — `<UTC>-task-<N>-implementer-outcome.md` etc. directly in `.wip/` — with nothing disambiguating which plan lineage a file belongs to, and the write-time UTC stamp means the orchestrator's brief can only name a path *template*: the exact path exists only after the subagent replies and cites it. That indirection has no benefit, because the orchestrator is the coordinator and knows exactly how many dispatches each task has had. Deterministic paths also make the fix-loop brief cleaner — the respawned implementer gets the findings file's literal path, pre-computable before the reviewer even returns.

Decision: Adopt the nested, index-named convention (user-proposed NN indexing over UTC stamps):
- Task folder: `.wip/implement/plans-NNN/task-NN/` — `plans-NNN` names the plan lineage; `task-NN` matches the plan's own task ordinal (same number as `tasks/NN-<slug>.md`).
- Inside, one dispatch-sequence counter per task, shared across roles: `SS-implementer-outcome.md`, `SS-review.md`, where `SS` is the dispatch ordinal within the task's cycle, assigned by the orchestrator at dispatch time (e.g. `01-implementer-outcome.md`, `02-review.md`, `03-implementer-outcome.md`, `04-review.md`). Conditional writing per P7 means gaps are normal.
- On a resumed run, the next `SS` is max-existing + 1 — no session state needed.
- Run-level scratch (e.g. a progress ledger) sits beside the task folders at `.wip/implement/plans-NNN/`.

Rationale: Orchestrator-assigned ordinals give exact, pre-computable paths in every brief (replacing stamp templates the orchestrator couldn't resolve up front); the shared per-task counter makes the cycle's chronology readable from filenames alone; plan-lineage namespacing removes ambiguity when a thread carries multiple plan lineages. Per-role counters were considered and dropped — the orchestrator's audit already counts iterations per role, and one counter is the simpler mental model.

## P9: Task-report block moves to a run progress ledger

Point: the per-task report block: keep it synthesized, replace it with pointers to review files, or move it?

What you need to know: After each cycle the orchestrator writes an 8–15 line structured block (SKILL.md:52–68, 113): `Task <N>: implementer claimed <X> / verified <Y>`, a subagent audit (dispatches per role, fix iterations per review pass), 1–3 sentences of notes, and a `Next:` line. Today it goes "in chat output and/or the commit message body" — explicitly NOT to any file ("the chat output and the commit history together are the audit trail"), the final implementation report is folded from these blocks, and the final out-message repeats every task's status + audit + commit SHAs — so the same per-task information can appear up to three times in the orchestrator's output. Pure pointers-to-review-files cannot replace the block: (1) the files are conditional — a clean `PASS` writes no file; (2) the files hold findings, not the cycle audit — dispatch counts, claim↔verified divergence, and commit SHA are facts only the orchestrator knows; (3) `.wip/` is gitignored and disposable, so pointers from the durable audit surfaces (commit message, implementation report) would rot. Meanwhile upstream's v6 durable progress ledger ("conversation memory does not survive compaction"; re-dispatching completed tasks was their single most expensive observed failure) wants exactly this information in one durable-within-the-run place.

Decision: Keep the synthesized block, give it a home — merging upstream's ledger lesson with the existing block instead of adding a second overlapping mechanism:
- The orchestrator appends each task's block to a run ledger file at `.wip/implement/plans-NNN/progress.md`, immediately after the cycle's commit — one block per completed task, append-only.
- The commit message body keeps carrying the block (durable, self-contained git archaeology).
- Chat shrinks to one line per task (`Task 04: verified DONE, commit abc1234`).
- The implementation report is folded from the ledger file at the end, re-read from disk — which is also the compaction-recovery path: after context loss, the ledger + `git log` are the resume state, never recollection.
- Content contract: statuses, counts, deviations, unresolved concerns verbatim; resolved findings counted, never restated. Audit line updated for P1's merged reviewer: fix iterations per verdict lane.

Rationale: The block is the only place the claim-vs-verified audit exists, so it must stay synthesized and self-contained — but it was homeless ("chat and/or commit body") and redundantly re-emitted. Anchoring it to an append-only ledger file adopts upstream's compaction-resilience lesson at zero extra mechanism cost, shrinks chat to signal, and makes the final report a disk-fold rather than a memory-fold. This point also absorbs the upstream "durable progress ledger" adoption item — it is decided here.

## P10: No model-selection content in the skill

Point: model selection for dispatches — adopt upstream v6's mandatory explicit per-dispatch model choice ("always specify the model explicitly"; "turn count beats token price") or not?

What you need to know: Our skill says nothing about which model subagents run on, and the user omitted this deliberately already when drawing from v5. Upstream made model choice mandatory per dispatch with tier guidance. Constraints against adopting: our skills are harness-agnostic (distributed via skills.sh to arbitrary runtimes), so naming models or assuming a per-dispatch model parameter exists would rot; model preferences are already expressed through user-voice channels (appended at invocation, or in the user's CLAUDE.md), which harnesses treat as authoritative — skill text would add a third voice requiring arbitration. The one cost of silence: it doesn't codify "assume nothing" — an orchestrator could in principle improvise a down-tier on its own initiative, invisibly, on the most damage-sensitive dispatch (a reviewer).

Decision: Do not mention model selection in the skill at all — pure silence, as today. Model choice is a final-user concern expressed at invocation time or in the user's own configuration; the skill defines process, not resource allocation. The upstream section is not adopted in any form, including the considered one-line "model choice belongs to the user" guard.

Rationale: Separation of concerns (process vs deployment policy), no arbitration conflicts with user-voice channels, ages and travels better across harnesses — and empirically the current skill worked fine this way. Note: recommended (weakly) the one-line negative guard codifying "never re-tier a dispatch on your own initiative"; user chose pure silence on the grounds that observed behavior never showed improvisation and any model-talk in the skill invites drift. Accepted trade-off: the improvised-down-tier hazard remains fenced only by harness defaults, not by skill text.

## P11: Mechanical pre-flight only, plus orchestrator reads the plan's source artifact

Point: upstream's "Pre-Flight Plan Review": before task 1, scan the whole plan once for internal contradictions or plan-mandated content a reviewer would flag, and batch all questions to the human up front — adopt it?

What you need to know: Upstream needs this because nothing else in their pipeline reads the plan critically before execution. Ours does: `review-plan` exists precisely to check a plan against its spec, auto-fix plan faults, and route spec faults to the human — and the implement skill can't know whether `review-plan` ran, because plans deliberately carry no status frontmatter (no approval gate, by design). But P5 created something upstream's pre-flight accidentally covers and we now genuinely need: the index↔task-files consistency invariant — an index claiming 12 tasks over a folder holding 13, dangling pointers, non-contiguous ordinals. That's structural integrity, not semantic review, and it's cheap. Context economics: with P5 the orchestrator may never need full task-file contents in its own context (subagents read those); a semantic pre-flight would force reading everything up front, undoing part of P5's benefit. Discussion also surfaced a real P5 regression risk: the orchestrator used to read the whole plan and now reads only the index — but its judgment calls (gating terminal claims, verifying empty-diff `DONE`, escalating concerns, writing the implementation report) need intent-level context the index alone can't give.

Decision: Two parts. (1) Adopt a MECHANICAL pre-flight only, added to the existing pre-run gates: after resolving the plan lineage, verify the index and the `tasks/` folder agree — every index entry points to an existing task file, every task file is listed in the index, ordinals are contiguous and match filenames — and halt with a clear message before any dispatch if not (a malformed artifact, not a `BLOCKED` task). The semantic contradiction scan and batched-questions step are NOT adopted — that is `review-plan`'s lane; mid-run discoveries keep routing through the per-task `NEEDS_CONTEXT` protocol. (2) The plan index gains a `Source:` line (amending P5's index shape; `plan-strict` emits it): a thread-relative pointer to the upstream artifact the plan was compiled from (spec, proposal, decision log, issue URL, or `none — raw prompt`). At startup the orchestrator reads the index AND the source artifact when one exists — the orchestrator holds the intent (spec), subagents hold the mechanics (task files) — while task files stay lazily read (staging needs each task's `Files modified`; a judgment call may read a task file on demand).

Rationale: Structural integrity must be checked before dispatch one because a broken index invalidates everything downstream, while semantic review duplicated a dedicated pipeline stage. The source-artifact read repairs the orchestrator's post-P5 context loss at intent level rather than re-importing whole-plan mechanics — a cleaner division than the old whole-plan read. The `Source:` generalization (rather than "read the spec") covers all five plan input forms.

## P12: No cannot-verify relay protocol; explicit read-beyond-diff permission, unverified-concern rule, and no out-of-task findings ban

Point: upstream's reviewer "⚠️ cannot verify from diff" protocol — per-finding unverifiable items that the controller must resolve before marking the task complete — adopt it?

What you need to know: Upstream's reviewer works from a pre-generated diff package under an explicit read-only-review regime, so it genuinely cannot self-serve verification of requirements living in unchanged code — hence the relay to the controller. Grounded reading of our `references/plan-compliance-reviewer.md` showed our reviewer is NOT so constrained: the constraints ban modifying files, running tests beyond the task's verification block, and making findings outside the current task — but nothing restricts *reading*. Our reviewer can open unchanged files itself and mark a criterion SATISFIED or MISSING, so most of the problem upstream's protocol solves does not exist in our design (and the orchestrator has no access the reviewer lacks, so a relay adds nothing). Two residues are real: (1) the read permission is implicit — a timid reviewer can self-handicap; (2) a classification gap for criteria unverifiable by anyone in the run (external gateway config, runtime-only behavior, credentials nobody has): "default to blocking when uncertain" shoves those toward `ISSUES` (misfired fix loop — nothing to fix) or the run-killing `NEEDS_CONTEXT`. Discussion also surfaced that the existing "DO NOT make findings outside the current plan task" constraint is itself too restrictive — a reviewer that stumbles on a critical gap elsewhere should not be forbidden from reporting it.

Decision: Do NOT adopt upstream's relay protocol (no new orchestrator-must-resolve step). Instead, three changes to both reviewer lanes' method files:
1. Explicit read permission: the diff is the review *target*; the repo is readable *context* — reading unchanged code to verify a criterion is in-scope and expected.
2. Unverified-concern rule: a criterion unverifiable from within the run is recorded as a named "unverified" concern — non-blocking by default, riding into `DONE_WITH_CONCERNS` so it stays visible downstream — with `NEEDS_CONTEXT` reserved for when proceeding without the answer would be reckless (reviewer's judgment).
3. Drop the "DO NOT make findings outside the current plan task" ban. Replace with positive framing: the reviewer is told a plan is being implemented in task batches and its focus is task N's diff — no explicit prohibition on reporting other discoveries. Out-of-task observations go in a distinct section of the findings file, never drive this task's fix loop or its blocking classification (blocking remains defined against the current task), and the orchestrator routes them through the existing concern/follow-up mechanisms (task-report block, implementation report's follow-up routing).

Rationale: Our reviewers can self-serve what upstream's cannot, so the right adoption is unlocking that ability explicitly rather than importing machinery built for a more constrained reviewer. The unverified-concern rule closes the real gap (run-external unknowns) without misfiring fix loops or killing runs. Dropping the out-of-task ban follows the user's principle that the skill should be helpful, not a handicap — and aligns with upstream's own v6 anti-pre-judging lesson that reviewer prompts must never suppress findings; the routing note protects task isolation (fix loops and commits stay task-scoped) while legitimizing the report.

## P13: Anti-pre-judging rules for reviewer briefs, plan-mandated finding routing

Point: upstream's anti-pre-judging rules for constructing reviewer briefs — adopt them?

What you need to know: Upstream's v6 rule targets the *controller's* prompt-writing, not the reviewer: if the dispatch prompt contains "do not flag X," "don't treat X as a defect," "at most Minor," or "the plan chose X" — the controller is pre-judging, deciding review outcomes before the reviewer runs. Their companion rule: a finding that conflicts with the plan's own text is the *human's* decision — present the finding and the plan text, never silently accept the defect because "the plan said so" and never silently override the plan. Our skill is already half-aligned: it injects implementer assumptions into reviewer briefs explicitly "unclassified" (SKILL.md:96, 103) — the orchestrator hands over material without pre-rating it. But that's one narrow application; nothing generally forbids the orchestrator from writing severity caps or exclusions into a brief, and P12 just made reviewers freer to report — which is only worth much if the orchestrator can't quietly take it back at dispatch time.

Decision: Add a brief-construction rule to `## Subagent Briefs`, applying to both reviewer lanes: a reviewer brief must never pre-rate a finding's severity, exclude categories of findings, or declare a question settled — with the red-flag phrasings ("do not flag…", "at most minor…", "already decided/the plan chose…") listed as examples of what the orchestrator must not write; the existing "unclassified" injection stays and is cited as the pattern to follow. Companion rule: a finding that traces to the plan's own text (the diff faithfully implements something defective the plan mandated) is labeled *plan-mandated* by the reviewer, and the orchestrator routes it through the existing plan-fault path (surface via the deviation policy — `DONE_WITH_CONCERNS`/`NEEDS_CONTEXT` per severity), never silently accepting it because the plan said so and never patching the immutable plan mid-run.

Rationale: This is the dispatch-side complement of P12's reviewer-side freedom — without it, the orchestrator could suppress at brief-writing time what the reviewer is now free to report. The plan-mandated label reuses the existing deviation-policy routing rather than inventing a new escalation channel, and preserves the plan's mid-run immutability. Closes the upstream v6 adoption inventory.

## P14: implement-plan (single-agent) ripple scope

Point: `implement-plan` (single-agent) consumes the same plan artifacts — which of this log's decisions carry over to it?

What you need to know: `implement-plan` is the single-agent counterpart: same plan input contract, same four-state protocol, same per-task commit cadence and implementation report, but no subagents — one session implements and self-reviews. Plan-format decisions hit it unavoidably; orchestration decisions vary — some are subagent-specific machinery, some are run-durability mechanisms that apply to any long run (compaction risk is arguably worse single-agent, since one context does everything and upstream's "most expensive failure" — re-doing completed tasks after context loss — doesn't care who did the tasks).

Decision: Three groups.
- Mandatory carry-over (plan-format driven): multi-file plan handling (parse index, read task files itself), the mechanical pre-flight (P11), the `Source:` read at startup, removal of loose-granularity branches (P3), and the P4 deviation rewording (already logged for it).
- Carry over with adaptation: the P12 self-review changes — unverified-concern classification and positive focus framing instead of any out-of-task findings ban — and P2's awareness sentence (its per-task self-review is a task-scoped gate; broader review happens downstream).
- Carry over the progress ledger (P9): same `.wip/implement/plans-NNN/progress.md`, append after each cycle's commit, chat one-liner per task, implementation report folded from disk — without the subagent-audit lines.
- Excluded: P1 (no reviewers to merge), P7/P8 dispatch scratch conventions (no dispatches; only the ledger lives in the P8 layout), P13 (no briefs), P10 (already silent).

Rationale: Keeps the two implement skills symmetric wherever the mechanism is about plan format or run durability, and asymmetric only where the mechanism is genuinely about subagent dispatch. The ledger carry-over was the one judgment call, decided by the compaction argument applying at least as strongly single-agent.

## P15: Keep the Global Constraints block, add explicit on-demand source access for subagents

Point: reopening the constraints question: keep the Global Constraints block in the index (P5, verbatim copy), or drop it and have the subagents read the entire spec instead?

What you need to know: The tension is duplication vs. context economics. The copied block is a transcription step — if the spec is later revised without recompiling the plan, the copy drifts (bounded in practice: plans are disposable IR, and `review-plan` checks plan against spec, which would catch stale constraints — but the hazard is real). Reading the spec directly gives subagents richer intent context and constraints first-hand. On the other side: the P5 architecture exists to keep per-dispatch reading small — and the multiplier is large: a 12-task run with fix iterations means roughly 25–40 dispatches, so mandating spec reads means 25–40 full-spec reads per run, where the spec is typically the longest artifact in the thread. Mandatory spec reading would also partially re-import scope bleed and un-adopt the central upstream v6 lesson (they moved *away* from broad per-dispatch context and added Global Constraints precisely to compensate). Not every plan has a spec (five input forms), so pure spec-reading needs the index block as fallback anyway, leaving two mechanisms. P12 already established reviewers may read anything in the repo, and nothing bans implementers from reading thread artifacts — the `Source:` line sits in the index both already receive, so on-demand access exists in the design for free. Verified upstream of this point: the `spec` skill's semantic contract mandates a Constraints element (element 5 of eight, spec/SKILL.md:34), so the verbatim copy always has a guaranteed source when the plan compiles from a spec; for the other input forms, `plan-strict` copies what constraints the source states.

Decision: Keep the Global Constraints block in the index (baseline, verbatim, all input forms), and add explicit on-demand source access to the subagent briefs: the index's `Source:` artifact is readable context — consult it when the task file and index leave a question open — extending P12's read-permission framing from reviewers to implementers too. No mandatory per-dispatch spec reads.

Rationale: A-grade economics with B-grade ceiling: the common case stays cheap, the hard case has an explicit sanctioned escape hatch, and it removes the one thing genuinely wrong with the P5 status quo — the unstated implication that subagents shouldn't look beyond their brief. The honest core of the drop-the-block position (constraints belong to the spec) is preserved by keeping the block verbatim-copied and review-checked rather than hand-summarized. Accepted trade-offs: a small duplication remains, and knowing *when* to look deeper relies on subagent judgment.
