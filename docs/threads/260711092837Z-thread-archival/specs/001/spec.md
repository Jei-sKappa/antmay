# Spec — Thread archival (`archive-thread` skill + `finish` nudge)

## 1. Intended outcome

A new workflow skill, **`archive-thread`**, that relocates finished or intentionally-abandoned workflow threads out of the crowded active `docs/threads/` listing into a dedicated `docs/threads/archive/` folder, so active threads are easy to tell apart from closed ones — **without breaking the ability to follow existing cross-thread references.** Plus a small, natural-language addition to the existing `finish` skill that offers archival at the moment a thread closes.

When this is implemented, a maintainer can:

- run `archive-thread` with no input and be shown every closed thread, then archive the chosen ones in one sweep; or
- run it against one or more named threads to archive them directly; or
- run it to move a thread back out of the archive (un-archive);

and in every case the operation is a pure folder move that leaves cross-thread references still resolvable.

## 2. Context

Threads accumulate under `docs/threads/<YYMMDDHHMMSSZ-slug>/`. Over time the directory grows and finished/abandoned threads become indistinguishable from active ones. The obvious fix — move closed threads into an archive folder — risks breaking cross-thread links, which today are written as full repo-relative paths (`docs/threads/<slug>/…`) and would go stale on a move.

This spec is forward-designed from the genesis discussion for this thread. Every design fork was settled there; this spec references those decisions rather than re-deciding them. The authoritative source is:

`discussions/260711122736Z-thread-archival-design-decision-log.md` (decisions P1–P7).

The load-bearing reframing (per that log, P5) is that "link integrity" is satisfied by **link *resolvability***, not literal-path validity: because a thread's slug is globally unique and travels with its folder (archival only *prefixes* the path with `archive/`, never renames the slug), any stale `docs/threads/<slug>/…` reference still contains the unique slug and is resolvable in a single grep. This is why the skill does **no** link rewriting.

## 3. Scope / non-scope

**In scope:**

- Authoring a new standalone skill `archive-thread` (a `SKILL.md` natural-language procedure — not a script, not a workflow-convention doc) and fully integrating it into the repo per this project's "adding a new skill" conventions (per log P1).
- The skill's two operating modes, closed-thread detection, the archival move, and the inverse (un-archive) move (per log P2, P4, P5, P6, P7).
- A natural-language addition to the existing `finish` skill (`skills/workflow/finish-navigate/finish/SKILL.md`) offering archival after its handshake (per log P3).

**Out of scope (explicitly NOT to be built):**

- **No link rewriting.** The skill MUST NOT scan for, edit, or rewrite any inbound `docs/threads/<slug>/…` reference, and MUST NOT edit any file inside any *other* thread. (Per log P5 — this supersedes the rewrite step originally sketched in P1.)
- **No writes into the archived thread.** No "archived" ledger event, no marker file, no frontmatter change — the ledger grammar has no archival event and archival is a location change, not a lifecycle transition (per log P6).
- **No commit.** The skill does not stage-and-commit, push, or branch; it leaves the `git mv` for the surrounding session to commit (per log P6).
- **No change to the cross-thread reference convention.** Switching references to a location-independent form is a separate initiative, tracked in its own thread (`docs/threads/260711131650Z-location-independent-thread-references/`); it is not part of this work.
- **No hand-editing of the Raycast manifest** (`raycast-extension/assets/skills.json`) — it is a derived file regenerated by the sync script and picks up the new skill automatically.
- **No date sub-bucketing** of the archive (per log P4).

## 4. Expected behavior

### 4.1 The `archive-thread` skill — inputs and modes (per log P2)

The skill accepts input in two modes:

- **Explicit-target mode** — the user names one thread or a list of threads (by slug or path). The skill archives each named thread.
- **No-input mode** — the skill scans all threads, collects those whose ledger disposition is closed, presents the list, and archives only after the user confirms.

There is **no** date-range selector (a day's worth of threads is just a list the user passes).

### 4.2 Closed-thread detection (per log P2)

A thread is "closed" iff the **last disposition line** in its `ledger.md` is `closed: done` or `closed: dropped`. A thread with a last disposition of `deferred`, or with no disposition line at all, is **not** closed (deferred is a reversible pause; absence means active). The scan reads each thread's `ledger.md` and applies exactly this rule.

The scan MUST skip the reserved `docs/threads/archive/` container itself and anything beneath it (already-archived threads are not re-listed). Real thread folders begin with a digit (the UTC stamp), so the literal `archive/` entry is distinguishable and excluded.

### 4.3 The confirm-on-non-closed guard (per log P2)

In explicit-target mode, if a named thread is **not** closed (active or deferred), the skill MUST warn the user and require explicit confirmation before archiving it — it neither archives silently nor refuses outright (a deliberate override, e.g. shelving a dropped-but-unmarked thread, remains possible after the warning).

### 4.4 The archival move (per log P4, P5, P6)

Archiving a thread is a **pure folder move**: `docs/threads/<slug>/` → `docs/threads/archive/<slug>/`, performed with `git mv` so git records the rename. The archive is **flat** — no year/month/day sub-buckets. The slug is preserved unchanged (`archive/` is only a prefix). The move is the skill's **only** side effect on disk: it writes nothing into the moved thread and touches no other file.

### 4.5 Un-archive (the inverse move) (per log P7)

The skill can also perform the inverse: move a thread back out of the archive, `docs/threads/archive/<slug>/` → `docs/threads/<slug>/`. It is the same `git mv` reversed; no separate procedure is needed.

### 4.6 Guards by agent judgment; minimal skill (per log P7)

The skill stays small. It does **not** hard-code idempotency/validation machinery for obvious mistakes (e.g. re-archiving an already-archived thread, un-archiving a thread that is not archived); it relies on the running agent's judgment to notice and avoid those, and to apply the §4.3 confirm-on-non-closed guard. The essential job is one `git mv` (and its reverse).

### 4.7 The `finish` nudge (per log P3)

The `finish` skill gains a **natural-language** offer, after its terminal handshake, noting that the now-closed thread may be archived. This mention MUST NOT name the `archive-thread` skill, reference its path, or state an invocation condition — it is awareness in prose only, leaving a capable agent to act on it. Archival is **not** folded into `finish` as a handshake step; `archive-thread` remains the standalone owner of the procedure.

## 5. Constraints

- **C1 — Skill, not tooling.** The deliverable is a natural-language `SKILL.md`. No script, no new toolchain, no CI (per log P1).
- **C2 — Self-containment.** The `archive-thread` `SKILL.md` MUST obey this repo's skill self-containment rules: a one-sentence `description` (what it does + when to trigger it); no "when to use this skill" body section; no coupling to other skills by name/path/invocation-condition; no links to files outside its own folder; restate any needed rule inline. The `finish` edit MUST likewise not name or path `archive-thread` (this is the same rule that shapes §4.7).
- **C3 — Frontmatter format.** The new skill's frontmatter matches the repo template: `name: archive-thread` (matching its leaf directory name), a one-sentence `description`, and `metadata` with `author: https://github.com/Jei-sKappa` and `version: 1.0.0` (new skills start at `1.0.0`).
- **C4 — Repo registration (adding a new skill).** The new skill MUST be registered everywhere this repo requires: (a) `SKILL.md` at `skills/workflow/<group>/archive-thread/`; (b) a section in `README.md` under the matching group heading with description + `npx skills add …` install snippet linking the full nested path; (c) an entry `./skills/workflow/<group>/archive-thread` in the group's plugin `skills` array in `.claude-plugin/marketplace.json` (and, only if a brand-new group is introduced, a new `JeisKappa-<group>` plugin added in alphabetical order with `JeisKappa-deprecated` kept last); (d) `archive-thread` added to `conventionalCommits.scopes` in `.vscode/settings.json`, keeping that array alphabetically sorted.
- **C5 — `finish` version bump.** Editing `finish` is a behavior change, so its frontmatter `version` MUST be bumped from `2.0.0` per the repo's semver convention (a minor bump).
- **C6 — Move via `git mv`.** The relocation uses `git mv` (not a plain filesystem move) so the rename is tracked; the archive destination directory must exist for the move to land under `docs/threads/archive/`.
- **C7 — Never rewrite history / never force-push.** Nothing in this work rewrites git history or force-pushes; the skill does not commit at all (see §4.4, §6).
- **C8 — Edit the repo source of truth.** All edits target the repo's `skills/**/SKILL.md` sources (e.g. `skills/workflow/finish-navigate/finish/SKILL.md`), not any installed copy under `~/.claude/skills/`; installed and Raycast copies are regenerated downstream.

## 6. Commit policy

The `archive-thread` skill does **not** auto-commit (per log P6): it performs the `git mv` (leaving the rename staged) and stops; committing is the surrounding session's decision. The implementation work of authoring/registering the skill files is likewise not committed by this spec's executor unless the surrounding session decides to.

## 7. Explicit decisions (index)

All settled in `discussions/260711122736Z-thread-archival-design-decision-log.md` and inlined above where operative:

- **P1** — standalone `archive-thread` skill (not script, not convention doc). §1, §3, C1.
- **P2** — two input modes; no date selector; closed = ledger last disposition `closed: done`/`closed: dropped`; confirm-on-non-closed. §4.1–§4.3.
- **P3** — `finish` gets a natural-language archival offer only, no coupling. §4.7, C2.
- **P4** — archive at `docs/threads/archive/<slug>/`, flat; scan skips `archive/`. §4.2, §4.4.
- **P5** — move-only, no link rewriting; resolvability via slug-as-durable-locator. §2, §3 (non-scope), §4.4.
- **P6** — pure move: writes nothing into the thread, does not commit. §3 (non-scope), §4.4, §6.
- **P7** — un-archive in scope (inverse move); minimal skill; guards by agent judgment. §4.5, §4.6.

- **AC-1.4** `.claude-plugin/marketplace.json` lists `./skills/workflow/<group>/archive-thread` in the `skills` array of the plugin matching its group; if a new group was introduced, a `JeisKappa-<group>` plugin entry exists, plugins remain alphabetically sorted, and `JeisKappa-deprecated` is last.
- **AC-1.5** `conventionalCommits.scopes` in `.vscode/settings.json` includes `archive-thread`, and the array remains alphabetically sorted.
- **AC-1.6** The `SKILL.md` body contains no reference to another skill by name/path/invocation-condition, no "when to use this skill" section, and no link to any file outside its own folder.

### FR-2 — Input model: two modes (P2)

- **AC-2.1** The skill body specifies an explicit-target mode accepting a single thread or a list of threads.
- **AC-2.2** The skill body specifies a no-input mode that scans all threads, lists the closed ones, and archives only after user confirmation.
- **AC-2.3** The skill body does not define any date/day range selector.

### FR-3 — Closed-thread detection (P2, P4)

- **AC-3.1** The skill body defines "closed" as: the last disposition line in the thread's `ledger.md` is `closed: done` or `closed: dropped`; and states that `deferred` and no-disposition are NOT closed.
- **AC-3.2** The skill body states that the scan skips the reserved `docs/threads/archive/` container and its contents (already-archived threads are not re-listed).

### FR-4 — Confirm-on-non-closed guard (P2)

- **AC-4.1** The skill body states that, in explicit-target mode, a named thread that is not closed triggers a warning and requires explicit user confirmation before archiving — neither silent archival nor outright refusal.

### FR-5 — The archival move is a pure `git mv` to a flat archive (P4, P5, P6)

- **AC-5.1** The skill body specifies the move `docs/threads/<slug>/` → `docs/threads/archive/<slug>/` performed with `git mv`, preserving the slug unchanged.
- **AC-5.2** The skill body specifies the archive is flat (no year/month/day sub-buckets).

### FR-6 — No link rewriting, no writes into the thread (P5, P6)

- **AC-6.1** The skill body explicitly states it does NOT scan for, edit, or rewrite inbound `docs/threads/<slug>/…` references and does NOT edit any file in any other thread.
- **AC-6.2** The skill body explicitly states it writes nothing into the moved thread (no ledger event, no marker file, no frontmatter change).
- **AC-6.3** The skill body records the resolvability rationale (the unique slug travels with the folder, so stale references remain grep-resolvable) so a future maintainer does not "helpfully" re-add rewriting.

### FR-7 — No auto-commit (P6)

- **AC-7.1** The skill body states it does not commit, push, or branch; it leaves the staged `git mv` for the surrounding session.

### FR-8 — Un-archive and minimalism (P7)

- **AC-8.1** The skill body states it can also perform the inverse move (`docs/threads/archive/<slug>/` → `docs/threads/<slug>/`) as the same `git mv` reversed.
- **AC-8.2** The skill body relies on agent judgment for obvious-mistake guards (re-archiving an already-archived thread, un-archiving a non-archived thread) rather than hard-coded validation machinery, and the body stays focused on the move rather than defensive branching.

### FR-9 — The `finish` natural-language nudge (P3, C2, C5)

- **AC-9.1** `skills/workflow/finish-navigate/finish/SKILL.md` includes, after its handshake, a natural-language mention that the now-closed thread may be archived.
- **AC-9.2** That mention does not name `archive-thread`, does not reference its path, and states no invocation condition; archival is not added as a handshake step.
- **AC-9.3** `finish`'s frontmatter `version` is bumped above `2.0.0`.

## 9. Degrees of freedom

The *what* above is pinned. The following *hows* are explicitly left to the implementer:

- **DoF-1 — Which group the skill lands in.** `finish-navigate` is the recommended, natural home (thread-lifecycle housekeeping alongside `finish`, `record-verdict`, `whats-next`), and choosing it means no new plugin heading is needed. The implementer MAY choose another existing group or create a new one, provided the C4 registration steps are followed for whichever group is chosen. (This placement was not settled in discussion; it is granted as open here rather than silently pinned.)
- **DoF-2 — Exact wording, structure, and section order of the `archive-thread` `SKILL.md` body**, provided all §8 ACs hold.
- **DoF-3 — Exact wording of `finish`'s archival offer**, provided it stays natural-language and non-coupling (AC-9.1, AC-9.2).
- **DoF-4 — The exact `git mv` invocation and how the `docs/threads/archive/` destination directory is ensured to exist** (e.g. creating it first), provided the result is the tracked move in AC-5.1.
- **DoF-5 — The exact `description` sentence and the `finish` version-bump number** (any bump above `2.0.0` consistent with the change's size).
- **DoF-6 — How the skill presents the closed-thread list and phrases its confirmation prompts.**
