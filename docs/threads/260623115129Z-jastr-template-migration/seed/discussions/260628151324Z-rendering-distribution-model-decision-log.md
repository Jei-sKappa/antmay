# Decision log — jastr template migration (the seed)

Thread: docs/threads/260623115129Z-jastr-template-migration/
Target: the seed (seed/seed.md) — genesis discussion held before any proposal/spec/plan exists
Subject: settling the foundational design decisions for migrating the workflow skill suite to composable jastr templates, ahead of writing a handoff-grade spec. Covers the rendering/distribution model, what gets merged vs. partial-extracted, catalog location and self-containment, input/flag schemas, registry + Raycast sync, freshness checking, and migration scope/sequencing.

## P1: Rendering/distribution model — consumer runs jastr (router) vs. render self-contained skills at authoring time (inline)

Point: The foundational fork that gates everything else — does the consumer run jastr at invocation (thin "router" wrappers + a shipped `.jastr/` catalog), or do we render full self-contained `SKILL.md` files at authoring time and distribute those as today ("inline")?

What you need to know: jastr's router output is a thin wrapper `SKILL.md` that, when invoked, tells the agent to shell out to `jastr run <ref>` and follow the printed Markdown — this is what makes "conditional inputs surface only what's needed" work at agent runtime. But this suite is distributed to consumers who today get one self-contained file via `npx skills add Jei-sKappa/skills --skill <name>` and need nothing else installed; three registries (`README.md`, `.claude-plugin/marketplace.json`, `.vscode/settings.json`) plus a Raycast sync script that reads `SKILL.md` bodies all track every skill. A standing repo rule (`CLAUDE.md`) mandates every `SKILL.md` be self-contained and never link outside its own folder — jastr `include` pulls partials from a shared catalog outside the skill folder. The fork: (A) Runtime "router" wrappers — full power (true single merged skill, routing/includes resolved live), but breaks zero-dependency distribution (every consumer needs jastr + the catalog), breaks the self-containment rule, and breaks the Raycast sync. (B) Build-time "inline" render — templates + partials live in a `.jastr/` catalog used only by the author; a build step renders full self-contained `SKILL.md` into `skills/...` exactly where they sit today; "merge review-* into one template" becomes one template + a `target` flag/variants → 6 rendered `SKILL.md`. Decisive analytical test applied to the suite: a jastr runtime input only beats an in-body `if/else` when the branch value is known to the caller at invocation, because jastr runs once at the start before the agent reads anything — so any branch a skill resolves by doing its job (reading the plan, detecting granularity, inspecting the diff, reading the ledger for tier) can never be a jastr input. Auditing every branching family showed each branch is one of: (a) a distinct entry point with its own `description` (`review-*` by target, `plan-loose`/`plan-strict` by granularity, the `implement` trio) — better as a build-time variant; (b) skill-discovered at runtime (tier from the ledger, review-plan's four outcomes / loose-vs-strict detection) — impossible to pass as a jastr input; or (c) unconditional shared boilerplate (Commit Policy, Immutability, Disposition Frontmatter) — a plain `include`. None is the case runtime resolution is good at (a single-trigger skill with a caller-known parameter whose branches are large enough that pruning justifies the dependency). Two corollaries: the agent does the same classification work either way (router's only gain is removing non-taken branches from context, of which there is almost none within a single already-chosen skill); and collapsing a family to one runtime-routed skill actively harms routing by replacing N sharp descriptions with one generic one.

Decision: Option B — build-time render to self-contained `SKILL.md`, with no jastr dependency at the consumer side. Distinct-entry-point families are authored as one template + variants (one rendered `SKILL.md` per variant); shared boilerplate is extracted into `include` partials. This is now natively supported by `jastr generate agent-skill <ref> --out <path> --mode=inline` (added during this discussion): it emits `---\n<yaml>\n---\n\n<rendered body>` with includes resolved and conditionals evaluated, resolves inputs through `run`'s precedence pipeline, requires `targets.agent-skill`, and supports `--check` with mode-aware messages as the freshness gate. Input flags are valid only in inline mode.

Rationale: The seed frames this as a maintenance/DRY problem ("common parts maintained in one place rather than copy-pasted"), not a runtime-routing one. The branch audit showed no skill in the suite concretely needs or benefits from runtime input resolution, and runtime routing would regress agent routing by genericizing descriptions. B preserves every consumer-facing guarantee that exists today — zero-dependency install, self-contained files, Raycast sync — and keeps the self-containment rule intact (the rendered output is self-contained even though the source catalog is not). The original Option A intuition (the agent receives only the branch it needs) evaporated on inspection: the classifying work is identical and there is almost nothing to prune within a single chosen skill. The decision is not a one-way door — a future genuinely single-trigger, caller-parameterized skill can ship as a router wrapper (consumers would need jastr for that one skill only). The user's native `--mode=inline` support eliminated the only real con (a custom build step), making B both the lower-risk and now the lower-effort path.

## P2: Merge granularity — partials-first (C) vs. merge families into template+variants (B)

Point: Which skill families collapse into one base-template-plus-variants (routing on a locked input via if/else and includes), and which stay standalone, with DRY achieved purely through shared include partials? The spectrum runs from A (aggressive — every family merged), through B (middle — merge where the spine is real and divergence bounded: review's 5 artifact reviewers, plan loose+strict, the implement trio, discussion+seeded-discussion; structurally-distinct skills standalone), to C (conservative — merge nothing; every skill is its own template and all DRY comes from include partials, with conditionals/interpolation only inside partials).

What you need to know: The merge mechanism is a fact, not a choice — a merged family needs one named variant per emitted skill, because each emitted SKILL.md needs its own name + description (the routing trigger) and per-output frontmatter can only come from variants.<ref>.<variant>.agent-skill.frontmatter (CLI flags cannot set frontmatter). The merge trade-off: merging maximizes file consolidation but the source template carries every branch as an if/else superset; it backfires when members diverge structurally (e.g. review-lossless-mapping and adjust-plan-granularity share almost nothing with their nominal families). Verified load-bearing jastr constraint (includes.test.ts:107, "rejects unknown include roots and group root on standalone templates"; include_outside_root on boundary escape): a standalone template can include ONLY from its own directory — cross-skill partials are impossible without a grouped catalog. Since Commit Policy is shared by ~20 of 29 skills and Immutability by 15, this forces a single group (e.g. workflow) holding every skill template with shared partials at the group root, included via root="group". That same one-group layout makes a later C→B upgrade trivial: collapsing templates/review-* into templates/review + variants leaves the partials in place. Two further facts established: (1) the merge choice is consumer-invisible — whether review-spec is a standalone template or a review#spec variant, the emitted SKILL.md is byte-identical at the same path with the same name/description, so B-vs-C is a pure authoring-side detail with zero downstream effect; (2) the drift the analysis tool measures is section-body duplication, which partials eliminate completely, so B's marginal advantage over C is file consolidation, not additional deduplication (for review, the shared parts are section-bounded → partials, and the differing parts — the axes — would be per-variant conditionals under B anyway).

Decision: Start with C — every skill is its own template under one group, DRY achieved entirely through shared include partials (conditionals and interpolation used only inside partials to absorb small per-skill variation, never as a template-level routing layer). Adopt the single-group catalog layout from day one so a later, evidence-driven upgrade of a specific family to B (template + variants) is a cheap in-place consolidation that does not move partials and does not change any emitted skill. The drift candidates at 70-90% similarity (Disposition Frontmatter, Tier Awareness, Lineage Folder and Filename, the implement-trio sections) must be reconciled into a single canonical partial each — parametrized by interpolation or an internal conditional — rather than relocated verbatim, or the duplication is merely moved into the catalog instead of removed.

Rationale: C respects DRY for the duplication that actually exists (section-level), and B's extra benefit is file-count consolidation rather than further dedup — a benefit that is consumer-invisible and therefore safe to defer. Starting at C is YAGNI: it defers the routing-design commitment until real authoring experience shows a family is worth merging, while the one-group layout keeps that upgrade cheap and the byte-identical emitted output keeps it invisible to consumers. The recommendation on the table was B; the user chose C as the careful first step with a deliberate upgrade path, and the consumer-invisibility plus section-level nature of the measured drift make C a fully DRY-respecting starting point rather than a compromise. Note: the spec should record a concrete C→B upgrade trigger (e.g. a family whose sibling templates are mostly a parametrized-identical body, or where the same logical section has been edited across 3+ siblings in one change) so the boundary is explicit for whoever revisits it.

## P3: Catalog layout and self-containment rule reconciliation

Point: The concrete shape of the jastr catalog (where it lives, how the single group is structured, where partials live, how outputs map to the existing distribution buckets) and how the repo's existing "every SKILL.md is self-contained, never links outside its folder" rule is reconciled with templates that compose via includes.

What you need to know: jastr groups are flat (one level) and includes cannot cross a group root, so all 29 skills must live in ONE group to share partials (established in P2). The distribution layout (skills/workflow/<phase>/<skill-name>/SKILL.md) is bucketed by phase; these decouple cleanly because skill-names are already unique single-segment kebab (all valid jastr template-ids), so the jastr group is a flat namespace of 29 template-ids and each generation writes its output into the correct phase bucket — the phase organization survives in the output path, not the catalog. Proposed catalog, committed at repo root as the new source of truth (NOT gitignored): .jastr/config.yml (variants registry, minimal/empty under C); .jastr/workflow/.jastrgroup (group marker); .jastr/workflow/partials/ (shared fragments at the group root, reached via root="group" — commit-policy.md ×20, immutability.md ×15, disposition-frontmatter.md ×5, tier-awareness.md ×4, lineage-and-filename.md ×4, one per reconciled shared section); .jastr/workflow/templates/<skill-name>/TEMPLATE.md (one per skill, 29). Each TEMPLATE.md carries its own targets.agent-skill.frontmatter (name + description) and a body that is mostly ::include{root="group", path="partials/…"} lines plus its unique content. Generation example: jastr generate agent-skill workflow/review-spec --out skills/workflow/review/review-spec/SKILL.md --mode=inline. Self-containment reconciliation: the current rule was written for hand-authored files; under this model it splits into (1) the EMITTED SKILL.md (shipped via npx skills add, read by Raycast) must remain fully self-contained — and it is, because inline render resolves every include/conditional/interpolation, so the invariant is preserved; (2) the AUTHORING templates and partials compose via includes and are deliberately not self-contained. What does NOT change under C: the set of skills, their paths, names, and descriptions, and therefore README.md, .claude-plugin/marketplace.json, .vscode/settings.json, and the Raycast sync are all untouched.

Decision: Adopt the proposed layout exactly — a single committed .jastr/workflow/ group at the repo root, partials at the group root under partials/, one template per skill under templates/<skill-name>/, outputs generated into their current phase buckets via --mode=inline. Update AGENTS.md to: (a) state that SKILL.md files under skills/ are generated and must never be hand-edited (edit the template + regenerate); (b) restate the self-containment guarantee as applying to rendered output, enforced by the renderer rather than author discipline, while authoring templates/partials may compose via includes; (c) rewrite the "When adding a new skill" and "Layout" sections around the template-authoring flow.

Rationale: The layout is largely forced by jastr's flat-group / no-cross-group-include constraints, and the single-group-at-root choice is what enables cross-phase partial sharing (Commit Policy spans ~20 of 29 skills across phases) while keeping the phase-bucketed distribution surface unchanged. Splitting the self-containment rule keeps the consumer-facing guarantee intact (the shipped file is still standalone) while unlocking authoring-side composition — the renderer now enforces what author discipline used to. The user fully agreed with no changes.

## P4: Generation and freshness mechanism (and the author-side jastr binary)

Point: How the 29 skills are regenerated from templates and how their committed SKILL.md outputs are kept in sync with the catalog, given the repo has no CI and no git hooks today.

What you need to know: 29 skills = 29 generate … --mode=inline invocations; generated SKILL.md files stay committed (they are what ships) alongside the committed catalog, so this is the standard "committed generated artifact + freshness check" pattern (the same one jastr uses for its own BEHAVIOR.md). The author currently has jastr-dev on PATH but it is stale (0.1.0 (9e564d4), no --mode) and its symlink points at a separate checkout (tools/jastr), NOT at .library. Building the vendored repo (.library/sources/Jei-sKappa_jastr, bun install && bun run build) produces a fresh bundle at .library/sources/Jei-sKappa_jastr/packages/cli/dist/index.js at 0.1.0 (6674fd3) which DOES support --mode <router|inline> and inline input flags. An end-to-end smoke test with that binary — a single workflow group, a partial at the group root included via root="group", a flag-free (no-input) template — confirmed inline generation produces a self-contained SKILL.md (frontmatter + rendered body, partial inlined, no jastr reference) and that --check reports up to date. Observed cosmetic detail: the rendered body is emitted verbatim, so a template body that begins with a blank line after its frontmatter yields a double blank line beneath the closing ---; authors should start template bodies cleanly. This is benign and deterministic, not a blocker.

Decision: Add scripts/generate-skills.sh (alongside the repo's existing scripts/) with two modes: default regenerates every skill (per template: jastr generate agent-skill workflow/<id> --out skills/workflow/<phase>/<id>/SKILL.md --mode=inline), and --check passes --check through to every call and exits nonzero if any output is stale or missing (the freshness gate, mirroring jastr's living-docs --check). The jastr binary is indirected through a JASTR_BIN env var; because jastr-dev is stale and points elsewhere, the binary to use is the freshly built .library bundle (or the user re-points jastr-dev / rebuilds their tools/jastr checkout to an inline-capable version). Output paths are sourced from the existing skills/workflow/** tree (leaf dir = template-id); the exact manifest mechanism (walk the tree vs. read marketplace.json vs. an explicit map) is an implementer's degree of freedom. Authoring rule for C: prefer template-author default: over required: inputs (most C templates declare no inputs) so every regen/--check command stays flag-free and self-contained, since inline's suggested-fix messages deliberately do not reconstruct input flags. Freshness enforcement is documented discipline (run scripts/generate-skills.sh --check before committing) with an optional git pre-commit hook wrapping the check; given single-owner, the hook is optional rather than mandatory.

Rationale: The committed-artifact-plus-check pattern is already proven in jastr itself and adds no CI dependency, fitting this content repo. Indirecting the binary via JASTR_BIN avoids hard-coding a path and accommodates the stale jastr-dev situation surfaced during the discussion. Preferring defaults over required inputs keeps the regen command reproducible without a flag registry, aligning with the inline spec's deliberate non-reconstruction of flags. The pre-commit hook is left optional because a single owner running the documented check is sufficient and a mandatory hook adds friction without a second contributor to protect against. The user agreed and asked that the binary be built from .library and verified, which was done (6674fd3, inline-capable, smoke-tested).

## P5: Migration sequencing

Point: Whether to convert all 29 skills in one pass or phase the work (pilot one family first, then fan out) to keep the migration careful.

What you need to know: Options presented were A — pilot-first: fully convert one small cohesive family end-to-end (recommended pilot plan-loose + plan-strict, the most-similar pair, exercising the four highest-value partials and forcing a real drift reconciliation), ratify the pattern, then fan out family-by-family; B — boilerplate-first: extract the universal partials (Commit Policy ×20, Immutability ×15) across all skills first, then handle families; C — all-at-once: convert everything in one pass. The recommendation was A (pilot-first). The per-skill correctness gate decided in P6 (git-diff rebaseline) applies regardless of sequencing and provides per-skill safety independent of the order chosen.

Decision: C — convert all skills, but divide the work into task batches during plan emission. The batching scheme itself is explicitly deferred to the plan stage and is not decided in this discussion.

Rationale: The recommendation on the table was A (pilot-first) for risk control, but the user chose all-at-once with batching deferred to plan emission. This is acceptable rather than reckless because P6's per-skill git-diff rebaseline gate gives each conversion an independent, auditable correctness check, so the big-bang risk that motivated the pilot recommendation is mitigated at the per-skill level, and batching at plan time preserves a careful execution order without carving the scope into separate pilot/rest efforts. The recommended pilot family (plan pair) can still serve as the first batch and reference example if the plan chooses to order it first.

## P6: Equivalence / correctness rule for each conversion

Point: How the conversion of each skill from a hand-authored SKILL.md to a generated one is proven correct.

What you need to know: The migration of each skill is a one-time rebaseline — the generated SKILL.md replaces the hand-authored one, and correctness is judged by git diff of the regenerated file against the committed original. The bar: the diff must contain no meaningful content change (nothing added, dropped, or reworded) except (a) deliberate drift-reconciliations where a near-identical section was normalized to its canonical partial, and (b) benign formatting normalization the renderer introduces (frontmatter key order via YAML.stringify, blank-line/trailing-newline handling such as the double-blank-line observed in the smoke test). Every non-empty hunk must be explicitly reviewed and approved as one of those two categories; an accidental change is a migration defect. After the rebaseline is approved and committed, --check enforces byte-stability going forward. Deliberate consequence: strict byte-identity to the old files is NOT the bar (it would flag benign renderer formatting as failures); the bar is content-preservation-modulo-formatting, judged by a human reading the diff.

Decision: Adopt the git-diff rebaseline rule exactly as above as the correctness strategy for every skill conversion.

Rationale: Git diff against the committed original gives an exact, auditable, per-skill signal that cleanly separates intended drift-reconciliation from accidental content change, while accommodating the formatting normalization the inline renderer applies (so the bar is content-preservation, not byte-identity). After the one-time reviewed rebaseline, --check guarantees ongoing byte-stability. The user fully agreed, calling git diff THE review strategy.

## P7: Versioning policy under generated skills

Point: When a template or a shared partial changes, which skills' frontmatter version should bump?

What you need to know: Each skill carries a semver version in frontmatter. A shared-partial edit conceptually changes every skill that includes it, but not all of them necessarily change in rendered output. After regeneration, the git diff already identifies exactly which emitted SKILL.md files changed.

Decision: After regen, bump the version of any skill whose rendered output changed (i.e. whose git diff is non-empty). A shared-partial edit thus bumps exactly the skills it actually affects, no more.

Rationale: Ties the version bump to observed output change rather than to authoring intent, so it is precise and mechanically derivable from the same git-diff signal P6 already relies on; avoids both over-bumping (every includer) and under-bumping (missed transitive effects).

## P8: metadata frontmatter passthrough

Point: Where the existing metadata block (author, version) lives once skills are generated from templates.

What you need to know: Skills currently carry metadata.author and metadata.version in frontmatter. jastr passes author-supplied frontmatter fields through unchanged from targets.agent-skill.frontmatter.

Decision: Each template carries metadata: {author, version} inside its targets.agent-skill.frontmatter; jastr emits it unchanged into the generated SKILL.md.

Rationale: Uses jastr's existing frontmatter passthrough, preserving the current metadata contract with no special handling.

## P9: Deprecated skills are out of scope

Point: Whether skills under skills/deprecated/ are migrated to templates.

What you need to know: The migration targets the active workflow skill suite. Deprecated skills are retired and kept on disk but not part of the active authoring surface.

Decision: skills/deprecated/** stay hand-authored and are NOT migrated to templates. Only skills/workflow/** is converted.

Rationale: Deprecated skills are frozen artifacts with no ongoing maintenance need; migrating them would add work and risk for no DRY benefit. Confirmed by the user.

## P10: Reconcile-vs-keep-separate judgment for drift candidates

Point: The criterion for deciding, per drift-candidate section (the 70-90%-similar sections the analysis tool surfaces), whether to reconcile it into one shared partial or keep it as skill-specific content.

What you need to know: The drift tool flags sections by body similarity, but similarity alone does not say whether divergence is accidental (should be unified) or intentional (genuinely different per skill). Forcing semantically-different sections into one partial corrupts skills; missing a reconciliation merely leaves some duplication.

Decision: Reconcile a section into a shared partial only when its differences are incidental — wording drift, or an artifact noun absorbable by interpolation. Keep it skill-specific when the differences are semantic. When unsure, keep separate: false-merging is worse than missed DRY.

Rationale: Biases toward correctness over maximal DRY, since a wrongly-merged section silently changes a skill's instructions while a missed merge only leaves harmless duplication that a later pass can still catch. Pairs with P6's git-diff gate, which makes any reconciliation's content effect visible for review.
