# Task 13: Signal handling

**Objective:** Handle `SIGINT`, `SIGTERM`, and `SIGHUP` gracefully: stop scheduling, abort the active attempt, persist interruption durably, release the lock, and exit with the conventional codes — with an immediate hard path on a second signal.

**Input / context:** `spec.md` §"Signals and interruption"; `decisions.md DR40` (first/second-signal behavior, conventional codes, pre-checkpoint behavior, abort classified as interruption), `DR26` (attempt finished as `interrupted`), `DR16` (hard kills stay manual recovery). Builds on the runner's `AbortSignal` plumbing (Task 10 — the aborted category maps to waiting kind `interrupted`) and the `run` command wiring (Task 12).

**Steps:**

1. Create `cli/src/runner/signals.ts` exporting `installSignalHandlers(deps: { abort: AbortController; stderr: NodeJS.WritableStream }): SignalState` where `SignalState = { signaled: () => NodeJS.Signals | null; exitCodeFor: (sig: NodeJS.Signals) => number; uninstall: () => void }`: register `process.on` handlers for the three signals; the first signal writes a brief notice to stderr, records the signal, and calls `abort.abort(new SignalInterruption(sig))` (a typed reason class); a second signal during cleanup calls `process.exit` immediately with the conventional code (`130`/`143`/`129`). Export the `SignalInterruption` reason type.
2. In `cli/src/runner/runner.ts`, complete the interruption classification and extend `RunnerResult` with `{ status: "interrupted"; signal: NodeJS.Signals }`. When an attempt fails with category `aborted` and the abort reason is a `SignalInterruption`, finish the attempt as `interrupted`, persist `waiting-for-user` with kind `interrupted`, record the signal name in `diagnostics.origin`, retain any pending paths found by the post-attempt scan as evidence on that interruption object, and include the DR54 warning that the attempt's file changes are unvalidated and must be reverted or deliberately committed before resume. This signal-specific branch precedes ordinary non-DONE queue/error classification: a first-signal rejection is interruption, never `harness-error` or a relabeled `pending-queues` result. Emit the durable pause and return `interrupted`. Recheck the signal after persisting an `executing` attempt and after creating its log but before invoking the harness; a signal in that window finishes the reserved attempt as interrupted without launching. When the first signal arrives while the checkpoint is durably `ready` between stages, stop before attempt allocation, leave that cursor byte-for-byte unchanged, report the interruption without rendering a fictional `Waiting for user` condition, and return `interrupted`. Check the signal at the loop top and between attempt settlement and scheduling so no new attempt or stage starts after the first signal.
3. In `cli/src/commands/run.ts`, make signal ownership explicit in `RunDeps`: replace Task 12's injected bare `signal` with optional `createAbortController?: () => AbortController` and `installSignals?: typeof installSignalHandlers` seams, defaulting to a fresh controller and the real installer. Production installs the handlers before preflight; tests inject controlled implementations without emitting real process signals. A signal arriving before the initial checkpoint exists makes the command return the signal's exit code without creating a run (check `signaled()` before allocation and before launching the runner). Map the runner's `interrupted` result to `exitCodeFor(sig)`, release the lock first when one is owned, and never map it to ordinary durable-pause exit `2`. Uninstall handlers on every ordinary return path.
4. Add `cli/src/runner/signals.test.ts` (in-process: fabricate the controller and assert first-signal abort + recorded signal, exit-code mapping, reason typing) and extend `cli/src/runner/runner.test.ts` with: a signal after the executing checkpoint but before harness launch → reserved attempt becomes `interrupted` with zero invocations; a hang-until-abort fake-harness attempt aborted mid-flight → attempt `interrupted`, run `waiting-for-user` kind `interrupted` with signal origin, files/logs preserved, no further attempt started, and `RunnerResult.status === "interrupted"` (AC-17.1, AC-17.3); the same abort with a simultaneously created pending file remains kind `interrupted` while retaining that path; abort between stages in `ready` → checkpoint byte-for-byte untouched and no durable-pause rendering. Extend `cli/src/commands/run.test.ts` with a pre-checkpoint abort → no run directory created and the signal exit code returned, and an active interruption → persisted waiting checkpoint but conventional signal exit code rather than `2` (AC-17.1, AC-17.2). Cover the second-signal immediate-exit path in `signals.test.ts` by stubbing `process.exit`.

**Acceptance criteria:**

- First signal during an executing attempt: abort, attempt `interrupted`, run `waiting-for-user`, lock released, conventional exit code (AC-17.1).
- Signal before the initial checkpoint: exit without a run; second signal: immediate exit with the conventional code and no further cleanup guarantees (AC-17.2).
- Abort rejections are classified as interruption, never `harness-error`; nothing new starts after the first signal (AC-17.3).
