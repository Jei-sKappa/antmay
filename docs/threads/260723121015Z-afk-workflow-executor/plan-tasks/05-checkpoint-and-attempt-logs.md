# Task 5: Checkpoint schema, atomic persistence, run IDs, attempt logs

**Objective:** Implement the durable-state primitives: the `schemaVersion: 1` checkpoint model with manual validators, atomic checkpoint writes, run-ID generation with exclusive run-directory allocation, and exclusively created attempt logs with the required header.

**Input / context:** `spec.md` §"Durable state" (checkpoint schema, transitions vocabulary, atomic persistence, waiting taxonomy, attempt logs) and §"Creating a run" (run identity); `decisions.md DR26` (schema and conditions), `DR27` (atomic write, modes, fatal-checkpoint posture), `DR35` (run IDs, log naming, exclusive creation), `DR41` (waiting object, `waiting: null`), `DR50` (Git baseline fields), `DR48` (harness version line in the log header). Consumes `StageDescriptor`, `StageProfile`, `Recipe` types from Task 3. Field names and JSON nesting beyond the required content are this task's choice (spec degree of freedom 3) — pick them here and keep them stable.

**Steps:**

1. Create `cli/src/state/checkpoint.ts` exporting the checkpoint types and the manual validator:
   - `type RunCondition = "ready" | "executing" | "waiting-for-user" | "completed"`; `type AttemptResult = "executing" | "done" | "waiting" | "interrupted"`;
   - `type WaitingKind = "outcome-blocked" | "outcome-refused" | "pending-queues" | "malformed-outcome" | "harness-error" | "idle-timeout" | "interrupted" | "gate-error" | "git-policy-violation" | "commit-error"`;
   - `type WaitingInfo = { kind: WaitingKind; message: string; pendingFiles?: string[]; candidateLine?: string; diagnostics?: { category: string; errorClass?: string; errorMessage?: string; origin?: string } }`;
   - `type AttemptRecord = { attempt: number; stageIndex: number; stageId: string; startedAt: string; endedAt?: string; result: AttemptResult; outcome?: { token: "DONE" | "BLOCKED" | "REFUSED"; candidateLine: string; detail: string } | null; pendingFiles?: string[]; failure?: { kind: string; message: string }; logPath: string }` (`logPath` run-relative POSIX);
   - `type SnapshottedStage = StageDescriptor & { profile: StageProfile; resolvedTarget: string }`;
   - `type RunCheckpoint = { schemaVersion: 1; runId: string; executor: { pid: number; version: string }; createdAt: string; updatedAt: string; repoRoot: string; threadRelPath: string; workspace: { strategy: "current-checkout"; path: string }; dangerouslySkipPermissions: boolean; recipeName: string; stages: SnapshottedStage[]; stageIndex: number; condition: RunCondition; attempts: AttemptRecord[]; waiting: WaitingInfo | null; gitBaseline: { headAtStageEntry: string | null } }`;
   - `validateCheckpoint(doc: unknown): CheckpointResult` (`{ ok: true; checkpoint: RunCheckpoint } | { ok: false; errors: string[] }`) — a manual typed validator; an unknown `schemaVersion` is a distinct clear error; no migration.
2. Create `cli/src/state/persist.ts` exporting `writeCheckpoint(runDir: string, checkpoint: RunCheckpoint, fsOps?: FsOps): Promise<void>` and `readCheckpoint(runDir: string): Promise<CheckpointResult>`. Write path: serialize deterministic two-space JSON plus trailing newline; exclusively create a uniquely named temp file beside `state.json` (e.g. `.state.json.<random>.tmp`, flag `wx`, mode `0600`); write, `fsync`, close; `rename` over `state.json`; best-effort `fsync` the directory. Never truncate in place; no backup files. Readers treat only `state.json` as authoritative and ignore leftover temp files. `FsOps` is a minimal injectable seam (open/write/fsync/rename) defaulting to `node:fs/promises`, existing solely so tests can inject write/rename failures.
3. Create `cli/src/state/runs.ts` exporting `generateRunId(now: Date, random: (n: number) => Buffer): string` producing `<YYYYMMDDTHHmmssSSSZ>-<8-lowercase-hex>` (four crypto-random bytes), and `allocateRunDirectory(stateRoot: string, deps?): Promise<{ runId: string; runDir: string }>` — lazily create `<state-root>/afk-runs/` with mode `0700` (and the state root itself, `0700`), then exclusively create `<state-root>/afk-runs/<run-id>/` (mode `0700`), regenerating the ID on `EEXIST`. Also export `runsDirectory(stateRoot)` and `runDirectoryFor(stateRoot, runId)` path helpers for `list`/`resume` (which must *not* create directories — creation lives only in `allocateRunDirectory`).
4. Create `cli/src/state/logs.ts` exporting `type AttemptLogHeader = { runId: string; stageId: string; stageOrdinal: number; attempt: number; harness: string; model: string; harnessVersion: string; repoRoot: string; threadRelPath: string; startedAt: string }` and `createAttemptLog(runDir: string, header: AttemptLogHeader): Promise<{ absPath: string; runRelPath: string; append: (line: string) => void; close: () => Promise<void> }>` — create `logs/` under the run dir (`0700`), exclusively create `logs/<NN>-<stage-id>-attempt-<NN>.log` (two-digit zero-padded ordinal and attempt, flag `wx`, mode `0600`), and write the Antmay header (every header field, one per line, including the observed harness version) before returning. A second creation of the same path must fail (`EEXIST`), never append or overwrite.
5. Add tests: `cli/src/state/checkpoint.test.ts` (validator accepts a full round-tripped checkpoint, rejects unknown schema version distinctly, reports field errors); `cli/src/state/persist.test.ts` (two-space + trailing-newline serialization; temp-file mode `0600`; injected write failure and injected rename failure both leave the previous `state.json` intact and readable; leftover temp files are ignored by `readCheckpoint`); `cli/src/state/runs.test.ts` (ID format regex `^\d{8}T\d{9}Z-[0-9a-f]{8}$`, collision regeneration via injected random, directory modes `0700`, lazy creation only via `allocateRunDirectory`); `cli/src/state/logs.test.ts` (name pattern, exclusive creation, header content and order, mode `0600`).

**Acceptance criteria:**

- `RunCheckpoint` carries every field listed in the spec's checkpoint schema; unknown schema versions fail clearly (AC-13.1).
- Atomic-write behavior proven under injected failures: reader sees previous or complete new document, never a partial one (AC-13.2).
- Run IDs match the required pattern with exclusive-create collision regeneration (AC-7.5 identity portion).
- Attempt logs are exclusively created at `logs/<NN>-<stage-id>-attempt-<NN>.log` with the full header including harness version (AC-13.4).
- State directories `0700`; checkpoints and logs `0600` (AC-13.5).

**Consumes:** `StageDescriptor`, `StageProfile`, `Recipe` from `cli/src/recipe/types.ts` (Task 3).

**Produces:** the checkpoint type family and `validateCheckpoint(doc)` from `cli/src/state/checkpoint.ts`; `writeCheckpoint(runDir, checkpoint)` / `readCheckpoint(runDir)` from `cli/src/state/persist.ts`; `generateRunId(now, random)`, `allocateRunDirectory(stateRoot)`, `runsDirectory(stateRoot)`, `runDirectoryFor(stateRoot, runId)` from `cli/src/state/runs.ts`; `createAttemptLog(runDir, header)` and `AttemptLogHeader` from `cli/src/state/logs.ts`.
