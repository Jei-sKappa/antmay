# Task 5: Checkpoint schema, atomic persistence, run IDs, attempt logs

**Objective:** Implement the durable-state primitives: the `schemaVersion: 1` checkpoint model with manual validators, atomic checkpoint writes, run-ID generation with exclusive run-directory allocation, and exclusively created attempt logs with the required header.

**Input / context:** `spec.md` §"Durable state" (checkpoint schema, transitions vocabulary, atomic persistence, waiting taxonomy, attempt logs) and §"Creating a run" (run identity); `decisions.md DR26` (schema and conditions), `DR27` (atomic write, modes, fatal-checkpoint posture), `DR35` (run IDs, log naming, exclusive creation), `DR41` (waiting object, `waiting: null`), `DR50` (Git baseline fields), `DR48` (harness version line in the log header). Consumes `HarnessId` from Task 2 and `StageDescriptor`, `StageProfile`, `Recipe` types from Task 3. Field names and JSON nesting beyond the required content are this task's choice (spec degree of freedom 3) — pick them here and keep them stable.

**Steps:**

1. Create `cli/src/workspace/types.ts` with the Antmay-owned, JSON-serializable `type WorkspaceExecution = { cwd: string; sandbox: "none"; branchStrategy: "head" }` and `type WorkspaceConfig = { strategy: "current-checkout"; path: string; execution: WorkspaceExecution }`; these values let the strategy supply cwd/sandbox/branch behavior without leaking Sandcastle types. Create `cli/src/state/checkpoint.ts` exporting the checkpoint types and the manual validator:
   - `type RunCondition = "ready" | "executing" | "waiting-for-user" | "completed"`; `type AttemptResult = "executing" | "done" | "waiting" | "interrupted"`;
   - `type WaitingKind = "outcome-blocked" | "outcome-refused" | "pending-queues" | "malformed-outcome" | "harness-error" | "idle-timeout" | "interrupted" | "gate-error" | "git-policy-violation" | "commit-error"`;
   - `type WaitingInfo = { kind: WaitingKind; message: string; pendingFiles?: string[]; candidateLine?: string; diagnostics?: { category: string; errorClass?: string; errorMessage?: string; origin?: string } }`;
   - `type AttemptRecord = { attempt: number; stageIndex: number; stageId: string; startedAt: string; endedAt?: string; result: AttemptResult; terminalResult: { token: "DONE" | "BLOCKED" | "REFUSED" | null; candidateLine: string | null; detail: string } | null; pendingFiles?: string[]; failure?: { kind: string; message: string }; logPath: string }` (`logPath` run-relative POSIX). `terminalResult` stores the candidate line even when no token parses; it is `null` only before the attempt has returned a terminal text result.
   - `type SnapshottedStage = StageDescriptor & { profile: StageProfile; resolvedTarget: string }`;
   - `type RunCheckpoint = { schemaVersion: 1; runId: string; executor: { pid: number; version: string }; createdAt: string; updatedAt: string; repoRoot: string; threadRelPath: string; workspace: WorkspaceConfig; dangerouslySkipPermissions: boolean; recipeName: string; stages: SnapshottedStage[]; observedHarnessVersions: Partial<Record<HarnessId, string>>; stageIndex: number; condition: RunCondition; attempts: AttemptRecord[]; waiting: WaitingInfo | null; gitCursor: { stageIndex: number; headAtStageEntry: string | null; observedHead: string | null } }`. `observedHarnessVersions` is operational probe evidence kept outside the immutable stage/profile snapshot: new runs retain every version line collected by their all-selected-harness preflight, while resume may temporarily replace only the current harness's entry with its fresh probe result for the attempts in that process.
   - `validateCheckpoint(doc: unknown): CheckpointResult` (`{ ok: true; checkpoint: RunCheckpoint } | { ok: false; errors: string[] }`) — a manual typed validator; an unknown `schemaVersion` is a distinct clear error; no migration. Validate every required scalar, enum, nested descriptor/profile, ISO-8601 UTC timestamp, absolute host path, and normalized repository-relative/log/pending path. Validate cross-field invariants as well as field types: `waiting-for-user` requires exactly one non-null `waiting` object, every other condition requires `waiting: null`, pending paths are lexically sorted and unique, `stageIndex` must be within the condition-appropriate bounds, attempt stage references must resolve inside the snapshot, attempt numbers are positive and collision-free for their stage, a `done` attempt carries parsed DONE, and exactly the final attempt is `executing` iff the run condition is `executing`. Every harness selected by a snapshotted stage must have a non-empty `observedHarnessVersions` entry, current-checkout `workspace.path` equals `workspace.execution.cwd`, and `gitCursor.stageIndex` must name the current stage when its HEAD fields are non-null.
2. Create `cli/src/state/persist.ts` exporting `writeCheckpoint(runDir: string, checkpoint: RunCheckpoint, fsOps?: FsOps): Promise<void>` and `readCheckpoint(runDir: string): Promise<CheckpointResult>`. Write path: serialize deterministic two-space JSON plus trailing newline; exclusively create a uniquely named temp file beside `state.json` (e.g. `.state.json.<random>.tmp`, flag `wx`, mode `0600`); write, `fsync`, close; `rename` over `state.json`; best-effort `fsync` the directory. Never truncate in place; no backup files. Readers treat only `state.json` as authoritative and ignore leftover temp files. `FsOps` is a minimal injectable seam (open/write/fsync/rename) defaulting to `node:fs/promises`, existing solely so tests can inject write/rename failures.
3. Create `cli/src/state/runs.ts` exporting `generateRunId(now: Date, random: (n: number) => Buffer): string` producing `<YYYYMMDDTHHmmssSSSZ>-<8-lowercase-hex>` (four crypto-random bytes), and `createRunDirectory(stateRoot: string, runId: string): Promise<{ kind: "created"; runDir: string } | { kind: "collision" }>` — lazily create `<state-root>/afk-runs/` with mode `0700` (and the state root itself, `0700`), then exclusively create `<state-root>/afk-runs/<run-id>/` (mode `0700`), returning `collision` on `EEXIST`. Generation/retry stays in the run command because the same candidate ID must be written into the workspace lock before the directory is created; a collision requires releasing that candidate's lock and retrying the locked allocation with a new ID. Also export `runsDirectory(stateRoot)` and `runDirectoryFor(stateRoot, runId)` path helpers for `list`/`resume` (which must *not* create directories — creation lives only in `createRunDirectory`).
4. Create `cli/src/state/logs.ts` exporting `type AttemptLogHeader = { runId: string; stageId: string; stageOrdinal: number; attempt: number; harness: string; model: string; harnessVersion: string; repoRoot: string; threadRelPath: string; startedAt: string }`, pure `attemptLogPaths(runDir, stageOrdinal, stageId, attempt): { absPath: string; runRelPath: string }`, and `createAttemptLog(paths, header): Promise<void>`. The pure helper produces `logs/<NN>-<stage-id>-attempt-<NN>.log` (two-digit zero-padded ordinal and attempt), allowing the runner to persist the attempt record with its final log path before touching the log file. Creation then makes `logs/` under the run dir (`0700`), exclusively creates the computed file (flag `wx`, mode `0600`), writes and flushes the Antmay header (every header field, one per line, including the observed harness version), and closes the file before returning. Sandcastle's file logger appends the verbose stream to this preinitialized file. A second creation of the same path must fail (`EEXIST`), never append or overwrite.
5. Add tests: `cli/src/state/checkpoint.test.ts` (validator accepts a full round-tripped checkpoint, rejects unknown schema version distinctly, reports field and cross-field invariant errors, retains tokenless terminal candidates, and round-trips the HEAD cursor and observed harness-version map); `cli/src/state/persist.test.ts` (two-space + trailing-newline serialization; temp-file mode `0600`; injected write failure and injected rename failure both leave the previous `state.json` intact and readable; leftover temp files are ignored by `readCheckpoint`); `cli/src/state/runs.test.ts` (ID format regex `^\d{8}T\d{9}Z-[0-9a-f]{8}$`, explicit-candidate collision reporting, directory modes `0700`, lazy creation only via `createRunDirectory`); `cli/src/state/logs.test.ts` (pure path/name generation, exclusive creation, header content and order, header flushed and file closed before return, mode `0600`).

**Acceptance criteria:**

- `RunCheckpoint` carries every field listed in the spec's checkpoint schema, including both durable HEAD observations and tokenless terminal candidates; the separate observed-version map makes every resumed attempt header executable without broadening resume's one-harness probe. Unknown schema versions and inconsistent condition/waiting/cursor shapes fail clearly (AC-10.2, AC-12.7, AC-13.1, AC-14.1).
- Atomic-write behavior proven under injected failures: reader sees previous or complete new document, never a partial one (AC-13.2).
- Run IDs match the required pattern; exclusive directory creation reports collisions so Task 12 can regenerate the candidate and keep the lock record, run directory, and checkpoint on one ID (AC-7.5 identity portion).
- Attempt logs are exclusively created at `logs/<NN>-<stage-id>-attempt-<NN>.log` with the full header including harness version (AC-13.4).
- State directories `0700`; checkpoints and logs `0600` (AC-13.5).
