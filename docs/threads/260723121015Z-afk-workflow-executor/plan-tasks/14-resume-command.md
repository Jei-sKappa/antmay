# Task 14: `antmay afk resume`

**Objective:** Implement `resume <run-id>`: full checkpoint revalidation, per-waiting-kind resume behavior — including declared queue-resolution `advance`/`rerun`, harness-free Git-boundary finalization, and abandoned-`executing` recovery — then continuation through the snapshotted stages.

**Input / context:** `spec.md` §"Resuming a run" plus the resume-relevant parts of §"Git boundaries and executor commits" (finalization without another harness invocation) and the exit-code table; `decisions.md DR36` (revalidation list, snapshot-only, no overrides, preflight leaves checkpoint unchanged), `DR53` (`advance`/`rerun` after DONE-finalized queue pauses; non-DONE queue pauses retry same stage), `DR50` (Git-kind finalization; cross-pause HEAD warning), `DR54` (clean-worktree rule and its two exempt kinds), `DR16`/`DR26` (executing → manual lock removal → `interrupted` + new attempt), `DR9` (recorded repo path must resolve; never search), `DR48` (probe only the current stage's harness), `DR56` (re-print unrestricted warning). Reuses the runner: after the kind-specific entry action, continuation is just `executeRun` from the (possibly advanced) cursor.

**Steps:**

1. Create `cli/src/commands/resume.ts` exporting `resumeCommand(args: { runId: string }, deps: RunDeps): Promise<number>`. Install Task 13's signal handlers at command entry and uninstall them on every ordinary return. Resolve only the state root through `resolveStateRoot`; resume never resolves a config root, reads settings, or imports recipe definitions.
2. Implement preflight, in order, every failure exiting `1` with the checkpoint unchanged: locate `runDirectoryFor(stateRoot, runId)` (absent runs directory or run dir → unknown run); `readCheckpoint` + `validateCheckpoint` (malformed → clear error); a `completed` run reports completion and exits `1`; verify the recorded `repoRoot` still exists and canonicalizes to the Git worktree top-level containing the recorded active thread, with `threadRelPath` resolving to exactly one direct child of that root's active `docs/threads/` (a non-resolving or escaped recorded path fails clearly, never searched — AC-15.2); revalidate non-empty `seed.md`/`decisions.md`; probe **only** the current stage's snapshotted harness via `probeHarnessExecutables([currentProfile.harness], repoRoot)` and require its one non-empty version entry; resolve `resolveCurrentCheckoutWorkspace(repoRoot)` and require its canonical path to equal the checkpoint's `workspace.path`; enforce the clean-worktree rule via `isWorktreeClean` — required for every waiting kind except `git-policy-violation` and `commit-error`, and for `ready` and `executing` conditions; a dirty tree exits `1` with a commit-what-you-want-to-keep-or-revert message (AC-15.1). Check the installed signal state after each awaited phase and immediately before lock acquisition; a first signal exits with its conventional code and the checkpoint unchanged. Build the process-local harness-version map by copying `checkpoint.observedHarnessVersions` and overriding only the current harness with this resume probe; later stages in the same resumed process retain their run-creation observations without broadening the one-current-harness resume probe or mutating the immutable stage snapshot.
3. Acquire the recorded workspace lock before any checkpoint mutation. Existing-lock contention — including an abandoned `executing` checkpoint whose stale lock has not been manually removed — prints the lock metadata/path, points to the stale-lock documentation and its verify-the-process-first instruction, and exits `1`. Recheck the signal immediately after acquisition and, if set, release the lock and return its conventional code before mutating the checkpoint. Once acquisition succeeds, an `executing` checkpoint is atomically recovered under the lock: mark its abandoned attempt `interrupted` with `diagnostics.origin: "manual-recovery"`, persist a retry-at-same-stage cursor, and only then perform further resume transitions. A signal during a resumed attempt uses the same durable interruption path and conventional exit code as `run`; a signal during preflight or before a new attempt leaves the pre-existing or recovered durable cursor unchanged.
4. Re-scan both queues under the acquired lock, recheck the signal before any queue-driven checkpoint transition, and print the startup summary (re-print the prominent unrestricted warning when the persisted flag is set). A signal at this boundary releases the lock and returns its conventional code with the current durable cursor unchanged. Otherwise handle the scan result before any harness action:
   - a scan error on this existing run atomically persists `waiting-for-user` with kind `gate-error`, releases the lock, and exits `2`; if persistence fails, use the fatal-checkpoint path;
   - a waiting run with non-empty queues remains byte-for-byte unchanged, prints the remaining files, releases the lock, and exits `2`;
   - a ready or recovered-executing cursor with non-empty queues persists a tokenless pre-attempt `pending-queues` pause without allocating an attempt, releases the lock, and exits `2`.
5. With queues empty, branch on the stored state:
   - waiting `pending-queues` whose relevant finished attempt is `done` with parsed DONE (DONE-finalized): apply the stage's snapshotted `queueResolution` — `advance` performs the normal successful-stage cursor transition without rerunning; `rerun` clears waiting and starts a fresh attempt at the same stage;
   - waiting `pending-queues` whose relevant attempt is `waiting` (non-DONE), or which has no relevant attempt because the pause happened at the pre-attempt gate: clear waiting and start a new attempt at the same stage;
   - waiting `git-policy-violation`/`commit-error`: perform boundary finalization **without a harness invocation**. Compare the current HEAD with `gitCursor.observedHead` and warn on a difference, but call `evaluateBoundary` in resume-finalization mode (`enforceHead: false`, `allowRequiredChangeToBeAlreadyCommitted: true`) before `finalizeBoundary`; this accepts an intended required diff the user deliberately committed while paused and never treats cross-pause HEAD movement as a violation. Read HEAD after the finalization attempt and store it as the new observation. On continued violation or commit failure, refresh the corresponding waiting object, release, and exit `2`. On success, change the preserved DONE attempt from `waiting` to `done` and clear waiting. If the original Git-kind pause listed pending files, apply the stage's declared `queueResolution` (`advance` or same-stage `rerun`) now that both the boundary and queues are resolved; otherwise perform the normal successful-stage advance. An advance resets the cursor; a rerun preserves the updated observed HEAD until the new attempt starts.
   - every other waiting kind, and `ready`: clear waiting where applicable and start a new attempt at the stored stage.
   Every advance uses one shared transition helper: increment `stageIndex`, reset the HEAD cursor, and persist `ready`, or `completed` with `waiting: null` at the end of the snapshot. Then delegate any runnable cursor to `executeRun` for continuation through subsequent snapshotted stages until pause, interruption, or completion; map results to exit codes exactly as in `run`, including conventional signal codes, and release the lock in `finally`.
6. Wire the real handler in `cli/src/program.ts` with a selected-handler-only dynamic import (replace the `resume` placeholder). `resume` accepts no execution overrides and never rereads settings or recipe definitions — it must not import `loadSettings` or `builtInRecipes` for resolution (grammar already blocks the flags). Keep command/config/state/Git/harness modules out of the static help/version path and keep `cli/src/main.ts` dependency-free beyond its version guard and dynamic import.
7. Add `cli/src/commands/resume.test.ts` on the repo fixture with the fake harness, seeding state by running `runCommand` with scripted pauses: unknown/malformed/completed runs exit `1` (AC-15.2); a config-only environment error does not block state-only resume; dirty-worktree refusal for an `outcome-blocked` pause vs. acceptance for a `commit-error` pause (AC-15.1); waiting + still-pending files → unchanged checkpoint, files printed, exit `2`; locked queue-scan failure → durable `gate-error`; pre-attempt `pending-queues` with no attempt evidence re-attempts the same stage; DONE-finalized `pending-queues` with `advance` continues to the next stage without rerunning, with `rerun` re-attempts the same stage; non-DONE `pending-queues` re-attempts; `git-policy-violation` finalization commits the preserved diff without any invoker call before the next stage, and an intended diff manually committed to an empty worktree advances; a Git-kind pause that originally listed pending files applies `queueResolution` after finalization, including a `rerun` case; `executing` + present lock is refused, while after manual stale-lock removal the lock is acquired before the abandoned attempt is marked `interrupted` and a new attempt runs (AC-15.3); only the current harness is freshly probed, while a later different-harness stage gets its retained observed version in the log header; a settings edit between pause and resume changes nothing (AC-15.4, AC-20.2); stages already advanced past never rerun (AC-15.4); cross-pause HEAD movement warns by comparison with the pause-time observed HEAD and never pauses (AC-12.7); unrestricted-flag runs re-print the warning (AC-18.1); signals during resumed execution persist interruption, release the lock, and return the conventional code.

**Acceptance criteria:**

- The full AC-15.1 revalidation sequence runs in order with checkpoint-preserving failures; only the current stage's harness is probed.
- Every AC-15.3 branch behaves as specified, including attempt-less pre-gate retries, `advance`/`rerun`, queue-read failure, harness-free Git finalization after manual commits, Git-kind pauses that also carried queue files, and lock-owned `executing` recovery.
- AC-15.4 holds: no settings/recipe rereads, no overrides, no session resumption, no advancing past a stage lacking DONE finalization, continuation through the snapshot after success.
- Resume of a completed run and a vanished repository fail per AC-15.2.
