# Task 10: Stage runner

**Objective:** Implement the generic, recipe-agnostic stage loop that drives a run from its checkpoint cursor to completion or durable pause: attempt lifecycle, queue gates, boundary finalization, classification, checkpoint transitions, and display events.

**Input / context:** `spec.md` §"Executing a stage", §"Interpreting a stage result", §"Git boundaries and executor commits" (finalize-before-queue-pause), §"Durable state" (transitions), §"Terminal display" (event points); `decisions.md DR4`/`DR46` (no branching on recipe/stage/skill names), `DR26`/`DR27` (transitions, fatal-checkpoint posture), `DR52` (DONE finalization before queue pause; attempt recorded `done`), `DR44` (scan points), `DR54` (unvalidated-changes warning on every non-DONE pause), `DR10` (lock held across the uninterrupted stage sequence). Composes Tasks 3, 5, 6, 7, 8, 9. The harness and display are injected; everything else is direct function calls.

**Steps:**

1. Create `cli/src/display/types.ts` exporting the `Display` interface the runner emits to — methods (all synchronous, fire-and-forget): `attemptStarted(info: { stagePosition: string; stageId: string; harness: string; model: string; attempt: number; logAbsPath: string })`, `harnessEvent(event: HarnessEvent)`, `heartbeat(elapsedMs: number)`, `stageSucceeded(info: { stagePosition: string; durationMs: number })`, `runPaused(info: { waiting: WaitingInfo; runId: string; logAbsPath: string | null; resumeCommand: string; checkpointPath: string })`, `runCompleted(info: { runId: string; recipeName: string; totalElapsedMs: number; checkpointPath: string })`, `warn(message: string)`. Also export `nullDisplay: Display` for tests.
2. Create `cli/src/runner/runner.ts` exporting `executeRun(ctx: RunnerContext): Promise<RunnerResult>` with:
   - `type RunnerContext = { checkpoint: RunCheckpoint; runDir: string; stateRoot: string; lock: LockHandle; invoker: HarnessInvoker; display: Display; signal: AbortSignal; clock?: () => Date }`;
   - `type RunnerResult = { status: "completed" } | { status: "paused"; waiting: WaitingInfo } | { status: "fatal-checkpoint"; message: string }` — skill-local to the runner/commands seam; the commands map it to exit codes.
3. Implement the per-stage loop, starting from `checkpoint.stageIndex` while the lock is held:
   1. **Pre-attempt queue gate:** `scanPendingQueues`; a scan failure pauses as `gate-error`; non-empty pauses as `pending-queues` without allocating an attempt or launching (attempt history untouched).
   2. **Attempt setup:** record `headAtStageEntry` via `readHead`; create the attempt log via `createAttemptLog` (header from the snapshotted stage/profile plus the probed harness version carried in the context — add `harnessVersions: Record<string, string>` to `RunnerContext`); atomically persist the checkpoint with a new `executing` attempt and condition `executing` *before* spawning; a persistence failure here prevents the launch (fatal-checkpoint).
   3. **Invoke:** render the stage prompt here via `renderStagePrompt(profile.harness, stage.skill, stage.resolvedTarget, profile.prompt)` — pure and deterministic from the snapshot, so resume reproduces it without rereading settings; call `invoker.invoke` with the profile, prompt, log path, `dangerouslySkipPermissions` from the checkpoint, the abort signal, and `onEvent` forwarding to `display.harnessEvent`.
   4. **Post-attempt gates:** re-scan queues; on `kind: "completed"` parse the terminal outcome; on a parsed DONE run boundary evaluation (`collectBoundaryStatus`, `evaluateBoundary`, then `finalizeBoundary` when ok) producing the `BoundaryDisposition`; call `classifyAttempt`.
   5. **Transition:** `advance` → finish the attempt as `done` (store outcome triple), record `stageSucceeded`, increment `stageIndex`, persist `ready` (or `completed` with `waiting: null` when the index reaches the stage count) and continue the loop; `pause-done` → finish the attempt as `done`, persist `waiting-for-user` with the `pending-queues` waiting object (pending files listed) *without advancing*; `pause` → finish the attempt as `waiting` (or `interrupted` when the abort signal caused it — the aborted category maps to kind `interrupted` with its origin), persist `waiting-for-user` with the waiting object; every non-DONE pause message appends the DR54 warning that the attempt's file changes are unvalidated and must be reverted or deliberately committed before resuming.
   6. Every persisted checkpoint refreshes `updatedAt`; a persistence failure after a stage return reports fatal-checkpoint and never advances or launches another stage.
4. On pause/completion the runner emits `runPaused` (with the exact resume command `antmay afk resume <run-id>` and the stored waiting object) or `runCompleted`, and returns; the *caller* releases the lock (commands own acquire/release symmetry). Implement the five-minute display heartbeat as an interval timer started per attempt, cleared on settle, calling `display.heartbeat` — independent of harness output and of Sandcastle's idle timer.
5. Add `cli/src/runner/runner.test.ts` (plus `cli/src/test-helpers/fake-harness.ts` (NEW) exporting a scriptable `HarnessInvoker` fake: per-call scripted `AttemptOutcome`s, optional filesystem side effects before returning, hang-until-abort behavior). Drive full runs on the repo fixture with a **synthetic two-stage recipe** as well as `standardRecipe` (proving recipe-agnosticism, AC-6.3): all-stages-DONE completion with per-stage transitions and `waiting: null`; DONE with valid changes → boundary committed then advance; DONE + a file dropped into `.pending-decisions/` → boundary committed first, attempt `done`, pause `pending-queues` (AC-11.3, AC-12.1); BLOCKED/REFUSED/malformed/idle-timeout/provider-error pauses with attempt `waiting`, correct kinds, DR54 warning text; pre-attempt non-empty queue pauses with no new attempt (AC-11.2); boundary violation and commit-error pauses preserving the attempt; fatal-checkpoint on injected persistence failure before launch and after return (AC-13.3); missing `spec.md` for a later stage still launches (no artifact preconditions, AC-6.4).

**Acceptance criteria:**

- Checkpoint transitions match AC-13.3 for every path: `executing` persisted pre-spawn, `done`+advance, `pause-done` as `done` without advancing, other pauses as `waiting`, `ready`/`completed` with `waiting: null`, fatal-checkpoint halting.
- DONE boundaries are finalized (committed when validly changed) before any queue pause; queue scans happen at every DR44 point inside the loop (AC-11.1 loop portion, AC-12.1).
- Every non-DONE pause message carries the unvalidated-changes warning (AC-12.6).
- The runner consumes only snapshotted stage data and typed inputs; the synthetic-recipe suite passes through the identical code path (AC-6.3).

**Consumes:** `renderStagePrompt` (Task 3); checkpoint types, `writeCheckpoint`, `createAttemptLog` (Task 5); `scanPendingQueues`, `LockHandle` (Task 6); `HarnessInvoker`, `HarnessEvent` (Task 7); `parseTerminalOutcome`, `classifyAttempt` (Task 8); `collectBoundaryStatus`, `readHead`, `evaluateBoundary`, `finalizeBoundary` (Task 9); `createRepoFixture` (Task 4).

**Produces:** `Display` interface and `nullDisplay` from `cli/src/display/types.ts`; `executeRun(ctx: RunnerContext): Promise<RunnerResult>` with `RunnerContext` carrying `harnessVersions` from `cli/src/runner/runner.ts`; the scriptable fake harness from `cli/src/test-helpers/fake-harness.ts`.
