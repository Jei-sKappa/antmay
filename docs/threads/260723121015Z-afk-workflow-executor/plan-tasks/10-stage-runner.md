# Task 10: Stage runner

**Objective:** Implement the generic, recipe-agnostic stage loop that drives a run from its checkpoint cursor to completion or durable pause: attempt lifecycle, queue gates, boundary finalization, classification, checkpoint transitions, and display events.

**Input / context:** `spec.md` §"Executing a stage", §"Interpreting a stage result", §"Git boundaries and executor commits" (finalize-before-queue-pause), §"Durable state" (transitions), §"Terminal display" (event points); `decisions.md DR4`/`DR46` (no branching on recipe/stage/skill names), `DR26`/`DR27` (transitions, fatal-checkpoint posture), `DR52` (DONE finalization before queue pause; attempt recorded `done`), `DR44` (scan points), `DR57` (a queue-scan failure during boundary finalization keeps the Git-boundary kind and folds the scan diagnostic in), `DR54` (unvalidated-changes warning on every non-DONE pause), `DR10` (lock held across the uninterrupted stage sequence). Composes Tasks 3, 5, 6, 7, 8, 9. The harness and display are injected; everything else is direct function calls.

**Steps:**

1. Create `cli/src/display/types.ts` exporting the `Display` interface the runner emits to — methods (all synchronous, fire-and-forget): `attemptStarted(info: { stagePosition: string; stageId: string; harness: string; model: string; attempt: number; logAbsPath: string })`, `harnessEvent(event: HarnessEvent)`, `heartbeat(elapsedMs: number)`, `stageSucceeded(info: { stagePosition: string; durationMs: number })`, `runPaused(info: { waiting: WaitingInfo; runId: string; logAbsPath: string | null; resumeCommand: string; checkpointPath: string })`, `runCompleted(info: { runId: string; recipeName: string; totalElapsedMs: number; checkpointPath: string })`, `warn(message: string)`. Also export `nullDisplay: Display` for tests.
2. Create `cli/src/runner/runner.ts` exporting `executeRun(ctx: RunnerContext): Promise<RunnerResult>` with:
   - `type RunnerContext = { checkpoint: RunCheckpoint; runDir: string; stateRoot: string; lock: LockHandle; invoker: HarnessInvoker; display: Display; harnessVersions: Record<string, string>; signal: AbortSignal; clock?: () => Date }`;
   - `type RunnerResult = { status: "completed" } | { status: "paused"; waiting: WaitingInfo } | { status: "fatal-checkpoint"; message: string }` — skill-local to the runner/commands seam; the commands map it to exit codes.
3. Implement the per-stage loop, starting from `checkpoint.stageIndex` while the lock is held:
   1. **Pre-attempt queue gate:** `scanPendingQueues`; a scan failure atomically persists `waiting-for-user` with `gate-error`; non-empty queues atomically persist `waiting-for-user` with `pending-queues`, sorted paths, and no attempt evidence. Neither branch allocates an attempt or launches the harness, and attempt history remains untouched. The pause payload uses `logAbsPath: null`. This tokenless pre-attempt pause is explicitly distinguishable on resume from both a DONE-finalized queue pause and a non-DONE attempt pause.
   2. **Attempt setup:** read the attempt-start HEAD. When entering this stage for the first time, initialize `gitCursor` with the current stage index and `headAtStageEntry`; retries preserve that stage-entry value. Set `gitCursor.observedHead` to the attempt-start HEAD. Choose the next one-based attempt number within this stage from its prior records, compute the final paths with `attemptLogPaths`, then atomically persist the checkpoint with a new `executing` attempt carrying that run-relative log path and condition `executing`. Only after persistence succeeds, exclusively create, flush, and close the header-initialized log via `createAttemptLog` (header from the snapshotted stage/profile plus the observed harness version carried in the context — `harnessVersions: Record<string, string>` on `RunnerContext`, populated from the checkpoint's creation-time observations with any current resume probe overriding its harness). A persistence failure creates no log and prevents launch; a log-creation/header failure leaves the durable attempt as `executing`, launches no harness, returns `fatal-checkpoint`, and can be recovered by the ordinary abandoned-executing resume path without colliding with the next attempt's filename.
   3. **Invoke:** render the stage prompt here via `renderStagePrompt(profile.harness, stage.skill, stage.resolvedTarget, profile.prompt)` — pure and deterministic from the snapshot, so resume reproduces it without rereading settings; call `invoker.invoke` with the profile, prompt, log path, `checkpoint.workspace.execution`, `dangerouslySkipPermissions` from the checkpoint, the abort signal, and `onEvent` forwarding to `display.harnessEvent`. The runner and adapter never reconstruct workspace cwd/sandbox/branch choices from recipe or stage names.
   4. **Post-attempt gates:** re-scan queues; on `kind: "completed"` parse the terminal outcome; read the post-attempt HEAD for every settled attempt, including non-DONE results. On a parsed DONE run boundary evaluation (`collectBoundaryStatus`, normal-attempt `evaluateBoundary`, then `finalizeBoundary` when ok) producing the `BoundaryDisposition`; after any finalization attempt, read HEAD again so an executor commit becomes the in-memory `gitCursor.observedHead`. Call `classifyAttempt`; the selected transition persists that final observation, ensuring a later resume compares current history with the actual pause-time boundary rather than either the stage-entry baseline or the pre-commit HEAD.
   5. **Transition:** every returned text result stores its complete terminal candidate, nullable parsed token, and detail in the attempt record. `advance` → finish the attempt as `done`, record `stageSucceeded`, increment `stageIndex`, reset the HEAD cursor for the next stage, persist `ready` (or `completed` with `waiting: null` when the index reaches the stage count) and continue the loop; `pause-done` → finish the attempt as `done`, persist `waiting-for-user` with the `pending-queues` waiting object (pending files listed) *without advancing*; `pause` → finish the attempt as `waiting` (or `interrupted` when the abort signal caused it — the aborted category maps to kind `interrupted` with its origin), persist `waiting-for-user` with the waiting object; every non-DONE pause message appends the DR54 warning that the attempt's file changes are unvalidated and must be reverted or deliberately committed before resuming.
   6. Every persisted checkpoint refreshes `updatedAt`; a persistence failure after a stage return reports fatal-checkpoint and never advances or launches another stage.
4. On pause/completion the runner emits `runPaused` (with the exact resume command `antmay afk resume <run-id>` and the stored waiting object) or `runCompleted`, and returns; the *caller* releases the lock (commands own acquire/release symmetry). Implement the five-minute display heartbeat as an interval timer started per attempt, cleared on settle, calling `display.heartbeat` — independent of harness output and of Sandcastle's idle timer.
5. Add `cli/src/runner/runner.test.ts` (plus `cli/src/test-helpers/fake-harness.ts` (NEW) exporting a scriptable `HarnessInvoker` fake: per-call scripted `AttemptOutcome`s, optional filesystem side effects before returning, hang-until-abort behavior). Drive full runs on the repo fixture with a **synthetic two-stage recipe** as well as `standardRecipe` (proving recipe-agnosticism, AC-6.3): all-stages-DONE completion with per-stage transitions, HEAD-cursor reset, and `waiting: null`; DONE with valid changes → boundary committed then advance; DONE + a file dropped into `.pending-decisions/` → boundary committed first, attempt `done`, pause `pending-queues`, and store the executor commit's HEAD as the pause-time observation (AC-11.3, AC-12.1, AC-12.7); BLOCKED/REFUSED/malformed/idle-timeout/provider-error pauses with attempt `waiting`, correct kinds, tokenless terminal candidates retained, pause-time observed HEAD stored, and DR54 warning text; pre-attempt non-empty queue and queue-scan-error pauses persist their waiting object with no new attempt or log (AC-11.2); boundary violation and commit-error pauses preserving the attempt, including a boundary violation with a simultaneous queue-scan failure that persists the `git-policy-violation` kind with the scan diagnostic folded in rather than `gate-error` (AC-11.6); injected pre-launch checkpoint failure creates no log and launches nothing; injected log-header failure leaves a recoverable `executing` attempt and launches nothing; fatal-checkpoint on injected persistence failure after return never advances (AC-13.3); missing `spec.md` for a later stage still launches (no artifact preconditions, AC-6.4).

**Acceptance criteria:**

- Checkpoint transitions match AC-13.3 for every path: `executing` persisted pre-spawn, `done`+advance, `pause-done` as `done` without advancing, other pauses as `waiting`, `ready`/`completed` with `waiting: null`, fatal-checkpoint halting.
- DONE boundaries are finalized (committed when validly changed) before any queue pause; queue scans happen at every DR44 point inside the loop (AC-11.1 loop portion, AC-12.1).
- Every non-DONE pause message carries the unvalidated-changes warning (AC-12.6).
- The runner consumes only snapshotted stage data and typed inputs; the synthetic-recipe suite passes through the identical code path (AC-6.3).

**Consumes:** `renderStagePrompt` (Task 3); checkpoint types, `writeCheckpoint`, `attemptLogPaths`, `createAttemptLog` (Task 5); `scanPendingQueues`, `LockHandle` (Task 6); `HarnessInvoker`, `HarnessEvent` (Task 7); `parseTerminalOutcome`, `classifyAttempt` (Task 8); `collectBoundaryStatus`, `readHead`, `evaluateBoundary`, `finalizeBoundary` (Task 9); `createRepoFixture` (Task 4).

**Produces:** `Display` interface and `nullDisplay` from `cli/src/display/types.ts`; `executeRun(ctx: RunnerContext): Promise<RunnerResult>` with `RunnerContext` carrying `harnessVersions` from `cli/src/runner/runner.ts`; the scriptable fake harness from `cli/src/test-helpers/fake-harness.ts`.
