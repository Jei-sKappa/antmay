# Task 34: Author docs/project/v3 core documents

**Objective:** Create `docs/project/v3/README.md`, `thread-model.md`, and `skill-authoring.md` with their spec-assigned responsibilities.

**Input / context:** The cutover spec `specs/001/spec.md` § "Canonical documentation" (the required content mapping — this task's checklist) and AC-11.1–AC-11.3, AC-4.4, AC-15.2 (thread-relative to `docs/threads/260711150919Z-v3-workflow-kinds/`); decision log `seed/discussions/260711154147Z-v3-thread-kind-profiles-decision-log.md`: P17 (doc architecture and responsibilities), P15 (admission bar), P62 (layout tree), P24–P26 (seed contract), P9/P57 (decisions), P11/P60 (archive lifecycle), P12 (write authority; living documentation), P21 (the one cross-thread write exception), P27 (external references), P53 (reference syntax), P61 (branch-agnostic), P31/P33/P47 + P36/P38 + P40 (the three workspaces), P52 (named limitation), P13-as-amended-by-P31 (posture), P29/P55 (composition), P30 (coherent suite), P48 (metadata), P49 (primitive bar), P35 (reconciliation vs review), P16 (separate-skill bar), P63 (no legacy awareness). These are living project documents describing V3 as it IS — write them in present tense with no history of what V2 did, no `P<N>` citations, and no references to the cutover project. Prose and structure are yours (Degrees of freedom #1) provided every mapped element is present.

**Steps:**

1. Create `docs/project/v3/README.md`: Project V3's architecture — reusable capability skills composed into three separately documented workflows (Quick, Standard, Roadmap, each linked); the conventions-first stance (workflows guide, never govern; skipped steps do not invalidate a thread); the admission bar for any future workflow (only demonstrated distinct purpose, durable artifact structure, and natural completion shape — never a subject label); subject-neutrality (the user chooses process shape; nothing routes bugs/features/docs by subject).
2. Create `docs/project/v3/thread-model.md` covering, in whatever structure reads best: the authoritative thread layout tree (exactly the P62 set: `seed.md`, `decisions.md`, optional `proposal.md`/`spec.md`/`plan.md`, `plan-tasks/`, `implementation-report.md`, `roadmap.md`, `roadmap-feedback.md`, and the three gitignored workspaces `.pending-decisions/`, `.pending-reviews/`, `.implementation-runs/` — no other folders); the seed contract (three conceptual requirements; verbatim expanded suggestion; a thread is opened FROM a template, it does not HAVE a workflow type; sparse conditional metadata, `External:` only when real, `Supersedes:` only when a known supersession relationship is useful); `decisions.md` (eager creation, `D<N>` record shape, supersession by append) and the recording rule that ANY skill eliciting a new human decision appends it before acting on it; the archive lifecycle (active tree vs `docs/threads/archive/`; no intermediate states; abandonment recorded as a decision with rationale; completion evidenced by the terminal artifact; the pre-archival warning about non-empty workspaces, which are carried along, never deleted); write authority (the P12 boundaries restated as the current rules) with the single exception that a Roadmap descendant may append to its parent's `roadmap-feedback.md`; the living-documentation distinction and the update-during-implementation rule (where no living documentation exists, none is invented); passive external references; the cross-thread reference syntax (repo-relative thread-root directory paths; `Roadmap brief:` as `C<N>`); branch-agnosticism (no thread-to-branch mapping; no skill creates or switches branches on its own initiative); the three temporary workspaces with their bundle contracts (pending-decision routing header + suggested-action section; pending-review schema with `F<N>` findings AND their manual consumption contract — any capable agent addresses findings on explicit user instruction, re-review of the target is an explicit user request never an automatic loop, and a bundle is removed once its findings are judged addressed, dismissed, or superseded; invocation-scoped run directories with report-then-remove); and the NAMED accepted limitation that repo-relative references may break when a referenced thread is archived — stated as accepted, with no repair mechanism described or promised.
3. Create `docs/project/v3/skill-authoring.md` covering: interaction posture as an authoring principle (dialogue-driven vs completion-oriented vs one-shot deliverable; no `interaction:` field or mandatory mode sections; explicit AFK invocation overrides posture; a blocked AFK repository-writing operation emits a pending-decision bundle rather than asking); one-way composition (user-invoked entry points may invoke model-invoked primitives via `/skill-name` prose; primitives never invoke entry points; no cycles) and the absolute rule that a skill never reads another skill's `references/`; the coherent-suite assumption (a missing invoked skill is an installation error); invocation-role metadata (the `disable-model-invocation: true` + `agents/openai.yaml` `allow_implicit_invocation: false` pair, always synchronized; primitives omit both and open their descriptions with the bounded precondition); the primitive-extraction bar (multiple real consumers or an error-prone shared side effect — never speculative fragmentation); reconciliation versus review semantics (fixed per-skill mutation contracts; no invocation-time modes); when a materially different capability earns a separate skill; the no-legacy-awareness rule (skills state their expected inputs precisely and contain no detection heuristics for or mentions of earlier thread layouts; a mismatched input is handled by ordinary judgment — notice and ask); and the shared-reference authoring rule (canonical files live in `shared/references/`, mirrored by script into declaring skills' `references/shared/`, which are never hand-edited).
4. Ensure none of the three documents instructs migrating, interpreting, or detecting pre-V3 threads, and none modifies or discusses `docs/workflow/v1/` or `docs/workflow/v2/` content.
