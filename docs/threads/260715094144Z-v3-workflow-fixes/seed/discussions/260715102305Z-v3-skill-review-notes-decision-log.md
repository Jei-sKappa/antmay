# Decision log — disposing the owner's review notes on the V3 workflow skills (the seed)

Thread: docs/threads/260715094144Z-v3-workflow-fixes/
Target: the seed (seed/seed.md)
Subject: walking the 16 review notes the owner wrote after reading the V3 workflow skills shipped by docs/threads/260711150919Z-v3-workflow-kinds/, and settling what each fix should be before any spec or implementation in this thread.

## P1: AFK vs attended — how completion-oriented skills obtain human input

Point: Should completion-oriented skills (`implement*`, `spec`, `plan-*`, reconciles, …) keep their current dual-branch blocked protocol (ask in chat when attended / bundle to `.pending-decisions/` when "explicitly AFK"), or collapse to a single always-bundle protocol — and do we need a separate `.pending-clarifications/` folder?

What you need to know: The canonical rule (`docs/project/v3/skill-authoring.md`, "Interaction posture") defines three postures: dialogue-driven, completion-oriented, one-shot deliverable. For completion-oriented skills, asking is "exceptional"; an explicit AFK invocation ("run AFK", "finish without me") overrides posture and mandates bundle + stop.

The skills implement this unevenly: `spec` and `plan-brief` have "## Blocked under an AFK invocation" (bundle only in the AFK case, implicitly chat-ask otherwise), while `implement` has "## Blocked and AFK" with two explicit branches — attended → ask in chat, record to `decisions.md` "anything beyond a trivial input clarification"; AFK → finish everything derivable, then bundle via `/emit-pending-decisions` and stop.

The fragile hinge: AFK-ness is detected from invocation phrasing. If you launch `implement` and walk away without saying "AFK", the skill is licensed to ask in chat and will stall for hours. The model must guess the user's availability.

The owner's notes recall a prior decision: skills are either interactive or full-AFK, and `implement*`/`spec`/`plan-*` were "born AFK". The notes also floated a `.pending-clarifications/` folder for questions that aren't thread-level decisions (badly-worded request, all intent already exists), each file naming the invoked skill, the full user request, and the clarification needed.

Decision: Unified blocked protocol, no second folder. Completion-oriented skills lose the attended-ask branch entirely: blocked = finish what's safely derivable → `/emit-pending-decisions` → concise terminal notification → stop. There is no runtime attended/AFK detection and no behavioral difference between being invoked by the user or by a tool; the skill is designed by its interaction posture alone. If the user happens to be present they invoke the resolve skill directly in the same chat right after the notification; otherwise the bundle waits and is resolved in a separate chat later. No `.pending-clarifications/` folder: clarification-vs-decision is distinguished at resolution time (the resolver appends genuine new intent to `decisions.md`; a mere request-repair answer is not recorded as a decision), not at emission time; the bundle shape gains the originating user request so a clarification is answerable from the file alone. The three interaction postures in `skill-authoring.md` stay as-is. Scattered bare "ASK the user" phrases in completion-oriented skills are rewritten to conform to the unified protocol.

Rationale: The dual-branch model keys behavior on a fact the model cannot reliably know (user availability); the unified protocol deletes that guess instead of decorating it. The owner explicitly accepts the cons (small file ceremony when attended; a run blocked at minute 0 on a trivial ambiguity burns the AFK window). Driving motivation: a planned future orchestrator tool that runs the workflow autonomously and pings the human only when new files appear in the pending folders — the filesystem queue is the interface, so every open question must be durable by construction and skill behavior must be identical regardless of invoker. Two load-bearing dependencies noted: resolved bundles must be reliably removed on resolution (otherwise the pending folder cannot serve as the orchestrator's stop/ping signal), and "finish everything safely derivable first" is what keeps resume-by-re-invocation cheap.

## P2: Failed-commit handling — diagnose and fix before BLOCKED

Point: Replace the "a failed commit is immediately `BLOCKED`, never retry" rule with a bounded diagnose-fix-retry protocol.

What you need to know: The current rule (identical `### Failed commit` section in all three implement skills — `implement`, `implement-plan`, `implement-plan-with-subagents`) forbids any retry and treats every failure as "the project telling the implementer to stop and let the user diagnose."

The rule's own examples undercut it: a pre-commit lint failure or a test broken by the task's own diff is precisely what the implementer exists to fix. Worse, the most common hook behavior in real projects — a formatter hook that auto-fixes files and fails the commit so you re-stage — turns into a full run halt under the current rule, every time.

What the rule was legitimately protecting against: agents bypassing hooks (`--no-verify`), weakening checks to get green (deleting/skipping a failing test), stash-and-retry games, flailing retry loops, and history rewriting. None of that requires a zero-retry policy — it requires guardrails on what a fix may be.

Decision: Rewrite the `### Failed commit` section (same text in all three implement skills) as a bounded fix loop: (1) diagnose first — read the actual error output, never retry blind; (2) fix in-authority causes — mechanical failures inside the task's own footprint (lint/format violations in the task's files, a hook that auto-modified files needing a re-stage, a test broken by the task's own diff, a commit subject rejected by a message linter) are fixed as part of the current task, the failed check re-run, the commit retried; (3) bounded retries — at most 3 fix-and-retry attempts per task; past the cap, or when the cause is outside the task's authority (missing sign-off config, credentials, failures in files the task doesn't own, infra errors), fall back to today's behavior: `BLOCKED` with the diagnosis in the notes (not just "commit failed"), append the progress block, stop the run; (4) guardrails unchanged and explicit — never bypass hooks, never weaken/delete/skip a check to make it pass, never stash-and-retry, `### No history rewriting` stays exactly as-is; (5) audit trail — the task report and commit body note the commit failed N times and what was fixed.

Rationale: The zero-retry rule conflated the failure signal with unfixability; the common case (formatter hook, own-diff test break) is exactly the implementer's job, while the guardrails — not the retry ban — are what actually prevent hook bypasses and check-weakening. A genuinely unfixable commit failure stays report-only rather than emitting a pending-decisions bundle (a plain defect is not a decision); the future orchestrator will detect the `BLOCKED` outcome by parsing the run's terminal output — a concern for the tool, not the skills.

## P3: Degrees of freedom must not contain mission-critical decisions

Point: Add an explicit eligibility bar to the spec's `## Degrees of freedom` section so mission-critical decisions can't be parked there.

What you need to know: Current text (`spec/SKILL.md`, "Acceptance guidance and degrees of freedom"): degrees of freedom are "the *hows* deliberately left to the implementer's free choice", and Lossless authoring allows any unsettled specific to be either taken back to the user or "marked a degree of freedom". Nothing bounds what may be granted as freedom — the escape hatch is unguarded, so a spec author under pressure to finish can legally launder a product decision into a "freedom".

Decision: Give `## Degrees of freedom` an explicit eligibility bar: a listed freedom must be an implementation-level *how* where (a) every admissible choice satisfies all acceptance criteria unchanged, (b) no choice produces a user-visible behavioral difference the user would plausibly want to weigh in on, and (c) the choice is reversible later without revising the spec. Shorthand encoded in the skill: if the choice would change what a reviewer checks or what the user experiences, it is not a degree of freedom — it is an unsettled decision and must be routed to `.pending-decisions/` per the unified blocked protocol (P1).

Rationale: Mission-critical decisions belong to the user and the decision file; an unguarded freedom section lets them be smuggled past the lossless-authoring bar. The three-part test plus the reviewer/user shorthand gives the spec author (and any later reviewer) a checkable criterion instead of vibes.

## P4: Remove the spec's "Unresolved questions" element

Point: Remove "Unresolved questions" (element 7 of the eight semantic-contract elements) so a spec at emission is fully settled modulo granted freedoms.

What you need to know: Element 7 today: "open issues that do NOT block emission; the spec is shipped with these flagged." Lossless authoring's first legal move is "leave the gap as an `## Unresolved question`, or stop and flag."

This creates a third limbo state alongside decisions and freedoms: an open question shipped in a spec has no owner and no resolution path — downstream plan/implement can neither settle it nor treat it as granted freedom. Under P1 the "stop and flag" leg now has a concrete durable form (`.pending-decisions/` bundle), which removes the original justification for shipping flagged-but-open questions.

Decision: Delete element 7 (eight semantic-contract elements become seven) and the `## Unresolved question` leg of Lossless authoring. The two legal moves for an unsettled specific become exactly: (1) mark it a degree of freedom if it passes the P3 bar, else (2) emit a pending-decisions bundle per P1 and report the run blocked. Context worth flagging that is neither intent nor freedom lives in the spec body as a stated constraint or risk note — information, not a question. When a spec run blocks, it still writes `spec.md` as complete as the settled inputs allow — every section fully elaborated, with each blocked specific marked inline at its exact location pointing at the pending bundle. This is deliberately NOT framed as writing a "partial spec": the only permitted gaps are the marked ones tied to queued decisions; the framing must not license an agent to stop early or write half a spec.

Rationale: Every open question is either implementer freedom (bounded by P3's bar) or human intent (queued per P1); a third shipped-but-unowned state undermines both. Writing the fullest derivable spec on block honors "finish everything safely derivable" and keeps re-invocation cheap; downstream safety comes from the workflow rule that nothing proceeds while the thread has open pending bundles. The owner explicitly rejected the "partial spec" framing to avoid misleading agents into under-writing.

## P5: Unify the decision-record format via a record-decision primitive

Point: The `D<N>` record format is duplicated across 7 skills (Field rules in only 2); unify it — and decide whether a `record-decision` primitive should own it.

What you need to know: Today two variants exist: `discussion` and `resolve-pending-decisions` carry the full format plus the 5-bullet Field rules; `spec`, `plan-brief`, `plan-strict`, `propose`, `merge-artifacts` each carry a compressed "Recording elicited decisions" section (field guidance squeezed into the template placeholders — that compression is why Field rules were "missing" there; it was a deliberate shortening of an exceptional-path section).

P1 ripple: all five compressed copies exist only to serve the attended-ask branch ("if writing the spec requires asking the user…"). Under P1 those skills never elicit decisions — they bundle via `/emit-pending-decisions` and stop, and recording happens when the bundle is resolved. So their "Recording elicited decisions" sections (and `merge-artifacts`' attended branch, which has the same dual-branch structure) become dead text regardless.

After that ripple, the duplication is exactly two copies, both in dialogue-driven skills whose core job is recording decisions: `discussion` and `resolve-pending-decisions`.

Decision: Extract a `record-decision` model-invoked primitive that owns the bounded discipline: `D<N>` allocation (scan for highest existing, use next integer), the record shape, the Field rules, append-only-ness, and the "create `decisions.md` with a short heading if absent" edge. Callers (`discussion`, `resolve-pending-decisions`) supply title/scope/context/decision/rationale and keep all domain judgment (what was decided, whether it's decision-grade, supersession calls). The five dead "Recording elicited decisions" sections in `spec`, `plan-brief`, `plan-strict`, `propose`, `merge-artifacts` are removed as part of the P1 fix, folded into this thread's scope.

Rationale: The decision record is the single most load-bearing format in the workflow — `decisions.md` is what every downstream skill reads as settled truth — and the one place drift is unaffordable (the two existing full copies already show small wording drift). The repo already chose primitive-owned formatting for discussion points (`/discussion-point`) and pending bundles (`/emit-pending-decisions`); extracting `record-decision` completes that pattern and clears the extraction bar (two entry points genuinely reuse it; future dialogue-driven skills get it by one prose line).

## P6: Keep /create-thread as a primitive

Point: Weigh whether normalized thread-folder creation deserves to stay an extracted primitive or should be inlined into its callers.

What you need to know: It has exactly two callers: `open-thread` (interactive thread opening) and `materialize-roadmap-threads` (batch-creates one thread per roadmap child) — formally clearing the extraction bar. What it owns is more than templating: UTC-stamp allocation, the exact two-file creation (`seed.md` + header-only `decisions.md`, nothing else), the conditional-metadata rules (`External:`/`Parent:`/`Roadmap brief:`/`Supersedes:` written only when real, never `none`), verbatim reproduction of narrative and workflow text, and a refusal discipline: a caller that can't supply the complete authorization block is bounced to `/open-thread` rather than allowed to improvise a seed. The honest alternative — inlining ~30 lines into each caller with the seed template mirrored via `shared/references/` — would still duplicate the behavioral rules as prose in two skills, and the seed is the one frozen artifact every downstream skill treats as ground truth; drift between the two creation paths would corrupt threads at birth.

Decision: Keep `create-thread` as a primitive. Its name is reconsidered in the primitive-naming record (see the naming point): the owner flagged that `open-thread` (entry point) and `create-thread` (primitive) are indistinguishable to a user who hasn't read the bodies.

Rationale: Same shape reaffirmed in P5 — a bounded, side-effecting discipline genuinely reused by more than one entry point; the batch loop in `materialize-roadmap-threads` is exactly where a normalized creator earns its keep, and the refusal rule gives the suite a single choke-point guaranteeing no thread is born with an improvised seed. The perceived problem is its routing surface (name + description), not its existence.

## P7: Formats become shared references, not primitives (revises P5)

Point: Should reusable FORMATS — the discussion-point structure and the decision-record shape — live as shared references instead of callable primitive skills?

What you need to know: This record revises P5. During the naming discussion the owner observed that `discussion-point` and the P5-decided `record-decision` do not instruct an agent to DO something — they define a format the suite uses, and format reuse is the designed purpose of the `shared/references/` + manifest sync mechanism (existing precedent: the `## Suggested workflow` templates shared by `open-thread` and `roadmap` are also formats). The razor: a primitive earns skill-hood when it owns a side-effecting discipline — edge cases, refusals, and invariants that must execute identically regardless of caller (unique-filename allocation under concurrency, normalized folder birth, merge-in-place semantics). A format is passive material that constrains what the acting skill writes. `discussion-point`'s five-part structure and facts-vs-options discipline are all format; its refusal clauses exist only to protect an invocable skill from callers, and the callers' own loops already enforce one-point-at-a-time. `record-decision`'s shape and field rules dominate; its two lines of behavior (scan for highest `D<N>`, append, create file if absent) are legitimately part of the format definition (numbering IS format).

Decision: No `record-decision` primitive is created (revising P5's extraction), and the `discussion-point` primitive is dissolved. Both become canonical shared references (e.g. `shared/references/formats/discussion-point.md` and `shared/references/formats/decision-record.md`) declared in `shared/manifest.yaml`. Wiring: `discussion` and `resolve-pending-decisions` declare both format references; `emit-pending-decisions` declares the discussion-point format, replacing its `/discussion-point` invocation with writing each point per the format reference. Primitive count drops 6 → 5. The razor is encoded in `skill-authoring.md`: reused format → shared reference; reused side-effecting discipline → primitive. The razor must not creep: the remaining five primitives (`allocate-thread`, `emit-pending-decisions`, `emit-pending-review`, `update-implementation-report`, `append-roadmap-feedback`) are genuinely behavioral and stay skills. P5's other half stands: the five dead "Recording elicited decisions" sections still come out with the P1 fix.

Rationale: Formats-as-references is internally consistent with the suite's existing design, removes two model-invocable routing descriptions (directly serving the primitives'-descriptions concern), and dissolves the awkward `discussion-point` rename question (a reference file needs no verb). Accepted costs, judged acceptable: generated copies live in N+1 places in git and depend on the scripted sync never being bypassed; the refusal choke-point as defense-in-depth is lost; nothing enforces loading a reference — but prose saying "invoke /discussion-point" was equally unenforced, so no real regression.

## P8: Primitive naming — allocate-thread; verb-first convention

Point: Rename primitives so names are verb-first, self-explanatory, and not confusable with entry points — settling `create-thread` in particular, which collides with the `open-thread` entry point.

What you need to know: The primitives after P7: `create-thread`, `emit-pending-decisions`, `emit-pending-review`, `update-implementation-report`, `append-roadmap-feedback`. To anyone not reading skill bodies, "create" and "open" are synonyms, so `create-thread` (primitive) vs `open-thread` (entry point) is a real confusion hazard. The entry point's name is the one worth protecting — "open a thread" is the user's intent language (like opening a ticket) — while the primitive's own body language is allocation ("Thread allocation", "allocate the thread folder"). Remaining pairs are already well-separated (`emit-pending-decisions` vs `resolve-pending-decisions` are complementary opposites). The `discussion-point` rename question was dissolved by P7.

Decision: `create-thread` is renamed `allocate-thread`; the other four primitives keep their names. Convention encoded in `skill-authoring.md`: primitive names are verb-first, the verb naming the bounded side effect performed on behalf of a caller; entry-point names use the user's intent language. The rename's ripples are part of the fix's scope: folder name, `marketplace.json`, README, `.vscode` commit scopes, and every `/create-thread` prose reference in callers.

Rationale: "Allocate" kills the open/create synonym collision, matches the primitive's own body language, and is unmistakably mechanical — signaling "on behalf of a caller" rather than a user-facing capability. Runner-ups considered and rejected: `scaffold-thread` (construction-flavored), `materialize-thread` (collides with the `materialize-roadmap-threads` entry point).

## P9: Primitives' descriptions — uniform exclusive gate

Point: Rework the five remaining primitives' `description:` fields so the harness routes to them only when a composing caller actually needs them.

What you need to know: Primitives are model-invoked: the harness matches their descriptions against context, so a description is a routing gate, not documentation. The current five are close to right but inconsistent: `allocate-thread`, `append-roadmap-feedback`, `update-implementation-report` open with "Use only when…" while `emit-pending-decisions` and `emit-pending-review` open with the weaker "Use when…"; and `allocate-thread`'s description enumerates every field of the authorization block (60+ words), which is redundant with the body's Precondition-and-refusal section and adds spurious-match surface. Plausible misroutings worth guarding: an agent mid-run hits a defect and routes to `emit-pending-decisions` (its body refuses defects, but only after invocation); a user's casual "create a thread for this" matching the primitive directly instead of `open-thread`.

Decision: One uniform description template for all five primitives, applied in the same change as the P8 rename: (1) front gate always "Use only when an invoking caller supplies …" — the caller framing signals a building block a skill composes into, and "only" makes the gate exclusive; (2) inputs named collectively, not enumerated (e.g. "a complete caller-authorization block for a new thread"); the body owns the field list and refusals; (3) the bounded side effect named in one clause — what it writes and nothing else; (4) a short negative gate only where a misrouting is genuinely plausible (e.g. `emit-pending-decisions` gains "never for defects, observations, or report material — only genuine open human decisions"); negatives are not sprinkled everywhere.

Rationale: Shorter, exclusively-gated, low-keyword descriptions whose only job is letting a composing caller in and keeping everyone else out; enumeration belongs to the body that enforces it.

## P10: Discussion-point format — pros/cons on creative options; Recommendation creative-only

Point: Amend the canonical discussion-point format so creative-mode options carry explicit pros and cons, and the Recommendation element exists only in creative mode.

What you need to know: The current format (moving to `shared/references/formats/discussion-point.md` per P7) says creative options must be "real candidates, not strawmen" but never requires pros and cons per option — an author can legally list bare options. The Recommendation element is currently unconditional: "your reasoned pick with its principal trade-off." In practical mode that duplicates the Proposed solution, which by definition already carries the reasoning and the pick — so practical points end up saying the same thing twice, or the author pads a fake second opinion. The V2 `seeded-discussion` skill already has this right: "creative mode: lettered options, each with real pros and cons; Recommendation — in creative mode your recommended option and why; in practical mode the Proposed solution already carries this."

Decision: Three amendments to the format reference: (1) creative options are lettered (A/B/C…), each with real pros and cons — phrased softly in the V2 style ("present concrete lettered options, each with real pros and cons"), not as a shouted MUST; lettering so the reader can answer tersely ("go with B"); (2) Recommendation becomes creative-mode-only: the reasoned pick among the lettered options with its principal trade-off, keeping the existing "if no option is genuinely preferable, say so" clause; (3) practical mode explicitly has no Recommendation element — the Proposed solution is stated once with its reasoning inline, and the format says so to stop authors re-adding it out of template instinct.

Rationale: Pros/cons are what make options decidable at a glance and tersely referenceable; an unconditional Recommendation element forced duplication or padding in practical mode. Soft phrasing chosen by the owner over MUST — the V2 wording achieved the behavior without rigidity.

## P11: Rarely-hit conditional blocks become condition + reference pointer

Point: Move conditional blocks that most runs never execute (e.g. `## Roadmap-descendant feedback`) out of skill bodies into reference files, leaving only the trigger condition and a pointer inline.

What you need to know: A SKILL.md body is loaded whole on every invocation; references are read only on demand (progressive disclosure). The roadmap-feedback block (~120 words) is carried by all three implement skills but applies only when the thread's seed has a `Parent:` roadmap reference — for most runs it's dead weight. Because the same block sits in three skills, the shared-reference mechanism applies (P7's razor: passive reusable material → `shared/references/` + manifest, mirrored into each skill's `references/shared/`); a conditional unique to one skill would use that skill's own `references/` folder. Caveats: the Raycast wrapper embeds all references unconditionally, so the extraction saves context only in progressive-disclosure harnesses; and each extraction adds a read hop when the condition does fire, with a small risk the agent skips the read.

Decision: (1) Extraction bar, encoded in `skill-authoring.md`: extract a block to a reference when (a) it executes only under a condition most runs don't meet — not merely "optional" — AND (b) it's substantial enough that inlining costs real context (more than a couple of sentences); main-path content and short conditionals stay inline. (2) Pointer shape: the body keeps the full trigger condition inline (it must be evaluated every run) plus a naturally-worded pointer to the reference — flowing prose, not a mechanical "IF X READ Y AND FOLLOW BEFORE CONTINUING" construction; only the what-to-do moves out. (3) First and only current application: the roadmap-descendant feedback block → one canonical file in `shared/references/`, mirrored into the three implement skills via the manifest. (Worked examples such as plan-strict's `### Worked Example` are handled by the next record, not this bar.)

Rationale: Context economy on the common path with the detail one read away on the rare path; the bar prevents over-extraction (pointer-fragmentation is its own cost), and natural pointer phrasing keeps skill bodies readable as instructions rather than pseudo-code.

## P12: Worked examples move to reference files

Point: Extract illustrative worked examples (currently: plan-strict's `### Worked Example`) from skill bodies into reference files, and encode the convention.

What you need to know: This is not the P11 bar: a worked example isn't rare-conditional — it's consumed on essentially every drafting run, so extraction saves few tokens. The honest arguments are different: (1) attention placement — the body is loaded at invocation start, so an inline example sits through context-resolution and input-reading steps before it's needed, while a reference read at the drafting step puts the example immediately before the imitation happens, where examples work best; (2) normative/illustrative separation per the P7 razor — rules are instructions and stay in the body; an example is passive material. Plan-strict's `### Worked Example` is ~38 lines (a complete task file plus an index excerpt) and is the only worked example in the suite. It is used by one skill only, so it belongs in the skill's own `references/`, not `shared/references/`.

Decision: Move the worked example to `plan-strict/references/worked-example.md`; the body's drafting step points to it in natural prose (e.g. "before writing the first task file, look at `references/worked-example.md` for the complete shape"). Convention encoded in `skill-authoring.md`: worked examples live in `references/` (shared only if reused), are pointed at from the step where they're consumed, and never carry a rule that exists nowhere else — anything normative stated only inside an example is hoisted into the body before extraction (plan-strict's "only sequential numbered tasks" note is a restatement of existing body rules, so nothing to hoist there).

Rationale: Bodies stay rules-only and examples land where imitation happens; the no-orphan-rule invariant prevents the extraction from silently moving normative content out of the instruction path.

## P13: The operational-steps section is named "## Procedure" everywhere

Point: Standardize the section that carries a skill's end-to-end execution sequence under a single heading name.

What you need to know: The active suite is split exactly 12/12 between `## Operation` (all reviews, all reconciles, spec, propose, discussion, plan-brief) and `## Workflow` (all three implements, all three handoffs, merge-artifacts, plan-strict, open-ticket, take-snapshot, afk-exploration, meta-prompting) — pure authoring drift, no semantic difference. "Workflow" is a loaded term in this suite: it already names the thread-level concept (the seed's `## Suggested workflow`, `workflows/quick|standard|roadmap.md`), so a section heading reusing it makes prose cross-references and grepping ambiguous. "Operation" is collision-free but weak as an action cue — it labels what the section is, not what the reader does with it.

Decision: Standardize on `## Procedure` for every skill's execution-sequence section (all 24 skills renamed). Encoded in `skill-authoring.md`: the end-to-end execution sequence lives under `## Procedure`; `## Workflow` is never used as a section name — the term is reserved for the workflow-model concept. Artifact-format headings (e.g. the `**Steps:**` field inside plan task briefs) are untouched.

Rationale: "Procedure" keeps the "this is what you execute" signal the owner valued in "Workflow" without colliding with the suite's most loaded term. Alternatives ranked and rejected: `Execution` (more abstract), `Operation` (weak action cue), `Playbook` (informal), `Steps` (collides with the task-brief field), `Run` (collides with invocation-run vocabulary).

## P14: Skills never delete from the filesystem; run workspaces persist; queue consumption is the sole exception

Point: Strip the "then remove the run workspace" instructions from the implement skills and encode a suite-wide no-deletion rule — deciding what happens to the one deletion that is semantic rather than cleanup.

What you need to know: The active suite contains exactly two kinds of affirmative deletion. Cleanup deletions: all three implement skills remove the run's `.implementation-runs/<UTC>-<ref>/` directory on every normal terminal outcome — yet `progress.md` is the source the implementation report is folded from, so deleting it right after writing the report destroys the only evidence against which a bad fold could be audited; the skills already tolerate leftover dirs (interrupted runs persist for explicit resumption), and the dirs are gitignored and tiny. Semantic deletion: `resolve-pending-decisions` removes each settled point from its bundle and deletes the exhausted bundle file — emptiness IS the queue's meaning (`thread-model.md`: "the absence of `.pending-decisions/` means there are no known pending human decisions"), and P1 made this the orchestrator's ping signal. Existing guards already align: `archive-thread` and `finish` explicitly forbid deletion. The owner added two refinements: run files differ in KIND from pending files — a run directory is a record of what happened (useful by nature), while a pending bundle is a message consumed by being answered; and even MENTIONING deletion permission in a runtime body is risky priming — a long-running agent losing focus, feeling authorized to delete, may delete the wrong thing.

Decision: (1) The three removal instructions come out of the implement skills (the step clauses and the `## Run workspace` "the directory is removed" sentences). Run directories persist after every terminal outcome. (2) Runtime bodies stay SILENT about deletion — no "never delete" clause and no "you may delete" clause; at most one positively-framed sentence stating the run directory remains in place as the run's operational trace, worded without delete/remove tokens. (3) The norm is encoded in `skill-authoring.md` as an authoring rule: never write instructions that delete, empty, or erase user filesystem content into a skill unless the user explicitly asks, with the sole exception of pending-queue consumption, where removing a settled point or an exhausted bundle completes the communication. (4) `resolve-pending-decisions` keeps its consumption deletions. (5) The no-reference rule stays: no durable artifact ever references a run-directory path — run directories remain invocation-scoped in meaning even though they persist on disk.

Rationale: Run files track what happened during implementation and could be useful later; keeping them costs nothing (gitignored) and preserves report auditability. Pending bundles exist to be immediately addressed, so their consumption is queue semantics, not cleanup. The authoring-time/runtime split honors the priming concern: the rule constrains what skill authors write, while runtime contexts never carry deletion authorization for an agent to misgeneralize.

## P15: Run-workspace naming — optional <desc> replaces V2's <ref>

Point: Unify the `.implementation-runs/` directory naming across the three implement skills and purge the V2 plan vocabulary from it.

What you need to know: The three skills disagree, and two explain `<ref>` in V2 terms: `implement` uses `<UTC>-<ref>` ("a short slug for the input"); `implement-plan` uses `<UTC>-<ref>` ("the plan's descriptor, or `001` when the thread carries one plan"); `implement-plan-with-subagents` uses `<UTC>-plan-<ref>` with the same descriptor/001 language plus a hardcoded `plan-` infix. "Descriptor" and "001" are V2 lineage concepts (`plans/NNN/…`); in V3 a thread carries one plan at a fixed path (`plan.md`), so there is nothing for `<ref>` to reference. Uniqueness never needed `<ref>`: the `<UTC>` stamp alone guarantees it.

Decision: One naming rule in all three skills: `.implementation-runs/<UTC>[-<desc>]/` — `<UTC>` mandatory and sufficient; `<desc>` an optional short kebab descriptor of the run's objective, purely for human readability (derived naturally: from the input's subject for `implement`, from the plan's title for the plan skills). No `plan-` infix, no "descriptor", no `001`, no reference semantics; the UTC stamp alone is a valid directory name.

Rationale: The old naming carried dead V2 vocabulary that confused readers ("what is the ref when there are no plan lineages?"); the descriptor's only real job is human scannability, which an optional slug serves without inventing reference semantics.

## P16: Implementation report stays a singleton, unchanged

Point: Should a thread keep the single merged-in-place `implementation-report.md`, or one report per implementation run to differentiate objectives?

What you need to know: Today the singleton `implementation-report.md` at the thread root is maintained by `/update-implementation-report`, which merges in place — "reflects only the CURRENT outcome, replacing stale content and dropping now-resolved concerns." Its consumers (`review-implementation`, `finish`, `whats-next`) read it as current truth; none wants history. Two facts changed during this walk: P14 persists every run workspace (each run's `progress.md` survives, locally/gitignored), and commit message bodies already carry per-task report blocks — so per-run history exists in two places regardless of what the report does.

Decision: Keep the singleton current-state report, unchanged (no per-run reports, no `## Run history` section added speculatively).

Rationale: The report's job in the V3 model is to be the current answer, one read, zero staleness ambiguity — consistent with one `decisions.md`, one `spec.md`. Per-run reports would recreate the V2 fold-N-files sprawl the cutover eliminated (stale-but-authoritative-looking older reports, "latest wins" rules in every consumer). Run differentiation is already served by commit bodies (durable) and persisted run workspaces (local); a `## Run history` section remains a cheap future upgrade if a real need appears.

## P17: agents/openai.yaml — interface block on every skill

Point: Extend our `agents/openai.yaml` files beyond the bare `policy` key, following the mattpocock repo's convention.

What you need to know: The mattpocock convention (`.library/sources/mattpocock_skills/.agents/invocation.md` plus ~40 examples): every skill — both roles — carries `agents/openai.yaml` holding Codex UI metadata: `interface.display_name` (title-case picker name, e.g. "To Spec") and `interface.short_description` (a terse picker line, 4–7 words: "Turn a conversation into a spec"). User-invoked skills additionally carry `policy.allow_implicit_invocation: false`; model-invoked skills have the interface block only. Our repo today: entry points ship the file with only the `policy` key (Codex pickers show them with no display name or blurb); primitives ship no file at all. Compatibility: our rule "primitives omit both declarations" refers to the `disable-model-invocation` key and the `policy` block — a primitive gaining an interface-only `openai.yaml` doesn't violate it; the policy omission remains the model-invocable configuration.

Decision: (1) Every active skill gets `agents/openai.yaml` with an `interface:` block: `display_name` (title-case skill name) and `short_description` — a terse human-facing picker line in the 4–7-word style, deliberately not a copy of the SKILL.md `description` (which for primitives is a routing gate per P9, wrong register for a UI picker). (2) Entry points keep `policy.allow_implicit_invocation: false` beneath it; primitives carry the interface block only. (3) The convention is encoded in `skill-authoring.md` and the CLAUDE.md invocation-roles section: every skill ships the file; the interface block is universal; the policy block encodes the role, in lockstep with `disable-model-invocation`.

Rationale: The interface block is free UI quality in Codex-style harnesses and keeps the two-harness role declaration honest; separating picker copy (short_description) from routing copy (SKILL.md description) lets each be optimized for its actual reader.

## P18: Pre-run input ambiguity — refuse only when a bundle is physically impossible

Point: When a completion-oriented skill cannot resolve its inputs before starting, does it refuse in chat or emit a pending-decisions bundle? (Raised as the outcome-3 spec-fault blocker in the plan adherence review `plans/001/reviews/260715155532Z-two-case-rule-spec-fault-review.md`: the plan had introduced a "two-case rule" sending all pre-run ambiguity to chat refusal — a decision this log never made.)

What you need to know: P1 settled a single universal blocked protocol and deliberately routed clarifications through the bundle (the bundle header gained the originating user request "so a clarification is answerable from the file alone"), with the rationale that every open question must be durable by construction because the planned orchestrator pings on files appearing in pending folders. The two sub-cases behave differently: which-thread ambiguity (or no thread at all) mechanically cannot produce a bundle, because `.pending-decisions/` lives inside the very thread folder that failed to resolve; which-artifact ambiguity inside a resolved thread (two lineages, unclear referent, garbled input) can be bundled.

Decision: Narrow carve-out. A completion-oriented skill refuses in chat and stops — writing nothing — only when a bundle is physically impossible: no thread exists, or which thread is meant is ambiguous. The refusal is a concise message naming the failure and ends with the `Outcome: REFUSED` line per P19. Every other pre-run input ambiguity inside a resolved thread is a clarification: emit a pending-decisions bundle per P1 (carrying the originating user request), report blocked, stop.

Rationale: Preserves P1's durability-by-construction everywhere it is physically possible — a which-artifact question is a genuine "A or B?" the resolve flow already handles — and confines the ephemeral chat path to the one case that cannot be helped. The review's finding is accepted; this record settles the source so plan tasks 1, 4, and 5 encode the rule as decided rather than invented.

## P19: Standard machine-parseable terminal outcome for completion-oriented runs

Point: Should skills end their runs with a standard token the future orchestrator can parse, and what is its shape?

What you need to know: The planned orchestrator watches the pending folders to learn what needs doing, but folder state does not say how a run ended: P2 already assumed the tool would "parse the agent output for BLOCKED", and the plan adherence review showed a chat refusal is invisible to a files-only signal. Pending files and terminal status are two different channels.

Decision: Every completion-oriented user-invoked entry point ends its final chat message with exactly one line: `Outcome: <TOKEN> — <one-line reason or pointer>`, with a closed three-token vocabulary. `DONE` — the skill completed its job (a review that emitted a findings bundle is DONE). `BLOCKED` — the run started work and stopped: queued decisions (bundle written) or an unfixable in-run failure such as a commit failure past the P2 cap. `REFUSED` — the run never started: preconditions unmet, or a bundle physically impossible per P18. There is no FAIL token. The line composes with a skill's exact confirmation message — the confirmation becomes the reason part (e.g. `Outcome: DONE — Spec written: spec.md`). Scope: completion-oriented entry points only; dialogue-driven skills (no terminal outcome), one-shot deliverable skills (the response IS the deliverable and stays unframed), and primitives (the calling skill owns the run's terminal message) are excluded. The standard is encoded in `skill-authoring.md`.

Rationale: Two channels, cleanly split: pending folders say what needs doing, the token says how the run ended — making every terminal state (including refusals) parseable by regex while staying human-readable. FAIL was rejected as dead vocabulary: an in-protocol failure is BLOCKED with a reason, and an out-of-protocol crash emits nothing and is detected by the tool through exit/absence.
