# CLI tool to orchestrate completion-driven skills: AFK delegation, notifications, session management

## What triggered this work

We want a CLI tool that orchestrates the Modular Agentic Workflow's completion-driven skills in a way that fits the documented workflow (docs/README.md, docs/thread-model.md, docs/skill-authoring.md, docs/workflows/). The desired capabilities, as originally framed: delegate AFK work from a thread to the tool and get notified on completion or when the system needs a human (decisions/reviews); and/or manage agent sessions, possibly via tmux or herdr (panes, spawning sessions/agents via the herdr CLI, or a herdr plugin). All architectural options were open at the start.

A research pass explored five candidate foundation repositories vendored under `.library/sources/`, then ran three independent design passes arguing competing architectures. The findings below are the durable record of that research; the intended outcome of this thread is to settle the architecture through discussion and carry the tool from decisions to a shipped implementation.

## Repository findings

- **`mattpocock_sandcastle`** (MIT, TypeScript/Effect, `@ai-hero/sandcastle`): library for running coding agents headless (`claude --print --output-format stream-json`, codex, etc.) in sandboxes (docker/podman/none) on git worktrees, with `completionSignal: string[]` matching against output — passing `["Outcome: DONE","Outcome: BLOCKED","Outcome: REFUSED"]` maps 1:1 onto our terminal-outcome line and the result reports which token fired. Also: structured output extraction, session `resume()/fork()`, `onAgentStreamEvent` hooks, idle/completion timeouts, warm reusable sandboxes (`createSandbox()`). It has no daemon, no run registry, no notifications, no human-in-the-loop queue — exactly the layer our tool would add. Verdict: use as a dependency for a headless execution lane, not a fork.
- **`microsoft_conductor`** (MIT, Python, `conductor-cli`): deterministic YAML-DAG multi-agent workflow engine — a paradigm mismatch as a foundation (no artifact watching, no session pool, completion is internal engine state). Borrowable patterns: human gates resolvable over HTTP/CLI (`gate-respond`, `/api/gate-status`), background detach via fork + PID files + `stop`, JSONL event log + WebSocket dashboard + replay, checkpoints persisting harness session IDs for resume.
- **`rivet-dev_agentos`** (Apache-2.0, Rust + TS): agents run inside a WASM/V8 VM on a virtual filesystem — incompatible with our host-filesystem thread artifacts; pre-1.0 with lockstep versioning. Borrowable patterns: durable actor-per-session with SQLite, crash-resumable `workflow()`/`ctx.step`, session `waiting` state + `respondPermission` round-trip for human gating, cron with overlap policy.
- **`ogulcancelik_herdr`** (AGPL-3.0 OR commercial, Rust): the strongest live-session substrate. Client/server terminal multiplexer for AI agents; sessions survive detach; newline-delimited JSON socket API (`pane.split/run/read/send-text/report-agent/report-metadata`, `worktree create`, `events.subscribe` streaming `pane.agent_status_changed` and `pane.output_matched`, `wait agent-status --status idle|working|blocked|done`, `notification show` with toast/system delivery and done/request sounds, `api snapshot` for bootstrap); plugin system (`herdr-plugin.toml` with `[[events]]` hooks that run commands on herdr events, `[[actions]]`, `[[panes]]` TUI panes). Two caveats: agent state is screen-scraped per-agent (`idle/working/blocked` + `done` attention overlay) and herdr does not parse our `Outcome:` line — the tool must read the pane or use a hook to distinguish DONE/BLOCKED/REFUSED; and the AGPL means we drive it at arm's length over its socket/CLI as a separate process, never link or embed.
- **`Fission-AI_OpenSpec`** (MIT, TypeScript/Commander): closest philosophical cousin — an artifact-driven workflow CLI where state is derived from the filesystem (artifact exists = done; `ArtifactGraph` DAG for ready/blocked), with a strict `--json` agent contract (exactly one JSON document on stdout, prose to stderr, uniform error envelope with a pasteable `fix`), workflow-as-data `schema.yaml`, and markdown parsers with code-fence masking. Zero orchestration (no background sessions, no notifications) — validating that our tool is a missing layer, not a duplicate. Adopt its CLI/JSON-contract design and parser approach.

## Three design proposals (independent creative passes)

**A. herdr-native session steward.** Every skill run is an interactive Claude Code session in a herdr pane (`pane split → agent start → report-metadata → agent prompt`), with thread/skill/run identity stamped as pane metadata tokens so the whole session view is rebuildable from `herdr api snapshot` — no daemon needed (herdr plugin `[[events]]` hooks invoke the CLI on agent-status changes). Killer property: AFK without `--dangerously-skip-permissions` — a permission prompt flips herdr's scraped state to `blocked`, which fires a notification, and the user attaches to the live pane. Detection triangulates: herdr status change (trigger) → outcome-line regex over a pane read (classifier) → `.pending-decisions/`/`.pending-reviews/` filesystem check (semantics). Risks: hard dependency on herdr for delegation, screen-scrape fragility, AGPL boundary discipline.

**B. Headless job-runner daemon.** Thin CLI + lazy per-repo supervisor daemon; executes skills via sandcastle with the three-token `completionSignal`; durable run registry in a gitignored tool directory outside threads (`run.json`, `events.jsonl`, `outcome.json` per run); status works with the daemon dead by scanning the registry (filesystem is the state, daemon is just the muscle). Safety layering for headless AFK: sandbox by default when docker/podman present, settings-profile tool allowlist instead of skip-permissions outside sandboxes, opt-in worktrees, per-run turn/runtime budgets, advisory per-thread lock against double-runs. An `attend` command execs the user into an interactive `resolve-pending-decisions` session on BLOCKED. Risks: headless permission posture is the hard problem, daemon lifecycle complexity, Effect dependency via sandcastle.

**C. Conventions-first minimalist thread radar.** Owns zero execution. Derives a repo-wide attention board purely from thread folders — needs-decisions (point counts from bundle routing headers), review-findings (severity counts), run-active/run-stale? (via `.implementation-runs/*/progress.md` mtime with an explicit staleness window), delivered, idle, archived (suppressing attention signals on archived threads, matching the thread model) — watches with fs-events, routes notifications, and prints ready-to-run launcher lines (delegation as composition: `claude "/implement-plan <thread>/"` or `herdr pane run …`), optionally executing a user-owned runner template exactly once. Its argument: the workflow's doctrine routes every meaningful pause through a file, resolution is defined as interactive, re-review is never automatic — so an execution owner either respects those rules (reducing to a launcher plus a watcher) or breaks them. Honest limits: cannot see REFUSED (writes nothing to disk) or outcome reasons without an opt-in harness hook; liveness is a guess.

## Convergent design facts (all three proposals agreed independently)

1. The filesystem is the state; the tool is a lens. Any registry/cache is rebuildable and lives outside thread folders; the tool never writes inside `docs/threads/`.
2. Detection is a triangulation: outcome-line parse (stream, pane read, or hook) cross-checked against `.pending-decisions/` and `.pending-reviews/`; bundle presence always wins for "needs a human" semantics.
3. REFUSED is the universal hard case — it writes nothing to disk by design, so observing it requires output access (stream/pane) or an opt-in harness hook.
4. The BLOCKED handoff is identical everywhere: notify, then seat the human in an interactive `resolve-pending-decisions` session; never answer bundles mechanically, never auto-chain the follow-up.
5. Adopt OpenSpec's `--json` contract; notification kinds `done`, `blocked-decisions`, `blocked-defect`, `refused`, `review-findings`; sinks desktop + ntfy-style push + herdr toast when present.
6. No automatic loops, ever — conventions-first compliance means the tool observes, executes only on explicit instruction, and suggests; a DONE never enqueues anything.

An opt-in "outcome breadcrumb" was designed in passes A and C: a Claude Code Stop hook (installed only on explicit request) that greps the session's last assistant message for the outcome line and writes a small JSON record outside the repo, keyed by thread — the only path to REFUSED and outcome reasons that leaves every skill byte-for-byte unchanged.

## Research recommendation (input to discussion, not a settled decision)

The three designs compose as layers rather than compete, in this build order: (1) the radar core first — thread scanner, bundle-header parser, attention model, `--json` contract, notification routing — useful standalone, risk-free under conventions-first, and the input any later scheduler consumes; (2) delegation via herdr composition second — the pane choreography plus an `events.subscribe` watcher gives real AFK delegation without the skip-permissions dilemma, with herdr as an optional runner rather than a hard dependency; (3) the headless/sandcastle lane later as an opt-in for truly-away sandboxed runs, behind the same radar and notification core; (4) the outcome-breadcrumb hook early as an opt-in.

## Open questions carried into discussion

1. Permission posture for AFK runs: pane-based with `acceptEdits` plus blocked-state notifications, sandboxed skip-permissions, or both as lanes?
2. Is herdr acceptable as the (optional) session substrate, given AGPL-at-arm's-length and screen-scrape fragility — or should tmux/plain-terminal runner templates have parity from v0?
3. Where may the tool write: home-directory state only, or a gitignored in-repo directory?
4. Is an opt-in write to `~/.claude/settings.json` (Stop hook) acceptable for outcome capture?
5. Is ntfy.sh acceptable as the zero-setup away-from-machine push preset?
6. Single-repo watcher first, or multi-repo from day one?

## Intended outcome

Settle the architecture and scope through discussion (starting from the six open questions and the layered recommendation), then carry the tool through this workflow's own path — likely proposal/spec/plan — to a shipped v0 of the orchestrator CLI.
