# Decisions — CLI tool to orchestrate completion-driven skills

## DR1: Outcome detection reads harness outputs and transcripts; skills stay byte-for-byte unchanged

Context: The tool must detect a completion-oriented skill run's terminal outcome (`Outcome: DONE | BLOCKED | REFUSED — <reason>`), which is emitted only in chat — REFUSED writes nothing to disk by design. Three mechanisms were candidates: (1) an embedded completion script shipped inside each skill that writes a machine-readable record and prints the outcome line for the agent to emit; (2) reading harness conversation transcripts and launch outputs; (3) changing skills' terminal behavior to write blocked/refused markers to the filesystem. Mechanisms 1 and 3 were held to a gate: acceptable only if 2 proved unreliable, because they force filesystem writes that are unnatural to the skill itself. A verification pass (three research agents plus a live probe) established:

- **Claude Code** stores transcripts at `~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl`. The directory encoding is lossy (`/`, `.`, space all become `-`), but every record embeds the raw `cwd`, and `~/.claude/history.jsonl` maps every submitted prompt + raw project path + epoch-ms timestamp directly to `sessionId`. The last non-sidechain assistant record's `text` content blocks are extractable with a small jq recipe; subagent conversations live in separate files (`<sessionId>/subagents/agent-*.jsonl`), never interleaved. `claude -p --resume` appends to the same file; interactive resume/rewind can fork a NEW file with a new sessionId whose records carry `forkedFrom` — a reader must follow fork chains. Launch levers verified live: `--session-id <uuid>` pins the transcript to an exactly predictable path; `claude -p --output-format json` returns the final assistant text itself in `result` plus `session_id`, `is_error`, `stop_reason`; `stream-json`'s first event carries `session_id` and `cwd`. Breakers: `--no-session-persistence` disables transcripts entirely, transcripts are cleaned after ~30 days, compaction behavior is uncharacterized, and the schema drifts additively across versions (parse leniently, never version-gate).
- **Sobering empirical fact**: in 30 days of retained history on the primary machine, the outcome line never once appeared as the final top-level assistant text — historically skills ran inside subagents or sessions continued past the skill run, and raw grep massively over-matches (hundreds of hits in user records from file reads of this repo's own SKILL.md sources). "Last assistant message carries the outcome" holds only when the session ends exactly at the skill run — i.e., precisely the tool-spawned case.
- **Codex** stores rollouts at `~/.codex/sessions/YYYY/MM/DD/rollout-<timestamp>-<uuidv7>.jsonl`; line 1 (`session_meta`) carries `cwd`, git repo/branch, and start time; a `task_complete` event carries the full final assistant message verbatim (verified byte-identical to the last assistant record in 29/30 sampled sessions; interrupted turns leave `turn_aborted` instead). `codex exec --json` emits the thread id up front, which maps 1:1 to the rollout path; `--output-last-message <path>` writes the final message to a chosen file. Resume appends in place; fork copies to a new file; subagent rollouts share the cwd and must be filtered on `thread_source`. Codex has no interactive session-id pinning flag, so identifying an interactive Codex session the tool did not spawn stays a (strong) heuristic join on cwd + start time + first prompt.
- **Borrowable parsing** lives in `.library/sources/jhlee0409_claude-code-history-viewer` (MIT): UI-free, tolerant (all fields optional except `type`, bad lines skipped, no version gating) transcript parsing in `src-tauri/src/models/message.rs`, `src-tauri/src/commands/session/load.rs`, `src-tauri/src/export.rs` (`extract_blocks`), `src-tauri/src/utils.rs` (path decode), `src-tauri/src/commands/watcher.rs` (live-tail detection).
- **Scrape probe** (herdr 0.7.4 driving a live interactive Claude Code session, headlessly end to end): pane choreography, `wait output --match`, and `wait agent-status --status idle` all worked; herdr auto-detected the agent and tracked `working → idle` with no integration installed. But the Claude Code TUI wraps long messages by writing REAL newlines with two-space continuation indents (even herdr's `recent-unwrapped` read cannot undo them) and prefixes `⏺ ` — so the outcome TOKEN is reliably matchable at sane pane widths while the reason text is not reliably extractable from a scrape. Pane text has no role structure (an echoed instruction containing the pattern triggers the matcher), and screen content is version-fragile (a footer-text match failed across versions during the probe).
- **Generalization**: Gemini CLI and opencode also keep readable on-disk transcripts; Cursor uses reverse-engineered SQLite (weak fit). Transcript reading covers Claude Code + Codex now and extends naturally; scraping covers any terminal harness token-grade.

Decision: Adopt transcript/launch-output reading, leaving every skill byte-for-byte unchanged, structured as a four-rung detection ladder the tool walks strongest-first per run: (1) **launch output** for tool-spawned headless runs (`claude -p --output-format json`, `codex exec --json`) — fully deterministic, no file access needed; (2) **known-path transcript tailing** for tool-spawned interactive sessions with pinned identity (`--session-id` on Claude Code; thread-id capture on Codex) — deterministic file, yields token and full reason; (3) **multiplexer pane scraping** (output match + agent-status as trigger, anchored token match as classifier) — works with any terminal harness, token-grade only, reason best-effort, confirmed against a transcript whenever one exists; (4) **history-index matching** for sessions the tool never spawned (`~/.claude/history.jsonl`; Codex `session_meta`/`history.jsonl`) — explicitly labeled best-effort. Cross-cutting rules: `.pending-decisions/` / `.pending-reviews/` bundle presence always wins for "needs a human" semantics; matchers must tolerate their own pattern appearing in echoed input and file reads (never raw-grep a transcript or screen); parsers are lenient and additive-drift-tolerant. The embedded-completion-script and skill-writes-markers mechanisms are rejected on this evidence per the gate above — not forbidden forever, but not needed.

Rationale: The deterministic rungs fully cover the tool's delegation job, and REFUSED — previously unobservable — becomes observable with reasons in every spawned lane without touching a skill. The only genuinely weak spot (outcomes of interactive sessions the tool never spawned, where the outcome line sits mid-conversation and quote-vs-emit is ambiguous) is a radar nice-to-have, not a delegation requirement, and bundles cover its BLOCKED semantics anyway. Accepted trade-offs: harness coverage for reason-grade detection is limited to transcript-keeping CLIs; transcript formats are internal and additively unstable; `--no-session-persistence` and retention windows are guarded breakers.

## DR2: Execution model — mode-agnostic core; tool-spawned interactive multiplexer sessions are the v0 delegation lane

Context: Completion-oriented skills can be executed four ways: (1) **manual** — the user opens any harness, invokes the skill, and watches; no tool involved; (2) **tool-spawned interactive** — the tool launches the harness in a terminal-multiplexer pane, watches for the outcome, notifies, and lets the user attach; (3) **headless on the host** (`claude -p` / `codex exec`); (4) **sandboxed headless** (sandcastle-style: container + worktree + completion signal). The modes trade permission-friction against autonomy and map onto DR1's rungs (mode 2 → rungs 2–3; modes 3–4 → rung 1). The user's actual practice conditions the choice: locally they run automode and are effectively never stopped by permission prompts; long implementation runs happen on a disposable remote server deliberately run with all permissions skipped; they value watching agents work. The suite must remain fully usable with no tool installed (the CLI is optional by design).

Decision: The CLI is a mode-agnostic core — run identity (thread × skill × session), the DR1 detection ladder, notification routing, and status/attention reporting — with execution lanes bolted on. Mode 2 (tool-spawned interactive sessions in multiplexer panes) is the v0 delegation lane. Modes 3 and 4 are future add-ons that plug into the same core (mode 3 consumes DR1 rung 1 directly; mode 4 adds sandboxing later). Mode 1 is permanently guaranteed: the tool never becomes mandatory for any skill, and at most observes user-run sessions via DR1 rung 4. The core must never assume a pane exists — no pane concept may leak out of the execution-lane layer, so that adding modes 3/4 is an extension, not a rewrite.

Rationale: Mode 2 matches how the work actually happens today (long interactive runs on a remote box the user attaches to) while adding the two missing pieces — completion/blocked notification and outcome detection. The workflow's doctrine routes every meaningful pause through interactively-resolved bundles, and a live pane preserves full conversational context for that takeover, where headless lanes require a resume dance. It is also the cheapest lane to build (the multiplexer supplies session durability and attach). Permission-prompt preservation during AFK — mode 2's unique structural property — was noted but is NOT the driver for this user given automode/skip-permissions practice; the lanes carry per-session permission flags rather than imposing a posture.

## DR3: Multiplexer integration — thin adapter interface; herdr is the first adapter, tmux the next; no hard dependency

Context: DR2's mode 2 needs a terminal multiplexer to host durable, attachable interactive sessions. Candidates: herdr (installed locally, AGPL-3.0/commercial, 0.7.x; probe-verified rich socket/CLI API — `pane run/send-text/read`, blocking `wait output --regex` / `wait agent-status`, auto-detected per-agent status, events, toasts, worktree helpers, `--remote <ssh-target>`) and tmux (ubiquitous, decades-stable, and what the user already runs on their remote machine — the primary delegation environment; equivalent commodity primitives via `send-keys`, `capture-pane`, `pipe-pane`, `monitor-silence`, hooks, per-pane user options). The DR1 ladder makes the multiplexer's role deliberately dumb: rung 2 (transcript tailing with pinned identity) is multiplexer-independent, and scraping is only the fallback trigger — so the primitives the tool actually requires are commodity across multiplexers.

Decision: Define a deliberately thin multiplexer adapter interface — approximately: spawn a pane (cwd, env, command), send text, read screen, report liveness, list/enumerate with metadata stamping, and produce an attach handle — and keep ALL detection intelligence (transcript tailing, outcome classification, notification routing) in the mode-agnostic core, never inside an adapter. Ship the herdr adapter first; consume herdr's extras (agent-status scraping, blocking waits, events, toasts, worktree helpers) only as optional enrichments behind capability flags, never as load-bearing signals. The tmux adapter is the first follow-on milestone (`pipe-pane`/polling in place of `wait output`, `monitor-silence` hooks approximating idle, `@`-prefixed pane user options for metadata). herdr is driven at arm's length over its socket/CLI as a separate process only — never linked or embedded (AGPL boundary). No multiplexer is a hard dependency of the tool.

Rationale: The needed surface is commodity, so a hard herdr dependency buys almost nothing — the "simpler" path still needs the same six operations — while welding the only delegation lane to a young AGPL project whose scraped status was observed to be version-fragile during the probe. herdr's differentiators (dashboard, notifications, agent awareness) overlap with what this tool itself intends to provide. tmux support is not speculative: the user's remote machine — where long implementation runs live — runs tmux today. Caveat recorded: interactive Codex offers no session-id pinning, so Codex-in-a-pane identification stays heuristic regardless of adapter choice (a per-harness gap, not an adapter gap). Verification item carried forward: confirm notification delivery is useful when the multiplexer/server runs on the remote box, since only local delivery was probed.

## DR4: All tool-written state lives in the home directory; the repository stays byte-clean

Context: The tool's genuinely primary state is small and machine-local: the run identity binding for every spawned run (thread × skill × repo path × harness × session identity such as the pinned Claude Code `--session-id` or captured Codex thread id × adapter handle), a notify-once ledger so restarts do not re-notify, and cached detected outcomes (which outlive the ~30-day transcript retention). Alongside it sit rebuildable caches (tail cursors, the derived status index) and user-edited configuration. The candidate locations were home-directory-only state, a gitignored in-repo directory, or a split. The user's delegation practice syncs the project to a disposable remote box with rsync, and DR1's detection inputs (transcripts, history indexes) already live under the home directory.

Decision: All tool-written state lives under a per-user home-directory state root (e.g. `~/.local/state/<tool>/`), keyed by repository path. The tool writes nothing inside any repository: no state directory, no `.gitignore` edits — its optionality extends to leaving the repo byte-for-byte untouched. Whether an optional per-repo configuration file (user-edited, never tool-written) is offered is deliberately left as a separate later decision, as is how run identity is stamped and rebuilt (registry vs pane metadata).

Rationale: Nearly everything the tool records is machine-local — session IDs, pane handles, transcript paths mean nothing on another machine — so state synced between machines via git or rsync arrives actively wrong; under the rsync-based remote workflow, in-repo placement is not merely untidy but incorrect. Home-dir placement also gives multi-repo watching a single enumeration point. Accepted trade-offs: repo-path keying orphans records when a repo moves (mitigable by re-keying, since state is rebuildable from transcripts and thread folders), and each machine sees only its own registry — which matches reality, because the runs are per-machine.

## DR5: Remote delegation topology — symmetric install, the remote instance watches its own runs

Context: DR2's delegation lane must cover the user's remote practice: long runs happen on a disposable remote box (project synced over with rsync, sessions held in tmux) precisely so the work does not depend on the local machine staying on. DR1's detection inputs are machine-local by nature — launch stdout, `~/.claude`/`~/.codex` transcripts, and panes all exist on the box running the harness — so a watcher must either run there or tunnel every signal over SSH, and an SSH-driven local watcher dies exactly when the laptop closes. The candidate topologies were a symmetric install with a remote-side watcher, a local-only tool driving the remote over SSH, and a remote-only install with no local view.

Decision: The same tool is installed on both machines, and the instance on the box that runs a session is the authoritative watcher for it. A local `spawn --remote <host>` toggle performs the delegation choreography: rsync the project over → ensure the tool is present on the remote (provisioning is part of the choreography) → have the remote instance spawn the harness and record the run binding in its own DR4 registry → keep a local delegation stub (host, run id, attach handle, with the detected outcome repatriated into it when learned). The remote instance walks the DR1 ladder on its own filesystem and pushes notifications itself, directly from the remote box. Local `status` federates remote runs via `ssh <host> <tool> status --json`; attach wraps the multiplexer attach in `ssh -t`. The role split is deliberate: the remote box is the workhorse; the local machine is where decisions are resolved and where remote runs are observed. Consequent packaging constraint: the tool must be provisionable in one line on a fresh box (self-contained binary or npx-runnable). DR4 applies per machine — the remote registry dies with the disposable box, which is acceptable because outcomes are pushed on detection and durable artifacts live in the repo copy.

Rationale: Not depending on the local machine being on is the stated reason the remote box exists, which rules out local SSH-driven watching outright; the symmetric design is the remote-only option plus exactly the desired automation, and federation stays thin because it reuses the already-decided `--json` contract over SSH. Push delivery originating remote-side (plain HTTPS) largely dissolves DR3's carried-forward verification item — what remains is confirming a push sink works from the remote box, while desktop-toast sinks stay local-only by nature. Accepted trade-offs: a per-box provisioning step on every fresh disposable server, and two registries to reason about (matching DR4's per-machine reality).

## DR6: Sync ownership — one active session per project folder; sync-back is the user's; spawn's forward-rsync carries a clobber guard

Context: DR5's delegation leaves two repo copies (local and remote) with rsync — which overwrites wholesale rather than merging — moving work between them, raising the lost-update question: who owns synchronization, and under what concurrency assumptions? The user's practice is one active session per project folder (parallelism happens across git worktrees, i.e. separate folders), and resolving a `.pending-decisions/` bundle is legitimate against either copy since bundles are self-contained by design and BLOCKED is a terminal outcome — resolution is always a fresh session, never a rejoin. The tool already owns exactly one sync: DR5's `spawn --remote` pushes local → remote as its first step, which is where an un-pulled remote result would be silently destroyed by a stale local copy — a clobber performed by the tool itself, with no concurrency involved.

Decision: Three-part model. (1) One active session per project folder is an assumed invariant, advisorily enforced: `spawn` warns/refuses when the registry already shows an active run on that folder. (2) All sync-back (remote → local) is the user's responsibility: the tool never pulls artifacts on its own; at most it suggests the pull command in a done/blocked notification's suggested actions. (3) One guard on the tool's own push: `spawn --remote` refuses to forward-rsync when the registry shows a completed remote run on that folder whose results have not been pulled back since, overridable with an explicit flag.

Rationale: The single-active-session invariant matches the worktree-based practice and dissolves the concurrent-write hazard cheaply (proposal B's advisory lock, repurposed per-folder); user-owned sync fits the tool's conventions-first character (observe and suggest, never manage working copies). The guard exists because the one destructive rsync in the system is tool-initiated, so user responsibility alone cannot cover it — the tool merely refuses to be the instrument of a silent clobber while remaining overridable. Trade-off: the registry must track pulled-back-ness (or a proxy such as last remote completion vs last pull timestamp), and a stale registry could produce a false refusal — acceptable given the explicit override.

## DR7: Notifications go through a pluggable sink module; ntfy.sh is the default push preset, never a hard dependency

Context: DR5 requires pushes to originate on the remote box, so the away-from-machine channel must work with plain outbound HTTPS and no listening infrastructure on the user's side. The notification kinds are settled (`done`, `blocked-decisions`, `blocked-defect`, `refused`, `review-findings`). ntfy.sh fits the zero-setup requirement but has an honest trust model to surface: the topic name is the only credential (anyone who knows or guesses it can subscribe), and payloads transit and rest (cached ~12h) on a public third-party server without end-to-end encryption — while our payloads naturally carry thread slugs, repo names, and outcome reason text.

Decision: The core routes all notifications through a notification module with a pluggable sink interface — mirroring DR3's adapter pattern — so no sink is ever a hard dependency; sinks planned across the suite's life include desktop toast, push, and multiplexer toast where present. ntfy.sh is the default push preset: on first setup the tool generates a long random topic (topic-as-bearer-token), prints the subscribe URL/QR, and publishes via HTTPS POST. The ntfy server URL is a config field, so a self-hosted ntfy instance is the same sink with one line changed. A `terse` payload mode is offered that sends only kind + thread slug — no reason text, no repo path — for users unwilling to put work fragments on the public server.

Rationale: ntfy.sh is the only candidate that satisfies the remote-origin requirement with zero accounts and zero infrastructure, and committing to it as a preset behind a sink interface commits the architecture to nothing — heavier alternatives (Pushover, Telegram, Gotify) become additional sinks, not redesigns. The privacy trade-off is accepted knowingly: a long random topic is the same trust model as a bearer token, the exposure is mitigated by self-hosting or terse mode, and the default favors the zero-setup experience.

## DR8: No harness-hook outcome breadcrumb — the tool never writes outside its own state root

Context: An opt-in Claude Code Stop hook was designed during research (passes A and C) to capture outcome lines from sessions the tool did not spawn: installed in `~/.claude/settings.json` on explicit request, firing on every session stop, writing a small breadcrumb record to the DR4 state root. Its original justification — REFUSED being unobservable anywhere — was overtaken by DR1, whose rungs 1–3 make every tool-spawned run fully observable with reasons; what remained was record-keeping completeness for manual (mode 1) runs, where the user is by definition present and sees the outcome directly. The hook carried real costs: it would be the only tool feature writing outside the DR4 state root (into user-global harness config), it runs on every session stop across all projects, it is Claude Code-only (Codex has no equivalent), and hook schemas are internal, drifting surface.

Decision: The breadcrumb hook is dropped entirely — not built, not deferred, no breadcrumb record format reserved. Detection of sessions the tool did not spawn remains exactly DR1's rung 4 (history-index matching, explicitly best-effort), with `.pending-decisions/` / `.pending-reviews/` bundle presence carrying "needs a human" semantics as always. Consequent invariant, stated absolutely: the tool never writes outside its DR4 home-directory state root — no harness config files, no hooks, no exceptions.

Rationale: The hook's remaining value after DR1 was completeness of `status` for runs the user personally watched — a record-keeping nicety that does not justify machinery which edits harness config and executes on every Claude Code session across the machine. Accepted trade-off, eyes open: manual-run REFUSED outcomes stay invisible to the registry permanently (unless a future decision supersedes this one); the user judges that acceptable since manual runs are watched runs.

## DR9: Multi-repo by construction, cwd-scoped by default

Context: The seed asked whether the tool watches a single repository first or multiple repositories from day one. Prior decisions largely dissolved the question: DR4's registry is one home-directory state root keyed by repo path (there is no single-repo registry to outgrow), and DR5/DR6 made watching per-run rather than per-repo — the instance that spawned a run watches that run to its outcome wherever its repo lives, so runs across many repos are the same machinery as one. What remained was only the command-surface default and the radar's sweep scope. A repo becomes known to the registry the first time the tool acts in it; there is no registration ceremony.

Decision: The tool is multi-repo by construction and cwd-scoped by default. `status` run inside a repository reports that repository; `status --all` and the attention/radar view enumerate every repository the registry has seen, flagging pending bundles and unfinished runs across all of them. `--all` is implemented as nothing more than not filtering the registry by cwd; no per-repo daemons or watchers exist because watching is per-run.

Rationale: This matches actual practice — standing in one project most of the time, occasionally wanting a cross-project "anything need me?" sweep — and costs v0 nothing, since both views read the same state root. Trade-off: none of substance; the single-repo alternative would have saved no code while forcing a schema migration later.

## DR10: v0 surface is a plain CLI with interactive prompts plus push notifications; no bespoke TUI dashboard

Context: The tool's thesis is event-driven and away-from-machine — DR5 pushes notifications from the remote box rather than expecting the user to watch a screen — and the desire to watch agents work is already served by DR2/DR3's `attach` into the live multiplexer pane (the real agent, not metadata about it). A persistent auto-refreshing TUI dashboard was weighed against a plain CLI: the dashboard is the most expensive component to build, is a pull-model surface against an event-model tool, would re-implement the dashboard herdr already ships (with no tmux-parity story), and forecloses nothing if deferred because the `--json` contract makes any later view a mere client of it.

Decision: v0 ships as a plain command-line tool — a set of one-shot commands emitting human-readable text or `--json`, plus the DR7 push notifications — with no bespoke persistent TUI dashboard. Watching a running agent is done by attaching into its real pane (DR3), not by a metadata board. The CLI MAY use transient interactive prompts in the style of installer/scaffolder wizards (selection menus, confirmations) to gather input when flags are absent; these are one-shot, blocking, ask-then-exit prompts and are explicitly distinct from a persistent auto-refreshing dashboard, which v0 does not build. A cheap live `watch` command (an auto-refresh loop over `status --all --json`, no widget framework) is a permitted fast-follow if an at-a-glance board is later missed, but is not part of v0 and must not grow into a full TUI.

Rationale: A plain CLI matches the tool's reason to exist — get pinged, then attach to the real session — at the lowest cost while foreclosing nothing, since a TUI would be a later client of the already-adopted `--json` contract. Interactive prompts add installer-grade ergonomics without crossing into the pull-model dashboard that fights the event-driven thesis and duplicates herdr. Trade-off accepted: no single-glance live board of one's own in v0; the cross-project view is a command one runs (`status --all`), and `attach` covers the genuine watching need.

## DR11: v0 scope — local `spawn` (with `--attach`) plus a minimal `status` readout; everything else deferred

Context: With the architecture settled (DR1–DR10), v0's command surface needed sizing. The consideration: deferring both notifications and a defined `status` would leave DR1's detection ladder with no consumer, reducing `spawn` to a bare launcher over herdr/tmux. The chosen floor keeps the detection ladder plus one reader of its output, so v0 is genuinely "delegate locally, walk away, come back and see how it ended" rather than a launcher macro.

Decision: v0 ships exactly two commands, local-only:
- **`spawn`** — the delegation entry point (DR2 mode 2). Interactively prompts for (or accepts as flags) thread, skill, harness, and adapter; launches the session into a multiplexer pane through the DR3 adapter interface; stamps run identity and registers the binding in the DR4 registry; starts the per-run watcher walking the DR1 ladder to record the outcome. A `--attach` flag immediately attaches the user to the spawned pane so they can watch; without it, `spawn` launches and returns while the watcher runs. Honors DR6's per-folder active-run guard.
- **`status`** — the readout. Its v0 job is deliberately minimal: list the user's runs with their detected outcome (done/blocked/refused/active) and any pending `.pending-decisions/`/`.pending-reviews/` bundles, cwd-scoped by default per DR9, with `--json`. Richer attention-board framing is left for a later decision.

Deferred from v0 but kept live as future lanes, and not to be blocked by v0's implementation: remote/ssh delegation and `pull` (DR5/DR6 — v0's local `spawn` MUST route through the same adapter interface and identity/registry model so remote becomes an added lane, never a rewrite); notifications (DR5/DR7 — deferred as sequencing only, architecture unchanged); a separate interactive `resolve` command (the human runs the resolution skill inside a session directly); a foreground headless `run`/`exec` verb (arrives with DR2 mode 3); and the `watch` auto-refresh loop (DR10).

Rationale: `spawn` + detection→registry + a minimal `status` is the smallest slice delivering the tool's actual thesis while exercising every core subsystem (adapter interface, identity/registry, detection ladder) that the deferred lanes will extend — so the deferrals are additive, not architectural debt. Folding attach into `spawn --attach` avoids a separate command for v0 while preserving the watch-the-real-agent path. Deferring notifications is safe locally because a present user reads `status` directly. Trade-off accepted: without notifications, v0 is pull-model (the user checks `status`), which is acceptable only for local use and is the first thing the deferred notification lane upgrades.

## DR12: Public tool identity is `antmay`

Context: The specification leaves the tool, binary, and package name as an implementation degree of freedom and uses `<tool>` throughout its command examples. A stable public identity is needed before package metadata, command help, state-root naming, and end-to-end expectations can be authored.

Decision: Name the tool and executable `antmay`. User-facing command examples use `antmay spawn` and `antmay status`; package names and the per-user state root use the same `antmay` identity where their respective naming conventions permit it.

Rationale: `antmay` gives the workflow suite and its orchestration CLI one recognizable identity and removes a pervasive placeholder before implementation. The coupling between the suite name and executable is intentional; a later rename would touch packaging, state discovery, documentation, and test expectations rather than being treated as a free cosmetic substitution.

## DR13: TypeScript/Bun workspace follows the `jastr` CLI conventions

Context: The specification leaves the implementation language, runtime, and source layout open while requiring an `npx`-runnable package or self-contained binary and a mode-agnostic core separated from multiplexer concerns. The established local reference for CLI projects is `.library/sources/Jei-sKappa_jastr`, which uses a strict TypeScript ESM Bun workspace, separates a reusable engine package from the executable CLI package, builds for a Node runtime, and centralizes repository-wide build, typecheck, test, format, and check scripts.

Decision: Implement `antmay` as a strict TypeScript ESM Bun workspace targeting Node.js. Follow `jastr`'s package boundary with `packages/core` for the reusable mode-agnostic orchestration model and `packages/cli` for the `antmay` executable, command parsing, process integration, and concrete execution adapters. Mirror the reference's root workspace scripts and toolchain conventions: Bun for workspace management and builds, a Node-targeted executable package suitable for `npx`, Commander with typed command definitions, Vitest for tests, Biome for formatting/checking, and strict shared TypeScript configuration.

Rationale: This is the author's established CLI development shape and directly supports the spec's core/adapter boundary while keeping provisioning compatible with `npx`. The extra package boundary has some workspace overhead in v0, but it prevents command and terminal-process concerns from leaking into the reusable detection, registry, and status model that later execution lanes must share.

## DR14: Test the real CLI through injectable process boundaries and retain a hands-on herdr smoke path

Context: `antmay` is intrinsically interactive and process-heavy: `spawn` launches a harness through herdr, a watcher reads harness transcripts and pane state, and results are persisted in a per-user registry. The specification already requires every promptable input to have a flag so commands can run non-interactively, but a dry-run that avoids spawning would not exercise the orchestration behavior most likely to fail. The `jastr` reference establishes the preferred end-to-end shape: invoke the real CLI as a subprocess in isolated temporary roots, replace external executable boundaries deterministically, and assert exit code, stdout, stderr, and filesystem effects through traceable case manifests.

Decision: Test `antmay` by faking its external boundaries rather than adding a fake execution branch inside the product. The normal automated suite invokes the real built `antmay` CLI with complete flags against temporary repositories, state roots, and transcript roots. Concrete herdr, Claude, and Codex executable paths and harness-data locations are injectable so checked-in scripted shims can simulate process responses, pane lifecycle, transcript creation, terminal outcomes (`DONE`, `BLOCKED`, `REFUSED`), malformed or missing transcript data, and ended-without-outcome behavior. Case-driven end-to-end fixtures follow the `jastr` convention and trace cases to the specification's acceptance criteria while asserting observable command output, registry state, and write boundaries. Separately, provide a documented hands-on smoke path that uses real herdr with a scripted fake harness, exercising actual pane creation, attach behavior, watcher operation, and outcome transitions without spending a real agent run; retain an optional manual smoke check using real herdr plus a real Claude or Codex session. A test-only built-in harness mode is not added to `antmay`, and non-spawning or dry-run coverage is not treated as evidence that the orchestration path works.

Rationale: This preserves one production execution path while making its unstable external dependencies deterministic at process and filesystem seams. It gives fast repeatable regression coverage and the real terminal experience needed to validate an interactive tool. The principal cost is additional fixture and shim infrastructure, but that cost directly exercises the adapter and detection contracts instead of maintaining a parallel fake implementation that could drift from them.

## DR15: Each run is observed by a private packaged worker process

Context: DR11 requires `spawn` to return while observation continues, but a Node.js CLI process cannot keep timers, promises, or worker threads alive after the command exits. Continued detection therefore needs another process, an existing service, or delayed reconciliation by `status`. A hidden `antmay __watch` command was considered as shorthand for a detached process, alongside a single per-user daemon, herdr-owned hooks, and status-only observation.

Decision: Ship a private worker entry point inside the CLI package and start one detached Node.js worker process for each successfully registered run. The worker receives the run identity through an internal process contract, observes only that run, atomically records its terminal or ended-without-outcome classification, and exits. It is package-internal implementation—not an `antmay` subcommand, not shown in help, and not part of the public CLI contract. Worker diagnostics and any heartbeat or lease data live under the DR4 state root. `status` synchronously reconciles registry state against current transcript and pane evidence before printing, and restores observation for a still-active run when its worker is no longer healthy. No global or per-repository `antmay` daemon is introduced, and herdr plugins or events never own correctness; future notifications consume the same idempotent terminal transition recorded by the worker.

Rationale: A private per-run process is the smallest mechanism that genuinely survives the spawning shell, keeps runs isolated, and preserves the adapter-independent detection model. It avoids the IPC, installation, upgrade, and service-management surface of a daemon while retaining a recovery path and a direct extension point for notifications. The trade-off is one small process per active run, which is acceptable under the existing one-active-run-per-project-folder convention and can be reconsidered if real multi-repository use demonstrates process-count pressure.

## DR16: v0 invokes only an explicit catalog of Antmay completion-oriented skills

Context: `spawn` must offer or accept a skill and then observe its terminal outcome, but the installed skill metadata distinguishes invocation role only: dialogue-driven and completion-oriented user entry points carry the same frontmatter and picker-policy shape. Inferring execution posture by searching natural-language skill bodies for `Outcome:` would be brittle, while accepting any installed skill would allow dialogue-driven or one-shot operations that do not satisfy the watcher protocol.

Decision: Ship a versioned catalog in `antmay` containing the Antmay suite's supported completion-oriented entry points and the harness-specific invocation identity for each. The interactive picker lists only catalog entries, and `--skill` accepts only a catalog name. Before creating a pane or registry binding, `spawn` verifies that the selected skill is available to the chosen harness; an unavailable skill is a preflight command failure with a precise installation or remediation instruction and produces no run. The catalog belongs to the CLI distribution and is maintained alongside the suite in this repository; it requires no metadata or behavioral change inside any `SKILL.md`. Arbitrary installed skills and raw harness prompts are outside v0.

Rationale: `antmay` orchestrates the Modular Agentic Workflow rather than acting as a generic terminal launcher. An explicit catalog makes eligibility, picker contents, native invocation, failure behavior, and tests deterministic without coupling the optional tool to prose heuristics or adding tool-specific metadata to every skill. The trade-off is that newly added completion-oriented skills require a catalog update and a compatible CLI release before `antmay` offers them; that controlled versioning is preferable to silently watching a run that can never emit the expected terminal outcome.

## DR17: v0 supports macOS and Linux, not native Windows

Context: `antmay` depends on detached worker processes, Unix-oriented home-state conventions, harness transcript locations, executable discovery, and herdr process behavior. The environments it is intended to serve are macOS development machines and Linux local or remote hosts; native Windows would require additional process-lifecycle, path, shell, and dependency behavior that is not exercised by the target workflow.

Decision: Support macOS and Linux on the minimum Node.js version established by the workspace runtime contract. WSL may operate as a Linux environment when the required harness and herdr executables are available, but it carries no separate compatibility promise. Native Windows is outside v0 scope. Automated coverage should exercise both supported operating systems where the project CI environment permits, and the documented hands-on herdr smoke procedure must be applicable on macOS and Linux.

Rationale: This states the real deployment boundary and prevents implementation and acceptance work from silently expanding into an unused platform. macOS and Linux cover the local and remote workflows that motivated the tool; native Windows support can be reconsidered only from demonstrated demand and with explicit process, filesystem, and dependency acceptance criteria.

## DR18: Terminal outcomes never close or mutate the retained harness pane

Context: A worker can detect and record a terminal outcome while the interactive Claude or Codex process remains open in its herdr pane. Automatic closure would prevent pane accumulation, but it would also destroy the exact session a user may want to inspect, attach to, or continue manually after DONE, BLOCKED, or REFUSED.

Decision: v0 never closes, sends input to, or otherwise alters a pane merely because its run reached a terminal or ended-without-outcome classification. Recording the classification stops the run's worker and removes the run from the one-active-run guard even when the pane remains live. Human-readable and JSON status data retain the adapter attach handle or an exact attach hint for the preserved pane. Pane cleanup remains an explicit user action performed through herdr; `antmay` ships no automatic retention timer or implicit cleanup behavior in v0.

Rationale: Preserving the pane protects conversational context and matches the tool's watch-and-take-over use case. The accepted trade-off is that panes can accumulate until the user cleans them up, which is preferable to destructive automation in the first release; an explicit cleanup capability can be considered later from observed usage.

## DR19: v0 adds a first-class `attach` command

Context: This record supersedes DR11's requirement that v0 ship exactly two commands. `spawn --attach` joins a pane only at launch time; after an ordinary detached `spawn`, the registry still knows the adapter and pane handle but the user would have to copy and run a herdr-specific command to inspect the active or retained terminal session. That leaks the first adapter's command surface and would not generalize to tmux or remote execution.

Decision: v0 ships three public commands: `spawn`, `status`, and `attach`. `antmay attach <run-id>` non-interactively joins the pane recorded for that run through its execution adapter. When the run identifier is omitted, the command uses the sole attachable run in cwd scope or presents an interactive selection when several attachable runs exist; a non-interactive invocation with an ambiguous or missing run identifier fails with a precise remediation instead of prompting. Both active runs and terminal runs whose panes remain available are attachable. `spawn --attach` delegates to the same attach operation after successful launch and registration. A missing or unavailable pane is reported honestly and does not rewrite the recorded run outcome.

Rationale: Reattachment is part of the tool's core detach-and-return workflow, and the adapter already owns the necessary attach primitive. The small command prevents herdr-specific UX from leaking through `status`, gives later tmux and remote lanes one stable user-facing operation, and makes retained panes from DR18 useful without adding new background machinery. The trade-off is one additional v0 command and the need for stable public run identifiers, both justified by the primary interactive use case.
