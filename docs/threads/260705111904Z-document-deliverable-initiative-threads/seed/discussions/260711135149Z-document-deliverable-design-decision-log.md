# Decision log — document-deliverable & initiative threads (the seed)

## P1: Where the deliverable physically lands, given V2 declares its own docs immutable

Point: Where does this deliverable physically land, given V2 declares its own docs immutable? The thread fixes three gaps *inside V2*, but V2's own README says its reference docs are immutable by convention and a future ruleset lives at its own `docs/workflow/<version>/` rather than as in-place edits.

What you need to know: `docs/workflow/v2/README.md` (lines 52–57) says, verbatim: "the V2 reference docs are themselves immutable by convention… A future ruleset would live at its own `docs/workflow/<version>/` rather than as in-place edits here, so a reader can always cite a stable rule." The reason is load-bearing: published skills restate these rules inline, and a V1/older reader "can always cite a stable rule." So there's a genuine tension — the seed wants to change V2, but V2's own constitution says you don't edit it in place; you cut a new version. This changes what the junior dev's task literally is: "edit six files under `v2/`" vs. "fork `v2/` into a new version and edit there" vs. "add an addendum doc." It also interacts with the skills — every skill that restates V2 rules would need to know which version it's restating.

Decision: Edit the V2 docs in place (option A). The three changes are treated as additive gap-fills applied directly to the existing `docs/workflow/v2/` doc set; no new versioned ruleset folder (`v2.1/`, `v3/`) is cut, and no separate addendum doc is introduced.

Rationale: Justified by sole ownership and the additive nature of the changes — this is a personal repo with no external consumer base pinned to a frozen V2 snapshot, so the immutability convention's protection (letting an older reader cite a stable rule) is largely theoretical here, and the owner will read the full diff before committing/PR. Note: recommended option C (formalize an explicit "additive amendment in place, marked as such" carve-out rule so the in-place edit is legal rather than a silent breach of the written convention); the owner accepted plain in-place editing (A), accepting that the "immutable by convention" self-description is being edited in place along with everything else. The owner's stated reason ("I will read the files before committing/PR") addresses edit-correctness rather than the convention itself; the decision still holds on the ownership/additive grounds above. Residual to resolve separately: the README's "immutable by convention" paragraph becomes self-contradictory under A and must be reconciled (see next point).

## P2: The roadmap artifact — its home and its physics

Point: The roadmap artifact — its home and its physics. `tiers.md` requires a phased roadmap for tier 3 and `spine.md` calls it a living list inside the open thread; it must stay freely editable for the life of the initiative, yet nothing in the V2 folder set can hold a durable-but-mutable file. What model do we commit to?

What you need to know: `tiers.md` requires a phased roadmap for tier 3; `spine.md` (lines 132–135) calls it "a living list inside the open thread, not a frozen contract" — phases welcome or defer follow-ups over the life of the initiative, so it must stay freely editable (not append-only like the ledger, not frozen-at-emission like a record, not frozen-at-latch like a spec). The seed frames the fix as either "a third artifact class" or "a named exception like a root-level `roadmap.md` sibling to `ledger.md`." Those aren't rival options — the root-level file is the home, and "third class" is the physics that home needs. One more constraint settles the shape: the deliverable is "approved proposal + roadmap," but the proposal freezes at `approved` while the roadmap must keep breathing — so they cannot be the same file, ruling out "roadmap as a section of the proposal."

Decision: Root-level `roadmap.md` singleton plus a new "living document" physics class (option A). One roadmap per thread, fixed name, sibling to `ledger.md` and `seed/`, present only on initiative/umbrella threads. Physics: durable, freely mutable in place for the life of the thread, freezes only at thread close (the freeze guard's `closed: → reject all diffs` already covers it with no new rule). `lifecycle.md`'s "Two Artifact Classes, Plus the Ledger" section grows a third class: the living document — neither a record (frozen at emission) nor a versioned artifact (frozen at latch). The roadmap lives at the root of the dedicated initiative/umbrella thread; child threads carry no roadmap.

Rationale: Matches the `ledger.md`/`seed.md` precedent — a fixed-name root singleton is already a stable, unique identifier, so it needs no lineage-folder ceremony (rejected option B) and no version machinery. Folding it into the ledger (rejected option C) is the wrong physics: the ledger is append-only with a strict parser line grammar, whereas the roadmap is prose rewritten as phases shift. Naming the physics as an explicit third class in `lifecycle.md` is more honest than smuggling it in as an unnamed "exception." Invariant to the open tier question (P3): whether the umbrella is tier 3 or a new tier, the roadmap's shape is unchanged.
